Core Language Specification – svl v0.2
Date: March 27, 2026
Author: Shavais (with assistance from Grok)

1. Purpose and Design Goals

svl is a general-purpose, high-performance, cross-platform programming language primarily intended for game and engine development. It aims to be significantly more pleasant and productive to work with than C++ while retaining the ability to produce efficient native code on x86-64, ARMv8/v9, and other relevant architectures.

Key goals:

High performance with deterministic memory behavior (no garbage collection).
Natural mixing of abstraction levels within the same codebase.
Minimal ceremony (no header files, minimal namespace qualification, no mandatory imports from internal modules).
Excellent support for game-engine use cases using cross-platform C/C++ libraries (Vulkan/OpenGL, OpenAL, GLFW, etc.).
No deep dependency on .NET or any managed runtime in the final language or its standard tooling.

2. Abstraction Levels

svl does not use explicit annotations or marked blocks for abstraction levels.
The level is inferred naturally from the code written:

Raw bytecode opcodes ? low-level
C-style function calls with simple parameters ? mid-level
High-level constructs (Python-like comprehensions, concise functional code, etc.) ? high-level

The compiler must support seamless mixing of these styles within the same function or module.

3. Type System and Inference

Strong, static typing with aggressive inference.
Types are mutable during compilation. The compiler starts with the most efficient interpretation and revises it based on observed usage.
Example: A list literal begins as a homogenous, materialized, aligned, fixed-size array.
If later code appends elements ? becomes resizable.
If later code inserts a different type ? becomes non-homogenous and non-materialized.

4. Memory Management

No garbage collector.
Primary mechanisms:

Automatic reference counting for high-level objects (compiler-inserted, heavily optimized).
First-class arenas for performance-critical or temporary data.
Auto-release pools for short-lived objects.

Manual control remains available in mid- and low-level code.
Deterministic behavior and predictable performance are required.

5. Module and Build System

No header files.
No include, import, or using statements at the file level.
A project file declares configurations (debug/release), library search paths, and libraries to link against.
The compiler collects undefined symbols and resolves them against linked libraries at the end of compilation.
To improve performance, the compiler generates indexed manifest files for libraries and compiled modules (updated only when necessary).
Future svide IDE will maintain an LRU cache of manifests with configurable size limits.

6. Namespaces

Namespaces exist for organization but should impose minimal friction.
The compiler maintains a sorted symbol table (name first, then namespace path).
Qualification is required only in the rare case of an exact signature collision.
Blanket “using namespace” imports are not desired.

7. Bootstrap Compiler

The initial (bootstrap) compiler will be written in C++.
Reason: To avoid building any significant dependency on .NET and to ensure smooth interoperability with the cross-platform C/C++ libraries svl is intended to use (OpenGL, Vulkan, GLFW, OpenAL, libzip, stb_image, FreeType, etc.).
Once svl is sufficiently mature, the compiler will be ported to svl itself (self-hosting / bootstrapping).

8. Bytecode and Compilation Model

Intermediate representation: Register-based bytecode with type-tagged opcodes, explicit SIMD support, and memory fences.
Runtime modes will eventually include interpretation/light JIT for rapid iteration and full AOT for release builds.

9. Future IDE (svide)

The eventual IDE will be written in svl.
UI definition will reuse the existing custom text-based UI format from your C++ engine.
Any visual designer will output this same text format.

10. Library Philosophy

svl is designed to interoperate cleanly with cross-platform C and C++ libraries.
No deep .NET integration is planned.
The language and its standard libraries should favor direct use of native C/C++ libraries where appropriate.

Open Items (to be refined)

Exact syntax for high-level constructs.
Precise project file format.
Details of arena/reference-counting syntax and semantics.
Namespace qualification syntax (“with” block or alternative).
Bytecode instruction set.
Exact manifest and symbol resolution mechanics.



Compiler Architecture Overview

The svl compiler (svl) will follow a classic multi-stage pipeline, optimized for your stated goals: efficiency, minimal ceremony, flexible type inference, and clean separation between source, intermediate artifacts, and final output.
1. Command-Line Interface & Project Discovery

Command: svl [projectname] [mode]
projectname (optional) — name of the .svl project file (without extension). If omitted, looks for any .svl file in the current directory (error if multiple found).
mode — debug (default) or release.

Behavior if multiple .svl files exist:
If a specific projectname is given ? use that one.
Otherwise ? error (to avoid ambiguity).

The .svl project file specifies:
Configurations (debug / release)
Per-configuration settings: source roots, intermediate directory, output directory, target type (.exe, .lib/.a, .dll/.so), linked libraries, and library search paths.


2. Major Compiler Components
The compiler should be structured as a series of loosely coupled passes:

Project Loader & Configuration Resolver
Parse the .svl project file.
Resolve active configuration (debug/release).
Build list of source files by walking the folder tree (recursive from source root(s)).

Frontend (Source ? AST)
Lexer / Tokenizer (reads files in buffers for efficiency).
Parser ? produces a Concrete Syntax Tree (CST) or Abstract Syntax Tree (AST) per file.
Symbol table construction begins here (per-scope symbol tables with flexible type slots).

Semantic Analysis & Type Inference Pass
This is the most complex and important pass for svl.
Performs name resolution, type inference, and type revision.
Because types can change based on later usage (fixied size ? resizable, homogenous ? heterogenous, etc.), this pass must be able to revisit earlier decisions without excessive memory copying.
Uses a revision-friendly type representation (small, granular memory blocks + pointer tree, as you suggested).

Bytecode Generation
Lowers the semantically analyzed AST into register-based bytecode.
Inserts reference counting, arena management, and memory barriers where required.
Produces per-module .svm files (object + manifest).

Linker / Manifest Resolver
Reads all .svm manifests.
Resolves undefined symbols against libraries and other modules.
Performs final code generation / linking into the target binary.

Backend (Bytecode ? Native)
For release builds: full AOT code generation.
For debug builds: may emit interpreted bytecode or light JIT stubs (future).


3. File Naming and Intermediate Artifacts

Source files: .sv
Project file: .svl
Intermediate objects + manifests: .svm
To avoid name collisions: replicate the source folder hierarchy under the intermediate directory.
Example: src/game/player.sv ? intermediate/debug/src/game/player.svm

Output: Same base name as the project file, with appropriate extension (.exe, .lib, .dll, .a, .so, etc.) placed in the configured output directory.

4. Parse Tree / AST Design Considerations
Your instincts are good. Recommended structure:

Tokens: Use uint16_t (u16) for token kinds. 65,535 kinds are more than enough (you will likely need < 300).
AST Nodes: Small, granular nodes with pointers. Avoid large contiguous blocks that require frequent resizing.
Symbol Table: Per-scope tables. Each symbol entry contains:
Name (string table index)
Namespace path (if needed)
Current Type (flexible / revisable structure)
Flags (const, static, materialized, etc.)

Type Representation: Critical for your "mutable type" requirement.
Use a small struct with a type tag + union or pointer to additional data.
When revision occurs (e.g., list becomes resizable), update the type in place or replace the type node with a new small block. The pointer tree makes this cheap.


This design trades a small amount of indirection for the flexibility you want.
5. Efficiency Considerations

Read source files in large buffers (e.g., 64 KB or 256 KB chunks).
Use memory-mapped files where possible for large projects.
Build manifests incrementally — only recompile files whose dependencies have changed.
The manifest system will be key to fast incremental builds.

new language spec:

The svl Language Specification (v1.0)

1. Core Philosophy

Performance: AOT compiled to native code; no global Garbage Collector.
Memory Trinity: Explicit control via Arenas (bulk), Release Pools (tracked), and ARC (deterministic reference counting).
Morphic Inference: Types start "materialized/contiguous" and "morph" into virtual, dynamic, or concurrent versions, during compilation, based on usage.
Zero-Boilerplate: Global/namespaced visibility by default; no include or import required unless ambiguous.

2. Memory & Variables

Assignments:

= (Copy): Data isolation.
:= (Reference): Promotes both sides to shared pointers, unless both sides are in the same arena.

Pointer Types:

ref: Safe, shared ownership (Default).
weak: Safe observer; becomes a null Optional if the owner dies.
raw: Unsafe, C-style pointer for extreme optimization.

The "Ghost" Null: null is type-agnostic. For classes, it creates an "Optional" that silently fails (no-op) on method calls, preventing crashes. 

3. The Morphic Type System

Inference Locks: A var accumulates a "Union" of types (e.g., <i32, zs8, User>) based on all possible branches in its scope.
One-Way Morphing: Passing a variable or using it on the Right-Hand Side (RHS) of an expression never morphs the source; only the Left-Hand Side (LHS) or internal function logic evolves.
(Exception: reference assignment causes both sides to morph into shared pointers unless they are in the same arena. This may switch a structure on the right into being virtual rather than materialized.)
Built-in Structures: list, map, stack, queue, etc., are abstract intents. The compiler chooses the best backing store (const vs non-const, fixed size vs resizeable, materialized vs virtual, homegenous vs heterogenous, single threaded vs concurrent).

Queues have 4 optional qualifier hints: spsc, mpsc, mpmc, mpmc. (eg. spsc queue myQueu;) The backing store for each type is optimized for that type.

4. Concurrency

Inferred Thread-Safety: Objects crossing thread boundaries automatically upgrade to concurrent or atomic versions.
Concurrent Arenas: allocation into the arena is serialized/guarded.
Concurrent Pools: drain() is an async operation returning an optionally waitable promise with telemetry (bytes freed, zombie counts).
Generational Retirement: drain() retires active memory segments while allowing new allocations to start immediately in fresh segments.


5. Build System & Project Structure

File Layout: .svl (Project Configuration), .sv (Source), .xmo (Intermediate Objects)
Namespacing: Filenames act as the first segment of a namespace path. Paths can be shortened to the minimum unique segment.
Compiler CLI: svl [projectname] [config] (e.g., svl hello debug).

Things don't morph at run time, by the way. Morphing is strictly a compilation time phenomenon, it just means that the language will pick the most performant version of your type that it can and still accommodate everything you're doing with it. 

Lists and other structures and objects that allocated in arenas or heaps do morph into concurrent versions if they get used in multiple threads. 
It's just that their contained pointers don't become shared pointers if a reference assignment (:=) happens into something that is allocated
in the same arena or heap.

var a := mylist[i];  // reference assignment - normally this causes a to become a shared pointer, and mylist to become a list of shared pointers.

heap h1;

list mylist:h1;
..

var a:h1 = mylist[i]; // here, since a and mylist share the same application managed heap, they don't need to be shared pointers to be considered safe. 

If you reference something that's been deleted from an app managed heap, you'll get garbage, but you won't get an access violation crash.

	void FastWriteBinaryFile(const string& outPath, std::vector<uint8_t>& buf)
	{
		// Open file with Win32 API
		HANDLE hFile = CreateFileW( outPath.wstring().c_str(), GENERIC_WRITE,	0, NULL, CREATE_ALWAYS,	FILE_ATTRIBUTE_NORMAL | FILE_FLAG_SEQUENTIAL_SCAN, NULL);

		if (hFile == INVALID_HANDLE_VALUE)
		{
			oserror << "Failed to write output file: " << outPath << endl; // sets data::ErrorOccurred
			return;
		}

		// Pre-allocate file size to avoid fragmented allocation
		LARGE_INTEGER liSize;
		liSize.QuadPart = buf.size();
		if (SetFilePointerEx(hFile, liSize, NULL, FILE_BEGIN)) {
			SetEndOfFile(hFile);
			SetFilePointer(hFile, 0, NULL, FILE_BEGIN);
		}

		// Perform a single large write operation
		DWORD written;
		WriteFile(hFile, buf.data(), static_cast<DWORD>(buf.size()), &written, NULL);

		CloseHandle(hFile);
	}

void WriteParseTree(std::vector<uint8_t>& buf, const data::ParseTreeNode* node) {
		if (!node) {
			WritePod<uint8_t>(buf, 0);
			return;
		}
		WritePod<uint8_t>(buf, 1);

		// Code Blocks
		uint32_t cbCount = static_cast<uint32_t>(node->codeBlocks.size());
		WritePod(buf, cbCount);
		for (auto cb : node->codeBlocks) WritePod(buf, cb);

		// Patch Symbols
		uint32_t psCount = static_cast<uint32_t>(node->patchSymbols.size());
		WritePod(buf, psCount);
		for (const auto& sym : node->patchSymbols) WriteString(buf, sym);

		// Static Data
		uint32_t sdCount = static_cast<uint32_t>(node->staticData.size());
		WritePod(buf, sdCount);
		for (const auto& sd : node->staticData) {
			WriteString(buf, sd.name);
			WritePod(buf, sd.exportIdx);
			uint32_t byteCount = static_cast<uint32_t>(sd.bytes.size());
			WritePod(buf, byteCount);
			buf.insert(buf.end(), sd.bytes.begin(), sd.bytes.end());
		}

		// Function Metadata
		if (node->funcData) {
			WritePod<uint8_t>(buf, 1);
			WriteString(buf, node->funcData->name);
			WritePod(buf, node->funcData->totalStackSize);
			WritePod(buf, node->funcData->exportIdx);
			WritePod(buf, node->funcData->isLeaf);
		}
		else {
			WritePod<uint8_t>(buf, 0);
		}

		// Children
		uint32_t childCount = static_cast<uint32_t>(node->children.size());
		WritePod(buf, childCount);
		for (auto* child : node->children) WriteParseTree(buf, child);
	}


	void WriteXmoFile(const data::Xmo* xmo, const fs::path& outPath) {
		fs::create_directories(outPath.parent_path());
		std::vector<uint8_t> buf;
		buf.reserve(4096);

		// Serialize Header/Name
		WriteString(buf, xmo->name);
		WritePod(buf, xmo->tempGlobalOffset);

		// Serialize Code Buffer
		uint32_t codeSize = static_cast<uint32_t>(xmo->codeBuffer.size());
		WritePod(buf, codeSize);
		buf.insert(buf.end(), xmo->codeBuffer.begin(), xmo->codeBuffer.end());

		// Serialize Exports
		uint32_t expCount = static_cast<uint32_t>(xmo->exports.size());
		WritePod(buf, expCount);
		for (const auto& exp : xmo->exports) {
			WriteString(buf, exp.name);
			WritePod(buf, exp.offset);
			WritePod(buf, exp.refinementMask);
			WritePod(buf, exp.scopeId);
		}

		// Serialize Relocations
		uint32_t relCount = static_cast<uint32_t>(xmo->relocs.size());
		WritePod(buf, relCount);
		for (const auto& rel : xmo->relocs) {
			WriteString(buf, rel.targetSymbol);
			WritePod(buf, rel.offset);
			WritePod(buf, rel.type);
		}

		// Serialize Scope Tree
		uint32_t scopeCount = static_cast<uint32_t>(xmo->scopeTree.size());
		WritePod(buf, scopeCount);
		for (const auto& scope : xmo->scopeTree) {
			WritePod(buf, scope.id);
			WritePod(buf, scope.parentId);
			WriteString(buf, scope.name);
		}

		// Write the file
		FastWriteBinaryFile(outPath.string(), buf);
	}

	void SaveXmos() {

The Final "Dumb Compiler" Blueprint:
Forward Pass (The Mapper):
Linearize your tree into a global timeline of instruction indices.
Keep a simple table: Map<{ScopeID, VarID}, LastInstructionIndex>.
Every time a variable is referenced (even a "reach-in"), update its LastInstructionIndex.
Backward Pass (The Back-Patcher):
Start from the end of the timeline.
For each variable in your table, find the instruction at LastInstructionIndex.
Set the releases bit for that register in that specific instruction.
Finalize Pass (The Optimizer):
Walk the timeline forward.
Track the currentLiveMask.
For each block:
If (block.reserveMask & currentLiveMask) != 0, jump to altTableIndex (and log if it's a spill).
currentLiveMask |= block.reserveMask.
currentLiveMask &= ~block.releases.
Why this is the "Golden Path" for you:
Speed: It’s strictly where is the number of instructions. No complex graph-traversal.
Correctness: By definition, a register cannot be reused until the absolute last point the programmer could have possibly looked at it.
Transparency: Your verbose debug output for spills will perfectly reflect the programmer's code. If they see too many spills, they can see that "reaching in" for a variable 50 lines later is what's keeping the register tied up.
Since you've already got the COFF writer and the Emitter working, adding this Liveness Registry and a two-pass linear optimizer is the last major "bridge" to having a fully functional, high-performance XCM compiler.

This is the classic "Compiler Architect's Dilemma": balancing a global view of the world with the desire for incremental speed.
Since you are already using memory-mapped files (mmap) and a thread pool, you are actually in a great position to avoid the "massive file handles" and "slowdown" fears.
Here is a breakdown of how to handle the symbol vs. parse tree loading efficiently:
1. The "Thin" Global Index (Recommended)
Don't load everything, but don't be truly "lazy" (on-demand) either, as that kills your parallelization. Instead, use a two-phase header approach within your .xmo files:
Phase 1 (The "Export" Table): Map the .xmo and read only the symbol table/metadata. This should be a contiguous block at the start of the file. Store these symbols in your global table with a pointer/ID back to the originating .xmo object.
Phase 2 (The "Body"): Keep the file mapped, but don't touch the memory pages for the parse trees or code blocks yet. Because it’s an mmap, the OS won't actually pull those pages from the SSD into RAM until you access the memory addresses.
2. Why Memory Mapping is your Secret Weapon
If you map the whole .xmo but only read the first 4KB (the symbols), the rest of the file stays "virtual."
File Handles: Modern OSs (Windows/Linux) easily handle thousands of open handles. Unless your project has 10,000+ source files, this isn't a real bottleneck.
Lazy Loading for free: By using mmap, the Operating System is your lazy loader. You don't need to write if(!loaded) load(); logic. You just access the ParseTree* pointer, and the OS triggers a page fault to bring that specific chunk of the file into RAM.
3. Refined Execution Flow
Based on your current code, I’d tweak the sequence to this:
Parallel Map & Index: Run through all .xmo files in the thread pool. Map them and register their symbols into the global table.
The "Brains" (Analysis): Start your parser. When the parser sees ExternalType, it looks it up in the global table.
On-Demand Tree Access: If the parser decides it needs to inspect the implementation (the parse tree) of an external symbol for refinement, it just follows the pointer.
4. When to "Load All" vs. "Stay Thin"
If your language allows transitive refinements (changing code in File A causes a ripple effect that requires re-checking the parse tree of File B), you will eventually touch those pages anyway.
My suggestion: Don't fight the "load all" instinct, but optimize the layout. Ensure your .xmo format puts the "Global Symbols" at the front and "Private Parse Trees/Code" at the back. This ensures that even if you map everything, you only burn IOPS and cache on the data you actually use.

Parallel Symbol Registration Pattern
This approach uses sharding to minimize contention while threads parse their memory-mapped .xmo files.

#include <vector>
#include <string>
#include <shared_mutex>
#include <unordered_map>
#include <thread>

// Represents a symbol's metadata found in an .xmo
struct SymbolEntry {
    uint32_t xmoId;
    uint32_t typeInfo;
    void* parseTreeRef; // Pointer into the mmap'd region
};

class GlobalSymbolTable {
    static constexpr size_t SHARD_COUNT = 256; // High number to reduce collisions
    
    struct Shard {
        std::shared_mutex mutex;
        std::unordered_map<std::string, SymbolEntry> symbols;
    };
    
    std::vector<Shard> shards{SHARD_COUNT};

    // Simple hash to determine which shard a symbol name belongs to
    size_t getShardIndex(const std::string& name) const {
        return std::hash<std::string>{}(name) % SHARD_COUNT;
    }

public:
    // Called by worker threads as they scan .xmo headers
    void RegisterSymbol(std::string name, SymbolEntry entry) {
        auto& shard = shards[getShardIndex(name)];
        
        // Use a unique_lock for writing
        std::unique_lock lock(shard.mutex);
        shard.symbols[std::move(name)] = entry;
    }

    // Called by the parser during 'The Brains' phase
    bool TryLookup(const std::string& name, SymbolEntry& outEntry) {
        auto& shard = shards[getShardIndex(name)];
        
        // Use shared_lock for parallel reading
        std::shared_lock lock(shard.mutex);
        auto it = shard.symbols.find(name);
        if (it != shard.symbols.end()) {
            outEntry = it->second;
            return true;
        }
        return false;
    }
};

Why this works for your compiler:
Massive Parallelism: With 256 shards, 32+ threads can register symbols from different files simultaneously with almost zero chance of blocking.
Shared Reading: During the parsing phase, multiple threads can look up the same common types (like size_t or void*) simultaneously because std::shared_lock allows concurrent readers.
Memory Mapped Integration: The parseTreeRef can be a direct pointer into your memory-mapped .xmo. Because the OS handles the page faults, your registration stays "thin"—you aren't copying the whole tree, just its address.

	void DeserializeXmo(data::Xmo* xmo, const std::vector<uint8_t>& buf, bool loadParseTree) {
		uint64_t offset = 0;
		xmo->name = ReadString(buf, offset);
		xmo->tempGlobalOffset = ReadPod<uint32_t>(buf, offset);

		// Code Buffer
		uint32_t codeSize = ReadPod<uint32_t>(buf, offset);
		xmo->codeBuffer.resize(codeSize);
		std::memcpy(xmo->codeBuffer.data(), &buf[offset], codeSize);
		offset += codeSize;

		// Exports
		uint32_t expCount = ReadPod<uint32_t>(buf, offset);
		for (uint32_t i = 0; i < expCount; ++i) {
			data::XmoExport exp;
			exp.name = ReadString(buf, offset);
			exp.offset = ReadPod<uint32_t>(buf, offset);
			exp.refinementMask = ReadPod<uint32_t>(buf, offset);
			exp.scopeId = ReadPod<uint32_t>(buf, offset);
			xmo->exports.push_back(exp);
		}

		// Relocations
		uint32_t relCount = ReadPod<uint32_t>(buf, offset);
		for (uint32_t i = 0; i < relCount; ++i) {
			data::XmoRelocation rel;
			rel.targetSymbol = ReadString(buf, offset);
			rel.offset = ReadPod<uint32_t>(buf, offset);
			rel.type = ReadPod<uint16_t>(buf, offset);
			xmo->relocs.push_back(rel);
		}

		// Scope Tree
		uint32_t scopeCount = ReadPod<uint32_t>(buf, offset);
		for (uint32_t i = 0; i < scopeCount; ++i) {
			data::XmoScope scope;
			scope.id = ReadPod<uint32_t>(buf, offset);
			scope.parentId = ReadPod<uint32_t>(buf, offset);
			scope.name = ReadString(buf, offset);
			xmo->scopeTree.push_back(scope);
		}

		// Parse Tree (prefixed with size for fast skipping)
		uint32_t treeDataSize = ReadPod<uint32_t>(buf, offset);
		if (loadParseTree) {
			xmo->parseTree = ReadParseTree(buf, offset);
		}
		else {
			xmo->parseTree = nullptr;
		}
	}

		HANDLE hFile = CreateFileW(
			path.wstring().c_str(),
			GENERIC_READ,
			FILE_SHARE_READ,
			NULL,
			OPEN_EXISTING,
			FILE_ATTRIBUTE_NORMAL | FILE_FLAG_SEQUENTIAL_SCAN,
			NULL
		);

		if (hFile == INVALID_HANDLE_VALUE) {
			// Only log if the file actually exists but failed to open
			if (fs::exists(path)) {
				oserror << sformat("Failed to open binary file: %s", path.string().c_str()) << endl;
			}
			return {};
		}

		// 2. Get file size for mapping
		LARGE_INTEGER size;
		if (!GetFileSizeEx(hFile, &size) || size.QuadPart == 0) {
			CloseHandle(hFile);
			return {};
		}

		// 3. Create a file mapping object
		HANDLE hMapping = CreateFileMappingW(hFile, NULL, PAGE_READONLY, 0, 0, NULL);
		if (!hMapping) {
			oserror << sformat("Failed to create file mapping: %s", path.string().c_str()) << endl;
			CloseHandle(hFile);
			return {};
		}

		// 4. Map the view into memory
		void* pData = MapViewOfFile(hMapping, FILE_MAP_READ, 0, 0, 0);
		std::vector<uint8_t> content;

		if (pData) {
			// Copy the mapped data into our vector
			// This is typically faster than ReadFile because it bypasses extra system buffering
			content.assign(
				static_cast<const uint8_t*>(pData),
				static_cast<const uint8_t*>(pData) + size.QuadPart
			);
			UnmapViewOfFile(pData);
		}

		// 5. Cleanup handles
		CloseHandle(hMapping);
		CloseHandle(hFile);

		return content;

struct SymbolKey {
    char* name;         // pointer into our arena
    uint32_t scope_id;

    bool operator==(const SymbolKey& other) const {
        return scope_id == other.scope_id && name == other.name;
    }
};

struct SymbolKeyHasher {
    size_t operator()(const SymbolKey& k) const {
        // mixing constants
        static constexpr uint64_t M = 0xc6a4a7935bd1e995;
        static constexpr int R = 47;

        uint64_t h = 0x84222325 ^ (12); // Seed mixed with key size (8+4 bytes)

        // Mix the Pointer (8 bytes)
        uint64_t k1 = reinterpret_cast<uintptr_t>(k.name);
        k1 *= M; k1 ^= k1 >> R; k1 *= M;
        h ^= k1; h *= M;

        // Mix the Scope ID (4 bytes)
        uint64_t k2 = static_cast<uint64_t>(k.scope_id);
        h ^= k2; h *= M;

        // spread entropy across all bits (for a 0xFF mask)
        h ^= h >> R; h *= M; h ^= h >> R;

        return static_cast<size_t>(h);
    }
};

}
