5. REVIEWER PLAN (implement this next)
=======================================

The Reviewer runs after the Morpher has converged for each Xmo. It does codegen-prep work
that needs fully-resolved types but precedes block selection.

Fields the Reviewer writes (all currently zero in ParseTreeNode; declared in Parser.h):

    paramSlot      — u8 on each arg-expression child of ArgList
                     1=rcx, 2=rdx, 3=r8, 4=r9, 5=[rsp+32], 6=[rsp+40], … 0=unassigned
    totalStackSize — u32 on FuncDecl nodes; total prologue bytes for sub rsp,N
    funcFlags      — u32 on FuncDecl nodes; bit 0=isFallible, bit 1=needsListeningFlavor,
                     bit 2=needsDiscardingFlavor (all zero for hello.xm)

Symbol field written by the Reviewer (already declared on Symbol):
    offset         — u32; rsp-relative offset of each local variable's stack slot

ParseTreeNode fields to be written by the Reviewer:
    paramSlot
    totalStackSize
    funcFlags 

Stack frame layout (Windows x64 ABI):
    N ≡ 8 (mod 16) — so that rsp is 16-byte aligned at each call site inside the function
    rsp+0  .. rsp+31   shadow space (must always be allocated, even for 0-arg callees)
    rsp+32 .. rsp+N-1  extra arg slots: 8 bytes per arg beyond 4 in the deepest call
    rsp+N  ..          locals; Symbol::offset = N + 8*i in declaration order
    totalStackSize = 32 + (extra_arg_slots*8) + local_bytes, rounded to next ≡ 8 (mod 16)

Register assignment for call sites:
    Walk each Call node; read sym->declNode (ExternDecl or FuncDecl).
    Match each child of ArgList to the corresponding child of callee's ParamList.
    Read Param::regHint: "rcx"→1, "rdx"→2, "r8"→3, "r9"→4, no regHint→5,6,…
    Write paramSlot on each arg expression.

For hello.xm, the functions are:
    GetStdHandle(i32:rcx) → one arg, paramSlot=1
    WriteFile(u64:rcx, *u8:rdx, u32:r8, *u32:r9, *u8) → paramSlots 1,2,3,4,5
    ExitProcess(u32:rcx) → one arg, paramSlot=1
    main() → no args

hello.xm local variables in main:
    h       (u64): offset = totalStackSize + 0
    written (u64): offset = totalStackSize + 8

For hello.xm main: deepest call is WriteFile (5 args → 1 extra slot at rsp+32).
    totalStackSize = 32 (shadow) + 8 (1 extra slot) + 16 (h and written) = 56
    56 mod 16 = 8 → ≡ 8 (mod 16) ✓ — no rounding needed.

Implementation steps:
    1. Create Reviewer.cpp implementing static void Reviewer::Review(Xmo&, SymbolTable&, const CompileJob&)
    2. Walk each FuncDecl in xmo.ownedSymbols or by iterating the parse tree
    3. For each function:
         a. Walk ArgList children of every Call and assign paramSlot from callee ParamList regHints
         b. Count deepest-call extra args; count locals; compute totalStackSize
         c. Set funcFlags (all zero for hello.xm)
         d. Assign Symbol::offset to each VarDecl local


6. CODER PLAN (after Reviewer)
================================

The Coder runs in a thread per .xm file, it walks the parse tree and populates codeBlocks 
on each node — indices into a unified CodeBlocks table mapping to byte sequences. For 
hello.xm the Coder only needs to handle: ExternDecl (no bytes, but import record), 
FuncDecl prologue/epilogue, VarDecl, Assign, Call with args at known paramSlots, Return.

The Coder does NOT write bytes; it selects code-block indices. The Emitter reads those
indices to emit actual bytes.

Implementation is not yet planned in detail. Defer until Reviewer is complete and tested.


7. EMITTER PLAN (after Coder)
===============================

Two phases:
  Phase 1 (per-Xmo, parallelizable): walk parse tree, copy bytes from CodeBlocks table,
           accumulate relocations, populate Xmo::codeBuffer.
  Phase 2 (single-threaded): serialize each Xmo to .xmo; write COFF .obj via Coff class;
           Linker calls link.exe to produce .exe.

For hello.xm:
  - .text section: main() body (prologue, calls, epilogue)
  - .rdata section: "Hello, world!\n" string
  - import relocations for WriteFile, GetStdHandle, ExitProcess in kernel32.dll
  - COFF export for main (entry point for the linker)

Implementation is not yet planned in detail. Defer until Coder is complete and tested.


8. SETTLED DESIGN DECISIONS (do not relitigate)
=================================================

Parser → Morpher handoff:
  The Parser collects leaf nodes (childCount==0) into a local vector during Drive().
  After Drive() returns, Parser::Parse calls Morpher::MorphTree(xmo, symbols, job, leaves).
  MorphTree processes leaves sequentially. This is correct because all pendingChildren
  counters are correctly initialized by popNode() before any leaf is processed.
  DO NOT submit pool tasks from inside Drive() — this causes pendingChildren race conditions.
  [This plan is temporary. See Concurrent Morpher deferral below.]

No pool.wait() from inside a pool task:
  Parser runs as a pool task. Calling pool.wait() from within it either deadlocks or
  returns before submitted tasks complete. MorphTree is called directly by the Parser task,
  not submitted as a separate pool task.

Symbol::declNode on Symbol (not a per-run declSite map):
  When DeclareSymbol creates a symbol, it stores sym->declNode = node.
  Morpher::Ident reads sym->declNode to copy types instead of looking up a side table.
  This is required for the concurrent Morpher variant.

No morphQueue on Xmo:
  The old Xmo::morphQueue field was removed. The leaves vector lives as a local in ParseCtx.
  Xmo::ownedSymbols (for symbols) and Xmo::scopeTree (for scopes) are the only Morpher
  outputs accumulated on Xmo.

PushOwnedSymbol / PushScopeRecord are mutex-guarded:
  These are the only Morpher writes to Xmo that need synchronization (for the future
  concurrent variant). The parse tree fields themselves are written by only one thread at
  a time in the current sequential design.

Concurrent Morpher is deferred:
  The subscriber mechanism (§2c in UTH) is needed before concurrent pool-task-per-leaf
  morphing is safe. Without it, Ident tasks race with ExternDecl tasks.
  All the atomic machinery is in place (pendingChildren, Xmo::morphMtx_). Enabling
  concurrent morphing is a matter of: replacing leaf vector drain with pool task submission
  and implementing ResolveSymbol → subscribe-on-not-found.

Single-file projects only for now:
  Cross-file propagation requires the subscriber index. Not yet implemented.
  hello.xm is a single-file project, so this is not a blocker for the immediate goal.

Windows x64 only:
  All register names (rcx, rdx, r8, r9), ABI rules (shadow space, alignment),
  and COFF output are Windows x64 specific.
  [Eventually we hope to port to other platforms and CPU architectures, but that's a far off goal.]
