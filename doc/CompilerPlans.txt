
A recursive descent parser was attempted for XMC but quickly became a debugging and maintenance nightmare.
The volume of code required to simply validate syntax is substantial, and if you change your bnf, good luck fixing all that code.

I've written parsers before using a pattern like this:

We have big table of states, tokens and next states, like this:

{state: {tokens: nextstate, ohtertokens: othernextstate, ..}, ...}

We have another big table of state functions, like this:

[state: statefunc(string), ..]

We set the state to the start state, enter a loop like this:

while not EOF:
	get the next character (into curchar)
	look up the state
	iterate over the token sets for the state
	if curchar is not in any of the token sets, emit an error and break
	look up the state function for the next state and call it, passing the accumulate string + curchar
	
If the state function is null, no transition happens, the parser just adds curchar to the accumulated string that is eventually passes 
to the next non-null state handler.
Most state functions change the current state and clear the accumulated string.
Many or most state functions could potentially emit parse nodes.

Here's a tentative plan for the parsing and emitting phases of the compiler; each of these are a separate spawned thread per source file:

lexor: 
	- table driven state machine that posts tokens into a token queue for the parser 
	- emits syntax errors like "syntax error - file:line - expected one of [_,_,..] got [_]"
	- logs file:line:token entries into a token log file for a given project when lexor logging is requested
		(in the project file for the active config).
	- cleans up and halts if a "full scan required" event is raised

parser (thread per file):
	- consumes tokens from the token queue
	- adds symbol and action parse tree nodes to a parse tree
	- posts newly created parse tree node references into the typer queue
	- logs lines about created parse tree nodes into a parser log when parser logging is requested
	- cleans up and halts if a "full scan required" event is raised
		
typer:
	- consumes the typer queue 
	- builds a type dependency graph with (references to parse tree nodes)
	- emits typer errors to an oserror stream when immediate typer error conditions arise
	- discovers type resolutions and updates the types in parse tree nodes
	- posts typer-processed parse node references to the refiner queue
	- logs lines about discovered types into a typer log when typer logging is requested
	- raises a "full scan required" event if type changes impact xmo's (xmc object files) that aren't fully loaded
	- cleans up and halts if a "full scan required" event is raised
	
refiner:
	- consumes the refiner queue
	- builds a refinement dependency graph 
	- streams refinement related errors to an oserror stream when immedaite refinement error conditions arise
	- discovers required refinements and updates the refinements in parse tree nodes
	- raises a "full scan required" event if refinement morphs impact xmo's that aren't fully loaded
	- cleans up and halts if a "full scan required" event is raised

reviewer:
	- starts when the lexor, parser, typer and refiner have all finished if no "full scan required" event was raised
	- scans the parse tree for unresolved types and refinements
	- streams unresolved type and refinement errors to oserror
	- checks for and performs some parse node level optimizations, when possible and convenient
	- notifies the coder if reviewing completes without errors
	
coder: 
	- awakens when the reviewer notifies it of successfull completion
	- walks the parse tree
	- fills parse node code block id vectors:
		- looks up a vector of base type block id vectors for the node type, symbol basetype and action 
		- looks up a vector of refinement block id vectors for the node type, symbol refinement and action
		- stitches together block id vectors, tracking register usage
		- chooses alternate block id vectors when the defaults require registers that are in use
		- performs some block id vector level optimizations, when possible and convenient
	- notifies the emmitter upon successful completion

emitter: (already implemented and tested.. contains no intelligence) 
    - awakens when the coder notifies it of successfull completion
	- walks the parse tree, consumes parse node block id vectors and produces machine code in xmo code buffers 
    - saves loaded xmo files that end up with a dirty state
	- consumes xmo code buffers and produces the single COFF output for the project
	  (Section 20 of "The xmc programming language.txt" file has notes about xmo files and incremental and full builds.)

{	"statestring", 
	{	{"encounterstring", "nextstatestring"},		// if whitespace is not to be skipped, 
		{"encounterstring", "nextstatestring"},		// a "#" is prepended to encounterstring
	}
}

unordered_map<string, vector<pair<vector<string>, string>> NextStates
