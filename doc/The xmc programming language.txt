
The xmc programming language

1. AOT Compilation

- Compiles ahead of time straight to native code; does not use LLVM or any IL.
- Early compiler versions are written in C++ using platform SDKs, they write out COFF files to be linked by platform linkers.

2. Efficient, Deterministic Memory Management

- Does not garbage collect.
- Has arenas and release pools as first class objects.
- Objects not explicitly marked as raw that are not stored on the stack or in an arena or a release pool are reference counted and discarded immediately when the reference count reaches zero.

3. Global Code Visibility

- Anything in a global or namespaced scope that is anywhere in the code base is accessible from anywhere else, no includes, imports, or "using namespace" statements are required.
- Things can be put into namespaces, but when referencing something, its namespace path only needs to be specified when there is ambiguity otherwise.
- The translation unit (source code file) name is an optionally-specified first segment of a namespace path, so classes or functions of the same name can exist at the root of different files.
- Namespace path prefixes can be shortened to the shortest set of segments that eliminates ambiguity.
- Object files (.xmo) are created for translation units (.xm source files) during compilation to speed up future compilation.

Symbols are stored with their scope id's. In a sharded global symbol table (in addition to be stored in .xmo file exports), together with their scope id.
There is a global tree of scopes. 
A given scope has a stack of enherited scope id's.


4. Project Based Build System

An xmc target is either an executable, a static library or a dynamic library.

An invocation of the xmc command line compiler reads like this:

xmc projectname config

eg. xmc hello debug

There is a project file which is a [projectname].xmc file. (eg. hello.xmc) 
The source code is in .xm files. 
A root folder for the source tree can be specified in the project file. 
A root folder for the intermediate files must be specified.
The source folder tree is duplicated by the compiler under the intermediate root folder, and a .xmo file is created for each .xm file in the source folder tree.
A single linkable (.obj, on windows) is created for a target, in the intermediate root, of the same name as the project file. (eg., hello.obj)
The platform linker is invoked by the compiler (for now).
The compiler places a single linked target output (.exe, etc.) in the output folder specified in the project file.

Sample project file:
_________
# sample xmc Project File (hello.xmc)
[base]                          # base config, overriden by the default or the selected config
Arch = x64
Type = exe
EntryPoint = hello:main         # names the .xm file and the entry point function
Subsystem = console
CRT = static                    # static, dynamic, static-debug, dynamic-debug
LibPaths = C:/Windows/System32
StaticLibs = [glfw3, openal]
DynamicLibs = [kernel32, user32, opengl32]
SourceRoot = src                # optional, the default is the current working directory

[debug:default]                 # a config doesn't have to be debug or release, it can have any name
optimization = 0
symbols = yes
libpaths = deps/x64/Debug
intermediate = obj/x64/Debug
output = bin/x64/Debug

[release]
optimization = 9
symbols = no
libpaths = [deps/x64/Release]
intermediate = obj/x64/Release
output = bin/x64/Release
---------

5. Compile Time Inferred Type Morphing And Conversions

a. built-in types and their refinements:

boolean:    b
integer:    i8, i16, i32, i64, u8, u16, u32, u64
float:      f32, f64
decimal:    d16, d32, d64       - fraction type that maintains an integer numerator and denominator (and doesn't incur floating point representation errors)
character:  c8, c16,            - single byte character, double byte character
            zs8, zs16, zs32     - zero-terminated string of c8's, of c16's
            s8, s16,            - length-specified (pascal) string of c8's, of c16's
            utf8, us            - unicode strings

atomic  - qualifier that yields atomic versions of a simple type. (eg. atomic boolean) 

built in structures and objects:

struct      - no functions (except as pointers), no coded constructors or destructors
class
list
queue
stack
btree       - binary tree
hashset     
hashmap
function
arena
heap
(others to be determined)

concurrent - qualifier that yields a concurrent version of a container 

These built-ins, structures and objects have various versions that the compiler infers between. 

The compiler revises it's inference of exactly what version the structure will used based on what code it encounters that interacts with it. 

When you first declare a built-in structure or object, it is assumed to be immutable, fixed sized, rigid shaped (no runtime added attributes), 
materialized (contains no pointers, is contiguous and aligned), and non-concurrent.

If you change it or its contents, it becomes mutable. 
If you add elements to it (assuming its a container of some sort) it becomes resizeable.
If you reference assign to it becomes virtual (becomes a pointer) and if you reference assign to an element in it, its elements become virtual.
If you put an element in it that is of a different type than the elements that are already in it, it becomes heterogenous (containing pointers to different types).
If you add a new member to it, it becomes dynamic (dynamic composition; supporting adding and removing members at run time).
If its referenced across thread boundaries, it becomes concurrent (or atomic).
If you assign a value of a different type to it, it becomes a variant (a tagged union).

There is a refinement bitmask associated with each typed symbol:

    Immutable           | Mutable           - does anything change its value? (thing = or thing := or thing +=, etc.)
    Fixed               | Resizable         - does anything add or remove elements from it?
    Materialized        | Virtual           - are there any reference assignments?
    Raw                 | Arc               - are there are non-raw-qualified reference assignments?
    Homogenous          | Heterogenous      - are different typed elements added into it?
    Rigid               | Fluid             - does anything add attributes ("thing.someAttribute = ") to it that it doesn't initially posess?
    Serial              | Concurrent        - is it non-locally accessed from a spawn block?
    Singular            | Variant           - is this a var (not a specific type) and is it assigned to a value of more than one type?

For parameter refinement markers in function signatures, the Serial vs Concurrent bit has a slightly different meaning:

    Serial              | Concurrent        - does this function access this parameter from a spawn block or call any function that does?

In this way, a programmer can write very fast code in hot paths, and very ergonomic code everywhere else.

var is a way to declare and potentially initialize a variable that infers its type.

var v;                  - Not an error; v eventually takes on whatever type is assigned to it. And then it may morph after that. 

User~mrrmrs u           - u is ifrmrs, actually, but any attempt to morph it higher than mrrmrs will result in a compile time error.

Note that if an object is materialized (not a pointer), the raw/arc bit has no bearing. Raw/Arc is only relevant for pointers.

b. Subtypes

Some built in containers have subtypes to which only a subset of the refinements apply.

The queue, for example, has several lockless subtypes that are automatically concurrent:

spsc_queue myQueue1;    - lockless single producer single consumer concurrent queue; base refinement is cbmrc.
spmc_queue myQueue2;
mpsc_queue myQueue3;
mpmc_queue myQueue4;

c. Variants

var u = Employee.Retrieve(id);      - u is a variant<User, ErrorNotFound>
u = 99;                             - now u is a variant<User, ErrorNotFound, u8>
u.TailState = TailStates.Wagging;   - now u has an runtime added attribute of type enum TailStates
u.message = "woof!";                - another attribute of type sz8
User u = Employee.Retrieve(id);     - u is User, if ErrorNotFound was returned, u is null

if (u is User) {..}                 - built in is operator returns true if the tag typeid matches the supplied type

d. Morphs And Conversions of Numeric Types

When unrestricted, numeric types will upgrade in size (but not downgrade) and switch toward bigger floats (but not away from them).
When restricted, numeric types will perform conversions.

var n = 123;                        - n is a u8;
n = 3.1415                          - oops, nope, it was an f16.
n = 489029.28376                    - oh, actually it was an f64
n = -39                             - still f64

i8 i = 234;                         - this is a negative number. The i8 won't let it switch types or sizes.
i := 13;                            - it's now a shared pointer to a heap allocated i8. numeric variables will morph refinements.

mat i8 j = 96;                      - materialized i8.
j := 13;                            - error. Can't reference assign to a materialized variable.

raw i8 k = 97; 
var p := k;                         - p is also raw.   [Our symbols have an rmask, which is the minimum/current rmask.. I guess they also need a maxrmask]

arc q = k;                          - shared pointer q gets a heap allocated copy of k

d16 d = 19.90d                      - No floats are used for this initialization. 

For decimal initialization from a constant like this, the digits after the decimal are counted and the decimal is assigned 199/10.
If a math operation overflows during compile time initialization, the fraction is converted to its minimal integer form and the operation is re-attempted.
If it still overflows, a compile time error is emitted since this particular symbol is restricted from upgrading in size by having a specified type.
 
Decimal operations will emit error results, either at compile time or at run time, if they would result in garbage values.
The other numeric types just swallow conversion errors, but if you really need to, you can check a thread_local __carry and __overflow.
The code is not generated to store it for you unless you use it in the scope where the operation was done.

6. Ergonomic Control of Pointer Types

There are two types of assignments:

v = mylist[i]           - copy assignment.

There, v receives a copy of the i'th element if mylist. Maybe not super efficient, but good for isolation.

v := mylist[i];         - reference assignment. 

If v has local scope (is temporary), or lives in the same arena with mylist, it becomes a temporary raw pointer, and mylist is unchanged. 
If v is a class member or from a non-local scope, and does not live in the same arena with mylist, and was not qualified as raw, it becomes a shared pointer, and mylist becomes virtual. 
(Normally, things on the right side of an operator don't morph, but reference assignment is an exception.)

pointer declarations:

class Node {
    ref Node next;              - Shared ownership
    weak Node parent;           - Safe observer (becomes null if parent dies)
    raw Node peer;              - fast, unsafe
}

7. First Class Arenas

You can declare an arena, and put things into it, like this:

arena a1(A1_SIZE);      - the size parameter is optional, if omitted, the arena will use a project-configured arena size, or try to guess at what a reasonable size might be.

list myList:a1 = ["here", "is", "a", "bit", "of", "text"];    - initially a fixed length array of pointers, placed in a1, to zs8's also placed in a1.

An arena must have the same or greater scope/lifetime of what is put in it so that by the time the arena is destroyed, the things in it are no more.
If an object is in the same arena with something that gets assigned to it, won't morph into a reference counted object.

var v:a1 := mylist[i];   - here, since myList is also in a1, v becomes a raw pointer to the i'th element of mylist, and mylist is unchanged.

Whether or not runtime subscript boundary checks are done is configured in the project file. Eg., globally they might not be, but in "debug" they might be.
By default they are not.

If you remove things from mylist, since it is in an arena, the count of items goes down, but the allocated size remains the same. You can't deallocate from an arena.
You can grow things in there, but you can't shrink them.


8. Comprehensions

List and dictionary comprehension syntax and "native" json syntax is borrowed from Python, but the implementations morph from ~imfrhrss to whatever ends up being required.

var l = [upper(s) for s in name];   // list comprehension. Generates a list of whatever type of elements are in name. Initially ~irmrhrss.

Let's say name is a zs8 and its elements are c8's.

l = [n*n for n in range(10)];       - now the elements of l are variants. Either c8's or integers. Not exactly ideal, but possible.

l[6] = "woohoo!"                    - now the possible variants are u8, c8 or zs8.

var v = l[6]                        - the variable v shadows whatever v was before. Now it's a variant, for the rest of the scope.

if (v == 6*6) { print("maybe, but no"); }
else if (v == "woohoo!") { print("there you go"); }
else if (v == someObjectInstance) { print("nope"); }     - comparison's don't morph things, they just return false if they don't make sense.

Functions may specify types they require, or they may take var types:

class User {
    
    sz8 name;
    list inventory;

    SetName(sz8 name)           - only sz8's can be passed. Passing something else is a compile time error. They're passed by reference, since they're not simple types.
    {
        me.name = name;                 - name gets a copy of the reference-passed name
        LastAssignedName := name;       - the (declared elsewhere) global, LastAssignedName, becomes a shared pointer, if it wasn't already. Lots of cascading compiler work.
    }
   
    AddInventoryItem(var thing)
    {
        inventory.add(thing)    - inventory's items will become variants (tagged unions) of each type of thing that is passed in here.
    }
}

..

ref ss8 LastAssignedName;       - Now the compiler knows ahead of time that this is a shared pointer to a zs8. Shorter build times. And it won't let you morph this.

ref var LastAssignedItem;       - This is a shared pointer to a morphing variant. Fun fun.


9. First Class null

Each built-in type has a null value.

var n = null;                   - the compiler has no idea what this is yet. Defer judgement.
if (n) DoSomething();           - n has no type yet. More deferring. 
n = true;                       - n is now a b8, so the above null becomes false, and the if statement can be coded (with a vector of machine code block ids)
if (n) DoSomething();           - coded without any deferral.

null is mapped to a value for each simple type. 
For materialize (non-pointer) structs and classes, assignment to null causes them to become fluid, and get a b8 attribute, __assigned, which is set to false.
For pointers, assignment to null sets them to 0.

User u = null;                  - a reference to a default constructed optional User instance that is flagged as unassigned.

u.SetName("Kristy")             - ignored at run time, because of the "unassigned" flag. Source of bugs, yes. Maybe static analysis could warn about it.



Resolving inference conflicts:

var okidokey = null;        - deferred because of the var, and null's type is ambiguous. Deferreds that are never resolved become noops.

User u;                     - Temporary. On the stack. Maybe not terribly useful in this case, but legal.

u.SetName(okidokey)         - okidokey becomes an zs8 (at compile time). So the above code assigns an empty string, and this passes an empty string to SetName. 
                              if okidokey is reference assigned in SetName, it becomes a virtual/arc - a shared pointer. 

okidokey = "Sam";           - "Sam" is copied into okidokey; if okidokey is materialized, this won't affect things in SetName. 
                              If it's virutal, things in SetName may be affected.

okidokey = 99;              - okidokey becomes a variant with zs8 and u8 components. 
                              the zs8 component is still null, and that is the component that is passed to u.SetName above.

okidokey = 1023985472       - inferred simple types will morph to larger versions. The u8 component becomes a u32. It won't morph back down.
                              specifically declared simple types won't morph to larger versions.

okidokey = MyObj.Create()   - This static factory returns an instance of MyObj. Reference? or copy? depends on optimization.
                              But okidokey's tagged union ends up with a materialized instance of a MyObj. (Copy assignment.) 

okidokey := MyObj.Create()  - The MyObj in okidokey becomes a shared pointer (or a raw pointer if there is an arena in common) instead of a materialized instance. 
                              When these statements are executed at run time, the assignment above will copy the value into an allocated space pointed at by the shared pointer.
                              
other.Set(okidokey)         - If Set takes a specific type that is not in okidokey's union, this passes a null value of that type into the function.
                              If Set takes a var, any operators used in Set may morph their lvalues.

onemore := okidokey         - if onemore is a var, it becomes a shared pointer to a tagged union, and okidokey becomes a shared pointer if it wasn't already. 
                              If onemore is a specific type that isn't in okidokey's tagged union, its value becomes null and okidokey is unaltered: 
                              Assignment (and other operators) don't morph the right side except to make them into shared pointers when called for.                


Passing things into functions may morph things inside those functions, but it does not morph what you're passing into it unless it has not yet been assigned a type.
Assignments and other binary operators may morph things on their left side, but they don't morph things on their right side,
except that reference assignments to things that are not shared pointers morph them into shared pointers, unless both sides live in the same arena.
Copy assignments into shared pointers decref whatever they were pointing at, create a new allocation, and point the shared pointer at it.
Copy assignments of arena managed objects into same-arena-managed raw pointers simply change where the raw pointer is pointing.

Simple types may change size, as mentioned:

c8 c = 'a';
zs16 s = c;                 - s remains an zs16, and it now contains a single c16 whose value is 'a'.
c = s;                      - c will not switch its base type or become variant because it was declared with a specific type. 
                              c gets the least significant byte of the first character in s. 
                              This potentially loses data and will emit a warning.

var z := "stack zs8"        - z is a shared pointer that points at a heap allocation that contains the string. 

10. Error Handling

a. The error Class Factory

There is a special built in error construct that facilitates this kind of syntax:

error FileOpenFailed(zs8 message);              // class with an inferred constructor, member, functor that inherits from XmcError
                                                // the functor returns a cloned instance of the class

This declares an error class with a constructor that takes a single parameter that is automatically forwarded to a member of the same name and type.
To instantiate this error, you would type something like this:

var result = FileOpenFailed(someErrorStr);      // err.message will have a copy of the value of someErrorStr

or this:
 
error err := FileOpenFailed();    // shared pointer to a heap allcoation. When parameters are left out of function calls, null is passed for them. err.message is "".

b. Error Blocks

There is a special block type, which, if any function in that block returns a variant whose tagged value type inhertis from error, it immediately exits the block:

{? 
    var file = open("data.txt"); // If this returns Error, skip to the end of {?}
    var head = file.read(10);    // This line never runs if 'open' failed
    print(head);                 // Neither does this
?} 


c. Error Operator

var u = database.find(id) | { errout("database.find failed: {u.message}"; return; }

User u = u;
u.login();


11. Nested Scope Labels and Break;

{:main
    {:nested
        if (someCondition) break :main;
        if (otherCondition) break :nested;
        DoStuff();
    }
    DoOtherStuff();
}

12. Reaching

You can "reach" into a nested scope after the scope to pull out values

if (file is FileOPenFailed) { errout("failed to open data.txt: {file.message}\n"); return; }

Variables in nested scopes are accessible in the parent scope afterward, but if the execution that populates them is skipped, their value will be null.

13. Shadowing

Variables can also be shadowed:

var u = database.find(id); // Morphic: <User | NotFound | Timeout>
if (u is User) {
    User u = u;            // Shadow: Inside this if, u is strictly a User.
    u.login();
}
   
// out here, u is the variant returned from database.find.

If you shadow a variable in the same scope, it is replaced by its shadow for the remainder of that scope.


14. External Call Definitions

// Explicit ABI mapping
extern kernel32:WriteFile(
    hFile:          u64 -> rcx,         // materialized u64 passed in RCX..
    lpBuffer:       raw u8[] -> rdx,    // raw (non-arc pointer to a) u8 array passed in RDX

15. Functions

a. Named Functions

The general function syntax borrows from C and C++.

type SomeFunction(arguments...) {       // arguments separated by commas
    statements                          // statements separated by semicolons
    return somevalue;
}

There's no "void" type, for a void function you just skip the type.

SomeProcedure(arg1, arg2){
    DoSomeThings();
    ..
}

You don't have to return, if you don't, null is returned. You can return without specifying a value, doing that returns null.

b. Anonymous Functions

There is no capture block specified for anonymous functions.

(arguments){statements}

The capture block is inferred based on what is used. Whether a capture is by value or reference depends on its refined type taking into consideration how its used.

c. Variadic Functions

print(arg1, arg2, args...){         
    // arg1 and arg2 are named within the function
    // argc has the anonymous argument count
    // argv has the anonymous arguments, it is used syntactically like an array of variants whose type list depends on the types of arguments that were passed
    // (exactly how argv is implemented depends on the context, hopefully we can come up with some optimizations)
}

d. Parameter Passing Syntax

Parameters can be ommitted from right to left. They can omitted from left to right by naming a parameter.

ThisProc(zs8 name, zs8 handle, u8 age){..}

ThisProc("SuzyQ");

ThisProc(handle="QueenB", 27);

e. Function overloading

If a function has a different return value or different parameters it is recognized as a different function.

zs8 TransformString(arglist);                                           //
s8 TransformString(arglist) { return TransformString(arglist); }        // different function, same arglist, converts a zs8 to an s8 and returns it.
var s = TransformString(arglist);                                       // ambiguous call error.
s8 s = TransformString(arglist);                                        // maybe not terribly efficient, but it works fine.

16. Structs

a. Struct Declaration

Struct syntax is borrowed from C/C++, for the most part.

struct TypeName         // TypeName is skippable, in which case we have an anonymous ustruct type
{
    typename variablename = defaultvalue;
    ..
} instanceName;         // instanceName is skippable, in which case nothing is instantiated.

Structs don't have function members, per se, but they can have named pointers to functions.

b. Instantiation and Initialization

A struct always has a default constructor that takes arguments matching its member list.

var s = SomeStruct {membervalue1, ...};         // initialization block parameters are assigned to struct members in the order they appear in the struct declaration
                                                // they can be omitted right to left, ommitted values become null

SomeStruct s = {membervalue1, membername3 = membervalue3, membervalue4, ..};       // they can be named, they can be skipped left to right by naming them

s = {arglist..};                                // they can be reassigned later using brace syntax.

s = [{arglist..}, {arglist..}, ..][selector];                                       // list comprehension used to select an assignment specifier
s = {"this": {arglist..}, "that": {arglist..}, "other": {arglist..}}[selector];     // dictionary comprehension used to select an assignment specifier

c. implicit construction

You can pass construction parameters to functions that take a struct just like you can in C++.

d. Struct Inheritance

A struct can inherit from multiple other structs.



17. Classes

Classes are the same as structs except they have function members, and you can supply coded constructors and destructors.

class MyClass {
    MyClass() {};
    ~MyClass {};
}


18. Generators, ranges and slices

a. generators

i8 MyGen(){
    for (i = 1; i<10; i++) yield i;     // you can say for (myvar ... ) and it'll infer the type
}

The generated i8's from the above generator can be consumed, for example, like this:

    for (n in Mygen()) print(n);        // you can also say "myvar in .." and it'll infer the type

    var n := MyGen.start();
    while (!MyGen.finished) {
        n := MyGen.next();
    }

The compiler merges the generator emmitted code with the consumer emmitted code for you.

(I wonder if Gemini and I, between the two of us, will be smart enough to figure out a way
to perform that merging in a way that will fairly reliably result in efficient emitted code.)

var l = [print(n) for n in MyGen()];        // list comprehensions are actually generators - so this is a generator within a generator.


b. Ranges and slices

r = range(1, n);            // r is a pointer to an unmerged generator
for x in r;                 // merge happens here

x = [1, 2, 3, 4][1:]        // this is just a list slice (~ifmrhrss, so array, basically) slice, from the 2nd element to the end
"hello world!"[x:-1]        // string slice from position x to the 2nd from the last position.


19. Threading

- spawn blocks

var myPromise = spawn DoStuff(arg1, arg2);  
var result = myPromise.Wait();

Here, myList is being accessed from a spwan block, which may trigger refinement morphing:

var myPromise = spawn {
    DoStuff(myList);        
}


Here, myList may not need to morph:

var myPromise = spawn {
    list myList;
    DoStuff(myList);        // might produce morphing or might not, depends on parameter rmask
}

- parameter refinements

Parameters in function signatures have minimum and maximum refinement masks which represent the nimimum refinements required and maximum refinements 
allowed for that function. By default a parameter is marked as serial, in the function, it is reference assigned into any non-local, or passed to another 
function in a parameter with a concurrent requirement, then the parameter is marked as concurrent. 

Literals are refinement-limited to being materialized (non-virtual, that is, non-reference assignable). 
If a literal is passed to a function that reference assigns that parameter, an error is thrown: "Error: [funcname]-[argname] must be reference-assignable."
The error should show the source code line (with filename and line number) that creates the rule and the one that violates it.


20. Implementation Notes

- compiler input and output

When the compiler is invoked, the command line is read and the project file identified on it is read.

XMC compilation of a project produces a single .obj for that target. The .xmo files exist to speed up incremental compilation.
So we have these .xmo files that have the declared things, their types and their minimum and maximum refinements, 
and indexed blocks of emitted machine code for the module.

So if one file is modified and no other files are dependent on it, only that file would need to be parsed.
We wouldn't need to load or parse the source for any other units, just their .xmo's.
The .obj can be reconstitued from the .xmo's. 
In fact that's how it's constituted to begin with; the xmo's are produced in memory, then the .obj is pieced together from the xmo's.

If all we're doing is changing refinement masks, we can reconstitute a .obj without parsing the source code at all.
But the question is, can we determine if morphing a type of something needs to cause something else's type to morph from just the xmo?
Nope. So the finalized parse trees from the previous compilation are also saved in the .xmo files, but that part of an xmo doesn't need to 
be loaded unless we need it.

- incremental compilation

Here's the incremental compilation process:

The source folders are walked and filenames and modtimes are loaded.
Files whose .xm's are newer than their .xmo's (or whose .xmo's aren't present) and considered modified.
The symbols, their types, their min and max refinements, and the indexed machine code blocks are loaded from all the unfmodified .xmo's.
The memory buffer for an .xmo is sized so that the entire .xmo can be loaded if needed.
The source code is loaded from the modified xm's.
The parse trees are regenerated for the modified xm's and added to the .xmo's in memory.
    - symbols are interned, their base types are determined, and refinements are morphed upward from defaults.
    - this doesn't happen by forward scanning or pre-scanning, it happens by putting information in "pending" structures and back filling.
    - If morphing of refinements spreads to unmodified xmo's, this triggers a full xmo scan to redetermine all symbol refinements. (see "full xmo scan" below)
Once symbol refinements are determined, a second scan of the loaded xmo parse trees is performed to generate vectors of code block id's
for each parse tree. (see "code block generation" below)
The xmo's that are marked as dirty are saved at the end of the compilation process. 
Once xmo's are finalized and saved, the coff file is written out and the linker is invoked.

Full xmo scan:

If a full xmo scan is triggered, the parse trees for all the unmodified .xmo's are loaded, and a single scan of all the xmo parse trees
is performed to remorph all the symbol refinements. A hashset is cleared and then filled during this, to determine which symbols have not yet been
visited and therefore remain at their default levels. Symbol refinements are copied from the global symbol table to the Xmo's that export or import them.
Xmo's whose symbol refinements are changed by this are marked as dirty.


- full build

Here is the full build compilation process -

The command line is parsed, the project file is parsed.
The intermediate directory is cleared, the .obj file and the .exe are deleted.
The source folders are walked and the .xm files are loaded.
An .xmo file is created in memory for each .xm file.
Parse trees are generated for each xmo.
    - Symbols are interned, their base types are determined, and their refinements are determend (morphed upward from defaults).
The parse trees are rescaned and code block id vectors are put into the parse trees.
The code block buffers of the xmo files are populated.
The xmo's are saved.
The coff file is written.


Symbol dependency vectors:

The symbol table contains vectors of symbols whose refinenments depend on that symbol's refinement. This information is saved in the xmo.
A symbol has a scope id vector that has the file id and the scope ids for the namespace scopes in the symbol's namespace path.


- code block generation

Code blocks are currently defined like this:

	// 32 bytes total:
	struct CodeBlock
	{
		uint32_t reserveMask;
		int32_t	 altTableIndex;		// 0 for the last block in the alternate chain, which is either the only available block or the spill-to-stack block.
		uint8_t  cost;
		uint16_t codeSize;
		uint8_t  patchOffset;
		uint8_t  code[19];			 
	};


The reserveMask is the registers used by the block:

	Standard GPR Register Mask (32-bit)

	Bit	Hex Value	Reg (64-bit)	Reg (32-bit)	Typical Role
	0	0x0001		RAX				EAX				Return Value / Accumulator
	1	0x0002		RCX				ECX				1st Argument (Windows)
	2	0x0004		RDX				EDX				2nd Argument (Windows)
	3	0x0008		RBX				EBX				Callee-Saved (Preserve!)
	4	0x0010		RSP				ESP				Stack Pointer (Always Reserved)
	5	0x0020		RBP				EBP				Base/Frame Pointer
	6	0x0040		RSI				ESI				3rd Argument (Linux) / Caller-Saved
	7	0x0080		RDI				EDI				4th Argument (Linux) / Caller-Saved
	8	0x0100		R8				R8D				3rd Argument (Windows)
	9	0x0200		R9				R9D				4th Argument (Windows)
	10	0x0400		R10				R10D			Temporary / Syscall
	11	0x0800		R11				R11D			Temporary
	12	0x1000		R12				R12D			Callee-Saved
	13	0x2000		R13				R13D			Callee-Saved
	14	0x4000		R14				R14D			Callee-Saved
	15	0x8000		R15				R15D			Callee-Saved

		// [id: prologue, reserve: 0x10 (RSP), release: 0x00, patch: 3]
		// 48 83 EC [XX] -> sub rsp, XX
		{
			0x10,           // reserveMask (RSP)
			0,              // altTableIndex (Last in chain)
			1,				// cost
			4,              // codeSize
			3,              // patchOffset (points to the immediate byte)
			{ 0x48, 0x83, 0xEC, 0x00 } // Machine code
		},

		// [id: epilogue, reserve: 0x10 (RSP), release: 0x10, patch: 3]
		// 48 83 C4 [00] -> add rsp, 
		// C3 -> ret
		{
			0x10,           // reserveMask (RSP)
			0,              // altTableIndex
			2,				// cost
			5,              // codeSize
			3,              // patchOffset (points to the immediate byte)
			{ 0x48, 0x83, 0xC4, 0x00, 0xC3 } // Machine code
		},

		// [id: lea_rcx_rel (2), reserve: 0x02 (RCX), release: 0x00, patch: 3]
		// 48 8D 0D [00 00 00 00] -> lea rcx, [rel ?]
		{
			0x02,           // reserveMask (RCX)
			10,             // altTableIndex
			1,				 // cost
			7,              // codeSize
			3,              // patchOffset (displacement starts at byte 3)
			{ 0x48, 0x8D, 0x0D, 0x00, 0x00, 0x00, 0x00 }
		},

		// [id: call_rel32 (3), reserve: 0x00, release: 0x00, patch: 1]
		// E8 [00 00 00 00] -> call ?

			0x00,           // reserveMask (standard calls handle their own regs)
			0,              // altTableIndex
			10,				// cost

			0,				// altTableIndex
			0,				// Cost (Idiomatic zeroing is often free)

There is also a data/AltCodeBlocks.h where we plan to set up alternate versions of the same code blocks that use different registers, 
or which spill to stack and recover them before using the same ones.


Currently in the emitting steps, emmitter.cpp::UpdateXmoCode function runs ProcessNode (which should probably be renamed) in parallel per xmo and that
recusursively scans an xmo parse tree an constructs the code block buffers in the xmo's from parse tree node code block id vectors.
Then WriteToCoff runs in a single thread and goes through all the xmo's in sequence and writes out the coff file including 
a single .text segment with all the machine code.

So the job of fitting the block id vectors together so that things are optimized and registers aren't clobbered, etc., falls to the 
parsing step.


- Roadmap / implementation work flow

a. Produce and ratify a BNF file for the core language syntax
b. Update implementation plans and update and test the existing data structures and process code (without adding parsing yet) to support the intended features
c. Create sufficient parsing to produce syntax errors / validate .xm files.
d. Introduce further parsing to intern symbols and determine refinements and produce type and morphing related errrors.
e. Iteratively create further parsing and updates and modifications to the primary and alternate code blocks to produce optimized code block id vectors in parse nodes.
    - generate a source code snippet to demonstrate and test a language feature
    - produce parsing that can facilate compilation and linking of the core language test program with the addition of that snippet
    - compile and execute the test program and do any debugging that's required
    - produce similar functionality in a C++ test program (in release mode with full optimizations) and compare the compile time and the dissassembled output 
    - improve parsing until all the feature tests that have been produced so far compile quickly and produce optimal code
f. Create syntax highlighting for use in Visual Code and Visual Studio
g. Add support to the coff file writing for including debug information sufficient for use by Visual Studio to be able to set break points and step and trace.
h. Produce a small standard library and some tests for it
i. Produce a set of wrappers for commonly used C libraries in game development, like OpenGL, Vulkan, stb_image, etc. and some tests for them
