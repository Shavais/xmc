The XMC Programming Language - Revision 2.

================================================================
Part I — Introduction
================================================================

1. AOT Compilation

- Compiles ahead of time straight to native code; does not use LLVM or any IL.
- Early compiler versions are written in C++ using platform SDKs, they write out COFF files to be linked by platform linkers.


2. Efficient, Deterministic Memory Management

- Does not garbage collect.
- Has arenas and release pools as first class objects.
- Objects not explicitly marked as raw that are not stored on the stack or in an arena or a release pool are reference counted and discarded immediately when the reference count reaches zero.


3. Global Code Visibility

- Anything in a global or namespaced scope that is anywhere in the code base is accessible from anywhere else, no includes, imports, or "using namespace" statements are required.
- Things can be put into namespaces, but when referencing something, its namespace path only needs to be specified when there is ambiguity otherwise.
- The translation unit (source code file) name is an optionally-specified first segment of a namespace path, so classes or functions of the same name can exist at the root of different files.
- Namespace path prefixes can be shortened to the shortest set of segments that eliminates ambiguity.
- Object files (.xmo) are created for translation units (.xm source files) during compilation to speed up future compilation.

Symbols are stored with their scope id's. In a sharded global symbol table and in .xmo file exports, together with their scope id.
There is a global tree of scopes. 
A given scope has a stack of enherited scope id's.


4. Project Based Build System

An xmc target is either an executable, a static library, or a dynamic library.

Command line invocation:

    xmc <project> [config] [file_filter] [--full]
    xmc <project> [config] [file_filter] --test [test_filter] [--full]
    xmc <project> [config] [file_filter] --suite [--full]

    project       The name of the project. The compiler looks for <project>.xmc in
                  the current directory.

    config        The configuration section to activate (e.g. debug, release).
                  If omitted, the section marked :default in the project file is used.

    file_filter   Optional wildcard (*, ?) matched against source file stems. When
                  present, one .obj and .exe is produced per matching source file,
                  named after that file, in the output folder.  Without a filter,
                  all sources are compiled into a single .obj named after the project.

    --full        Forces a full rebuild, ignoring .xmo intermediate state.

    --test [test_filter]
                  Compiles with a fabricated main() that enumerates and runs test
                  blocks declared in the source. If test_filter is supplied (a wildcard
                  matched against test block names; namespace paths use a single colon
                  as separator, e.g. Math:Vector*), only matching test blocks are run.
                  The resulting executable(s) are run and their output reported.

    --suite       Compiles and runs each matching source file as a standalone program
                  (each must have an entry point matching the EntryPoint key in the
                  project file). The compiler's component logs and the program's stdout
                  and stderr are compared against golden files found alongside the
                  source file:

                    <sourcefile>.lexer    
                    <sourcefile>.parser   
                    <sourcefile>.morpher
                    <sourcefile>.stdout   
                    <sourcefile>.stderr

                  A pass/fail report is produced for per source file per golden file that exists.

Examples:

    xmc hello debug                     normal build, debug config
    xmc hello debug --full              force full rebuild
    xmc hello debug *.xm --suite        run regression suite over all source files
    xmc hello debug Lexer* --test       run test blocks in files matching Lexer*
    xmc hello debug Main --test Scan*   run test blocks matching Scan* in Main.xm


There is a project file which is a <project>.xmc file (e.g. hello.xmc).
The source code is in .xm files.
A root folder for the source tree can be specified in the project file via SourceRoot.
The Sources key lists source files or glob patterns, relative to SourceRoot if set.
A single linkable output (.obj on Windows) is created for the project in IntPath,
named after the project file (e.g. hello.obj). When a file_filter is active, one
.obj is produced per matched file.
The platform linker is invoked by the compiler.
The compiler places the linked output (.exe, etc.) in OutPath.

Component logging is enabled per-component in the project file. The value may be 1
(write logs to IntPath) or a path string overriding the output directory.

Sample project file:
_________
# hello.xmc
[base]

[debug:default]

[release]

5. Compile Time Inferred Type Morphing And Conversions

a. built-in types and their refinements:

boolean:    b
integer:    i8, i16, i32, i64, u8, u16, u32, u64
float:      f32, f64
decimal:    d16, d32, d64       - fraction type that maintains an integer numerator and denominator (and doesn't incur floating point representation errors)
character:  c8, c16,            - single byte character, double byte character
            zs8, zs16, zs32     - zero-terminated string of c8's, of c16's
            s8, s16,            - length-specified (pascal) string of c8's, of c16's
            utf8, us            - unicode strings

atomic  - qualifier that yields atomic versions of a simple type. (eg. atomic boolean) 

built in structures and objects:

struct      - no functions (except as pointers), no coded constructors or destructors
class
list
queue
stack
btree       - binary tree
hashset     
hashmap
function
arena
heap
(others to be determined)

concurrent - qualifier that yields a concurrent version of a container 

These built-ins, structures and objects have various versions that the compiler infers between. 

The compiler revises it's inference of exactly what of version the structure will be used based on what code it encounters that interacts with it. 

When you first declare a built-in structure or object, it is assumed to be immutable, fixed sized, materialized (contains no pointers, is contiguous 
and aligned), raw (as opposed to reference counted), rigid shaped (no runtime added attributes), and non-concurrent.

If you change it or its contents, it becomes mutable. 
If you add elements to it (assuming its a container of some sort) it becomes resizeable.
If you reference assign to it becomes virtual (it becomes a pointer) and if you reference assign to an element in it, its elements become virtual.
If you cause it to become virtual, it is reference counted unless it is declared as being stored in a particula 
If you add an element into it that is of a different type from the elements already in it, it becomes heterogenous (containing pointers to different types).
If you add a new member to it, it becomes fluid (it supports adding and removing members at run time).
If its referenced across thread boundaries, it becomes concurrent (or atomic).
If you assign a value of a different type to it, it becomes a variant (a tagged union).

There is a refinement bitmask associated with each typed symbol:

    immutable           | mutable           - does anything change its value? (thing = or thing := or thing +=, etc.)
    fixed               | resizable         - does anything add or remove elements from it?
    materialized        | virtual           - are there any reference assignments?
    raw                 | arc               - are there are non-raw-qualified reference assignments?
    homogenous          | heterogenous      - are different typed elements added into it?
    rigid               | fluid             - does anything add attributes ("thing.someAttribute = ") to it that it doesn't initially posess?
    serial              | concurrent        - is it non-locally accessed from a spawn block?
    singular            | variant           - is this a var (not a specific type) and is it assigned to a value of more than one type?

For parameter refinement markers in function signatures, the Serial vs Concurrent bit has a slightly different meaning:

    Serial              | Concurrent        - does this function access this parameter from a spawn block or call any function that does?

The intent is for a programmer to be able to achieve C-like speed for hot paths, and python-like ergonomics elsewhere.
(With xmc, maybe you can write your game in the same language that was used to write your game engine without spending quite so many years doing it.)

var is a way to declare and potentially initialize a variable that infers its type and allows refinement morphing.

var v;                  // Not an error; v eventually takes on whatever type is assigned to it. And then it may morph after that. 

In most circumstances, the keyword "var" is the presumed default and can be skipped.

v = 13.0;               // if "v" doesn't exist already, this is the declaration, and it's var.

The keyword "ref" declares a variable whose type and full refinement mask are inferred from its first assignment, then frozen — no further morphing in any dimension. 
Where var sets the minrmask from inferred refinements but leaves maxrmask wide open, ref snapshots the rmask after first assignment and sets minrmask and maxrmask 
both equal to that snapshot. This is the form to reach for when you want inference convenience without the risk of subsequent morphing — function references in particular
always use ref (see §18a), but it's appropriate any time you want "whatever the compiler decided this is, lock it down."

ref u = CreateUser(id);     // u takes the type and refinements implied by CreateUser's return at this site, then is frozen.
ref x = 5;                  // x is u8, will not morph to a larger numeric type.

An "ugly mask" can optionally be specified when declaring a variable, that constrains morphing. The bits are specified from left to write
using the first letters of the words above from top to bottom, with the left side words corresponding to 0's in the bitmask, and right side words
corresponding to 1's. The letter for "Heterogenous" is e, since h is used for it's opposite, "Homogenous."

User~mrvaefcs u;        // the maximum refinement mask is all 1's here, except for the final one, Singular (non-Variant)

Attempts to assign a value of a different type to u are ignored. This "max refinement mask", of Singular, by the way, is the default for type-specified 
variables. Assignments of a variant to a singular type narrow to the singular type, or if they cannot, a warning is produced and the assignment is ignored.

Note that if an object is materialized (not a pointer), the raw/arc bit has no bearing. Raw/Arc is only relevant for pointers.

Variables and parameters have minimum and maximum refinement masks, referred to as minrmask and maxrmask in the code and in this documentation.
The rmask bit qualifiers in the lists above can be used to eiter pull the maxrmask down, using a qualifier from the 0, left side list, or bump
a minrmask up using a qualifier from the 1, right side list.

raw p := User();        // raw pointer to a heap allocated constructed User struct or class, or to whatever is returned by the function named User.

That sets the raw/virtual bit of the maxrmask to 0, preventing p from morphing to arc.

b. Subtypes

Some built in containers have subtypes with implied refinement mask bits. The queue, for example, has several lockless subtypes that are automatically concurrent:

spsc_queue myQueue1;    // lockless single producer single consumer concurrent queue, minrmask has the concurrent bit set
spmc_queue myQueue2;
mpsc_queue myQueue3;
mpmc_queue myQueue4;

c. Variants

var u = Employee:Retrieve(id);      // u is a variant<User, ErrorNotFound>
u = 99;                             // now u is a variant<User, ErrorNotFound, u8>
u.TailState = TailStates:Wagging;   // now u has an runtime added attribute of type enum TailStates
u.message = "woof!";                // another attribute of type zs8
User u = Employee:Retrieve(id);     // u is User, if ErrorNotFound was returned, u is null

if (u is User) {..}                 // built in is operator returns true if the tag typeid matches the supplied type

d. Morphs And Conversions of Numeric Types

When unrestricted, numeric types will upgrade in size (but not downgrade) and switch toward bigger floats (but not away from them).
When restricted, numeric types will perform conversions.

var n = 123;                        // n is a u8;
n = 3.1415                          // oops, nope, it was an f16.
n = 489029.28376                    // oh, actually it was an f64
n = -39                             // still f64

i8 i = 234;                         // this is a negative number. The i8 won't let it switch types or sizes.
i := 13;                            // it's now a shared pointer to a heap allocated i8. numeric variables will morph refinements.

mat i8 j = 96;                      // materialized i8.
j := 13;                            // error. Can't reference assign to a materialized variable.

raw i8 k = 97; 
var p := k;                         // p is also raw.   [Our symbols have an rmask, which is the minimum/current rmask.. I guess they also need a maxrmask]

arc q = k;                          // shared pointer q gets a heap allocated copy of k

d16 d = 19.90d                      // No floats are used for this initialization. 

For decimal initialization from a constant like this, the digits after the decimal are counted and the decimal is assigned 199/10.
If a math operation overflows during compile time initialization, the fraction is converted to its minimal integer form and the operation is re-attempted.
If it still overflows, a compile time error is emitted since this particular symbol is restricted from upgrading in size by having a specified type.
 
Decimal operations will emit error results, either at compile time or at run time, if they would result in garbage values.
The other numeric types just swallow conversion errors, but if you really need to, you can check a thread_local __carry and __overflow.
The code is not generated to store it for you unless you use it in the scope where the operation was done.

e. Composite literals

xmc has two literal forms for composite values: list-style with square brackets, and dict-or-initializer-style with curly braces.

List literals use square brackets:

var nums = [1, 2, 3];               // a list of three i32, default rmask (initially fixed)
var names = ["Alice", "Bob"];       // a list of two zs8

A list literal produces an immutable rigid singular list — fixed-size initially, default refinements — until usage forces the morpher to push refinement bits up. Adding to the 
list (.add(...)) morphs it to resizable; reference-assigning it across vault boundaries morphs it to arc; concurrent access morphs it to concurrent; and so on (§10 for
vaults, §24 for the morphing rules in detail). The literal is the seed; the morpher decides what concrete representation the variable ends up with. 

Anonymous struct literals use curly braces, distinguished by .fieldname = value form:

var p = { .x = 1, .y = 2 };
var rec = { .name = "Alice", .age = 30, .active = true };

A literal whose elements all use .field = value form produces an anonymous struct value (§23e) — a struct whose type was synthesized by the compiler with the named fields,
in declaration order, with types inferred from the values. The variable's inferred type is this anonymous struct type with default rmasks. Mixing .field = value form with
positional or key: value elements within the same literal is an error: a literal is one of list, dict, or anonymous struct, not a hybrid.

Dict literals use curly braces, distinguished by the presence of key: value pairs: var ages = { "alice": 30, "bob": 25 };

var lookup = { 1: "first", 2: "second" };

A dict literal produces an immutable rigid singular dictionary with the morpher's default representation (an unordered hash table). As with list literals, usage drives
morphing — adding entries makes it resizable, requesting iteration in sorted order makes it sorted, and so on.

Initializer lists also use curly braces, in declaration-assignment contexts where the type on the left determines the shape:

i32 a[5] = { 1, 2, 3, 4, 5 };       // explicit array initializer
u8 buf[1024] = { 0 };               // fill with zero
i32 a[10] = { 1, 2 };               // first two elements, rest zero
Point p = { 1, 2 };                 // struct initializer, positional
Point p = { .x = 1, .y = 2 };       // struct initializer, designated

In a typed declaration-assignment context, { ... } always means "initializer for the declared type's aggregate shape" — never a dict literal. The declared type tells the
compiler what to do with the elements: array elements, struct fields, etc. 

Disambiguation in inferred contexts. When the compiler infers the type from a curly-braced literal (var x = { ... };), it picks based on the elements' shape:

All elements are .fieldname = value form → anonymous struct.
All elements are key: value form (key is any expression) → dict.
No elements have keyed form → list.
Mixed forms → error.
var s = { .x = 1, .y = 2 };       // anonymous struct, no warning
var d = { "a": 1, "b": 2 };       // dict, no warning
var a = { 1, 2, 3 };              // list, no warning
var x = { .x = 1, "b": 2 };       // error: mixed forms
var e = {};                       // empty dict (the empty list literal is [])

Choosing between bracket forms. [ ... ] and { ... } (in inferred-list context) both produce list values, and both start with the same default rmask. 
The difference is intent: [ ... ] reads as "I have a sequence of values"; { ... } reads as "I am initializing storage." Use whichever matches your intent — the compiler 
does the same thing with both.


6. First Class null

Each built-in type has a null value.

var n = null;                   // the compiler has no idea what this is yet. Defer judgement.
if (n) DoSomething();           // n has no type yet. More deferring. 
n = true;                       // n is now a b, so the above null becomes false, and the if statement can be coded (with a vector of machine code block ids)
if (n) DoSomething();           // coded without any deferral.

null is mapped to a value for each simple type. 
For materialize (non-pointer) structs and classes, assignment to null causes them to become fluid, and get a b attribute, __assigned, which is set to false.
For pointers, assignment to null sets them to 0.

User u = null;                  // a reference to a default constructed optional User instance that is flagged as unassigned.

u.SetName("Kristy")             // ignored at run time, because of the "unassigned" flag. Source of bugs, yes. Maybe static analysis could warn about it.



Resolving inference conflicts:

var okidokey = null;        // deferred because of the var, and null's type is ambiguous. Deferreds that are never resolved become noops.

User u;                     // Temporary. On the stack. Maybe not terribly useful in this case, but legal.

u.SetName(okidokey)         // okidokey becomes an zs8 (at compile time). So the above code assigns an empty string, and this passes an empty string to SetName. 
                            // if okidokey is reference assigned in SetName, it becomes a virtual/arc - a shared pointer. 

okidokey = "Sam";           // "Sam" is copied into okidokey; if okidokey is materialized, this won't affect things in SetName. 
                            // If it's virutal, things in SetName may be affected.

okidokey = 99;              // okidokey becomes a variant with zs8 and u8 components. 
                            // the zs8 component is still null, and that is the component that is passed to u.SetName above.

okidokey = 1023985472;      // inferred simple types will morph to larger versions. The u8 component becomes a u32. It won't morph back down.
                            // specifically declared simple types won't morph to larger versions.

okidokey = MyObj:Create();  // This static factory returns an instance of MyObj. Reference? or copy? depends on optimization.
                            // But okidokey's tagged union ends up with a materialized instance of a MyObj. Copy assignment. 

okidokey := MyObj:Create(); // The MyObj in okidokey becomes a shared pointer (or a raw pointer if there is an arena in common) instead of a materialized instance. 
                            // When these statements are executed at run time, the assignment above will copy the value into an allocated space pointed at by the shared pointer.
                              
other.Set(okidokey);        // if Set takes a specific type that is not in okidokey's union, this passes a null value of that type into the function.
                            // If Set takes a var, any operators used in Set may morph their lvalues.

onemore := okidokey         // if onemore is a var, it becomes a shared pointer to a tagged union, and okidokey becomes a shared pointer if it wasn't already. 
                            // If onemore is a specific type that isn't in okidokey's tagged union, its value becomes null and okidokey is unaltered: 
                            // Assignment (and other operators) don't morph the right side except to make them into shared pointers when called for.                


Passing things into functions may morph things inside those functions, but it does not morph what you're passing into it unless it has not yet been assigned a type.
Assignments and other binary operators may morph things on their left side, but they don't morph things on their right side,
except that reference assignments to things that are not shared pointers morph them into shared pointers, unless both sides live in the same arena.
Copy assignments into shared pointers decref whatever they were pointing at, create a new allocation, and point the shared pointer at it.
Copy assignments of arena managed objects into same-arena-managed raw pointers simply change where the raw pointer is pointing.

Simple types may change size, as mentioned:

c8 c = 'a';
zs16 s = c;                 // s remains an zs16, and it now contains a single c16 whose value is 'a'.
c = s;                      // c will not switch its base type or become variant because it was declared with a specific type. 
                            // c gets the least significant byte of the first character in s. 
                            // This potentially loses data and will emit a warning.

var z := "stack zs8"        // z is a shared pointer that points at a heap allocation that contains the string. 


7. Pointer Types and Assignments

There are two types of assignments:

v = mylist[i]           // copy assignment.

There, v receives a copy of the i'th element if mylist. Maybe not super efficient, but good for isolation.

v := mylist[i];         // reference assignment. 

If v has local scope (is temporary), or lives in the same vault (arena or app managed heap) with mylist, it becomes a temporary raw pointer, and mylist is unchanged. 
If v is a class member or from a non-local scope, and does not live in the same arena with mylist, and was not qualified as raw, it becomes a shared pointer, 
and mylist becomes virtual. (Normally, things on the right side of an operator aren't morphed by the operation, but reference assignment is an exception.)

pointer declarations:

class Node {
    arc Node next;              // Shared ownership
    weak Node parent;           // Safe observer (becomes null if parent dies)
    raw Node peer;              // fast, unsafe
}


8. Explicit Pointer Types

The morphing system (section 7) handles the common case of references — the compiler infers when something needs to become a shared pointer and manages 
reference counting automatically. However, for systems-level work, FFI boundaries, and algorithms 
that require explicit pointer manipulation, xmc provides explicit pointer types that operate outside the morphing system entirely.

a. Pointer Type Syntax

A * prefix on a type denotes a raw pointer. A & prefix denotes an arc (reference-counted) pointer. These can be nested:

*Node p;          // raw pointer to a Node
&Node p;          // arc pointer to a Node
**Node pp;        // raw pointer to a raw pointer to a Node
*&Node pp;        // raw pointer to an arc pointer to a Node

Explicit pointer types do not participate in refinement morphing. The compiler will not upgrade or change their pointer depth based on 
how they are used. What you declare is what you get.

b. Address-of and Dereference Operators

The @ operator takes the address of its operand, yielding a pointer one level deeper:

*Node p = @someNode;     // p holds the address of someNode
**Node pp = @p;          // pp holds the address of p itself

The $ operator dereferences a pointer, yielding the value it points at:

Node n = $p;             // n is a copy of the Node p points at

$ binds tighter than ->, so $pp->next parses as ($pp)->next — dereference pp to get the inner pointer, then access next through it.
@ binds looser than ->, so @p->next parses as @(p->next) — take the address of the next field of the node p points at.

No whitespace is permitted between @ or $ and their operand.

c. Member Access Through Pointers

The -> operator dereferences a pointer and accesses a member in one step:

p->value = "hello";      // write through p to its value field
*Node q = p->next;       // q gets a copy of p's next pointer

When the member being accessed is itself a pointer, no $ is needed — the type system knows what you have:

p = p->next;             // advance p to the next node; both are *Node
d. Copy Assignment vs Reference Assignment With Pointers

Copy assignment (=) through a pointer dereferences and copies the pointed-at value:

*zs8 p;
zs8 s = p;               // s is an zs8, a copy of the value p points at

Reference assignment (:=) copies the pointer value itself, and propagates raw-constraint:
*zs8 q := p;             // q is also a *zs8 pointing at the same location as p

If the source pointer is raw-constrained, the target becomes raw-constrained. This constraint propagates transitively — 
anything that := assigns from a raw-constrained pointer also becomes raw-constrained. 

A raw-constrained source cannot be reference-assigned to an arc-constrained target; that is a compile-time error, because at the machine 
level an arc pointer requires a refcount header that a raw allocation does not have.

e. Pointer Arithmetic

Raw pointers support a small set of arithmetic operations:

*T + u64  ->  *T          // offset forward by n elements (u64 here is any expression that resolves to a u64)
*T - u64  ->  *T          // offset backward by n elements
*T - *T   ->  u64         // distance between two pointers of the same type

All other arithmetic on pointer types is a compile-time error. The subscript operator is the preferred idiom for offset access — p[n] 
is equivalent to dereferencing *T + n and is cleaner to read than explicit arithmetic in most cases:

*u8 buf;
buf[0] = 0xFF;            // write to first byte
buf[12] = 0x00;           // write to thirteenth byte
*u8 mid = @buf[len/2];    // pointer to the midpoint

f. Pointer-to-Pointer Example

The classic use case for pointer-to-pointer is list node removal, where tracking a pointer to whichever pointer currently points at the 
candidate node eliminates special-casing for head removal:

struct Node {
    zs8 value;
    *Node next;
}

RemoveNode(**Node pp, *Node target) {
    while ($pp != target)
        $pp = $pp->next;     // advance pp to point at current node's next field
    $pp = target->next;      // splice out target 
}

*Node head;

; ... populate list ...
RemoveNode(@head, nodeToRemove);


pp starts out holding the address of head itself. Each iteration advances it to hold the address of the next field of the current node. 
So $pp = target->next correctly updates whichever pointer was pointing at the target, whether that was head or some node's next field, 
without any special cases.

For code that does not need to manipulate pointer-to-pointer relationships, the arc version using & instead of * is identical in syntax 
and handles memory management automatically:

RemoveNode(*&Node pp, &Node target) {
    while ($pp != target)
        $pp = $pp->next;
    $pp = target->next;      // decrements old refcount, increments new one; no leak
}


g. Raw and Arc vs * and &

raw T sets a ceiling on the raw/arc bit and arc T sets a floor on it, but leave the other rmask dimensions free to morph based on usage. 
*T and &T pin all the bits of both minrmask and maxrmask, not just the raw/arc bits of one of them. * and & are the right choices for FFI parameters 
(where the pointee's layout has to match what the foreign function expects) and for situations where you want layout control rather than 
morphing convenience. The two forms are not interchangeable: *u8 is a fully-pinned raw pointer, while raw u8 is a pointer to a u8 that won't 
morph to arc, but may, for example, morph to fluid or concurrent.


9. Arrays

Arrays are xmc's tool for committing to a layout. When you call into FFI, embed a fixed-shape buffer in another structure, or build a tight custom data structure, you
need the compiler to leave the memory shape alone. Arrays accomplish this by pinning the fixed and rigid refinement bits in maxrmask — the morpher cannot promote them
to resizable or fluid, and any operation that would require either becomes a compile-time error.

Underneath, arrays and lists hold their elements identically. An ifmrhrss list and an array over the same data look the same in memory. The difference is one of
commitment: a list may morph as usage demands (.add() promotes resizable; concurrent access promotes concurrent; and so on), and an array cannot — the pinned bits
take those promotions off the table.

a. Declaration

The explicit form is array<T>:

array<i32> a(5);
array<u8> buf(1024);
array<f32> samples(SampleCount());      // runtime size is fine

The C-style bracket-suffix form is shorthand:

i32 a[5];                               // same as array<i32> a(5)
u8 buf[1024];                           // same as array<u8> buf(1024)

Sizes need not be compile-time constants. The size is fixed at the point of declaration and pinned thereafter; what matters is that the layout doesn't change after
that point, not that the value was knowable at compile time.

The bracket suffix attaches to the variable, not the type — "i32 a[5], b;" declares a five-element array a and a single i32 b, not two arrays.

b. Default Refinements and Pinning

By default an array pins two bits in maxrmask:

fixed       — the resizable bit cannot be promoted
rigid       — the fluid bit cannot be promoted

Operations that would require either promotion are compile-time errors:

i32 a[5] = { 1, 2, 3, 4, 5 };
a.add(6);                               // compile-time error: array's resizable bit is pinned

The pinning can be released by adding the corresponding high-refinement qualifier in front of the array spelling:

resizable array<i32> growable(10);      // can grow
fluid array<var> flexible(10);          // layout free to morph

Either qualifier puts the array on the same morph track as a list. Once you've added either one, the array<T> spelling is a documentation choice — you've stopped
committing to the layout, and what you have is effectively a list.

c. Initialization

Curly-brace initializer lists fill the elements in order:

i32 a[5] = { 1, 2, 3, 4, 5 };           // all five elements
u8 buf[1024] = { 0 };                   // first element 0, the rest zeroed too
i32 partial[10] = { 1, 2 };             // first two specified; remaining eight are zero

When fewer initializers are supplied than the size, the remaining elements are zero. The single-element-zero idiom { 0 } is the conventional spelling for a fully
zeroed array.

The size can be left out of the brackets when an initializer is present:

i32 a[] = { 1, 2, 3, 4, 5 };            // size inferred as 5
zs8 days[] = { "Mon", "Tue", "Wed", "Thu", "Fri", "Sat", "Sun" };       // size inferred as 7

The bracket-literal form [ ... ] from §5e produces a list, not an array — the literal context doesn't pin anything. To turn a literal into an array, assign it to an
array variable:

var xs = [1, 2, 3];                     // list, free to morph
array<i32> a = [1, 2, 3];               // array; pinning takes effect, size becomes 3

d. Indexing

The subscript operator a[i] reads or writes the i'th element, zero-based:

i32 a[5] = { 10, 20, 30, 40, 50 };
print(a[0]);                            // 10
a[2] = 99;                              // mutate (morphs a's mutable bit)
i32 last = a[a.length - 1];

The index expression must resolve to a non-negative integer; i32, u32, u64, and similar are all accepted. A constant index outside the declared range is a
compile-time error. Whether out-of-range indices are checked at runtime is configured in the project file: globally they might not be, but in "debug" they might be.
By default they are not, matching the cost profile of C array access. When checks are enabled, an out-of-range read yields a typed null in the reading expression's
context (per §6), and an out-of-range write is a no-op (per §27 noop contagion).

e. Length

Every array exposes its element count as a u64 .length member:

for (var i = 0; i < a.length; i++) print(a[i]);
for (var x in a) print(x);              // for-in is preferred for plain iteration

For a compile-time-sized array, .length is a compile-time constant; the morpher folds reads of .length into the constant value, with no per-iteration field access.
For a runtime-sized array, .length is set once at declaration and pinned alongside the layout; reads still fold to a single immutable value, just one resolved at
runtime.

f. Multi-Dimensional Arrays

Bracket suffixes stack to declare arrays of arrays:

i32 m[3][4];                            // 3 rows of 4 i32 each, 12 elements total
i32 grid[8][8] = { 0 };                 // 8x8, fully zeroed
i32 ident[3][3] = {                     // 3x3 identity matrix
    { 1, 0, 0 },
    { 0, 1, 0 },
    { 0, 0, 1 }
};

The explicit form nests in the natural way: array<array<i32>>.

Layout is row-major: m[0][0] is followed in memory by m[0][1] through m[0][3], then m[1][0], and so on. Indexing applies left-to-right: m[i][j] is element j of row i.

Subscripting with fewer indices than dimensions yields the corresponding sub-array:

i32 row[4] = m[2];                      // copy the third row into a new array

The fixed-and-rigid pinning applies per dimension. A multi-dimensional array can no more grow rows than it can grow row count.

g. Element Type Variations

The element type in array<T> can be any type-specifier. Two cases earn special mention:

array<var> mixed(10);                   // ten slots, each holding any value
array<ref> handlers(4);                 // four reference slots

array<var> is heterogenous: each element is independently typed, and the morpher tracks the variant set actually stored. This is the right type for argument-vector
storage and ad-hoc tables where the element types are known only at the call site.

array<ref> holds inferred references. Each slot is frozen at first assignment, but the slots can be of different ref types (including function references) — appropriate
for dispatch tables and similar structures where the references aren't known at declaration time.

The element type interacts with the array's heterogenous bit (bit 3) independently of the fixed-and-rigid pinning. An array<var> may be heterogenous; an array<i32>
stays homogenous because every slot holds the same concrete type.

h. Relationship to Lists

A list is xmc's name for an indexed sequence with all rmask bits free to morph. An array is what you get when you pin fixed and rigid in maxrmask. Same element layout
when both are at fixed/rigid, same indexing, same .length, same iteration, same slicing rules — what differs is commitment.

Choose array<T> (or T name[size]) when you have a layout commitment to make: you're calling into FFI, you're embedding the value in another struct's memory, you're
handing the address to code that assumes a stable shape. Choose list when usage might want to grow, shrink, or otherwise reshape.

If you write resizable array<T> or fluid array<T>, the pinning is released; you've chosen a list using the array spelling.

i. Slicing

The slice syntax described in §19d applies to arrays:

i32 a[6] = { 10, 20, 30, 40, 50, 60 };
var middle = a[2:5];                    // new array { 30, 40, 50 }
var reversed = a[::-1];                 // new array { 60, 50, 40, 30, 20, 10 }

A slice produces a new value, not a view; modifications to the slice don't affect the source. The slice's size is fixed at the slicing site. A slice of an array is
itself an array; a slice of a list is a list.

j. Arrays, Pointers, and FFI

The address-of operator yields a pointer to an array's first element:

i32 a[5] = { 1, 2, 3, 4, 5 };
*i32 p = @a;                            // p points to a[0]
*i32 q = @a[2];                         // q points to a[2]

xmc does not implicitly convert arrays to pointers. Passing an array to a function that takes *T requires the explicit @; passing it to a function that takes
array<T>, list, or var works without ceremony.

For FFI declarations, the pointers to element types are used for things like buffers and strings:

extern b:rax Kernel32:ReadFile(
    u64:rcx   hFile,                  // opaque 64-bit handle (Windows HANDLE)
    *u8:rdx   lpBuffer,               // pointer to the first byte in the buffer
    u32:r8    nNumberOfBytesToRead,   // size of the buffer
    *u32:r9   lpNumberOfBytesRead,    // pointer to a u32 to be filled with the number of bytes read
    *u8       lpOverlapped            // (a byte pointer passed on the stack — no register clause)
);                                    // b64:rax — the compiler casts the return value as an xmc boolean

*u8[] is "raw pointer to an array of u8" — the xmc spelling of C's u8* parameter type. The array's length is determined by a separate length parameter or by the
function's contract, not by xmc's type system.

The array's pinning is what makes this safe. Consider the call site:

u8 buf[4096];
u32 bytesRead;
Kernel32.ReadFile(hFile, @buf, 4096, @bytesRead, null);

If buf were a list, the morpher would be free to allocate a list header pointing to a separately allocated, growable data block. @buf would yield the address of the
header rather than the data; even if the morpher kept them adjacent, a later .add() in xmc code could reallocate the data block while the kernel was mid-write. The
fixed pinning takes both possibilities off the table — the buffer's address and size are stable for its lifetime, and the xmc compiler will refuse code that would
change either.

The same reasoning applies to embedded fixed-shape buffers:

struct Vector3 {
    f32 elements[3];
}

Vector3 is exactly 12 bytes laid out as three contiguous f32. If elements were a list, Vector3 would contain a list header pointing to data elsewhere — different
size, different layout, and any consumer assuming the C-equivalent struct layout would see something else entirely. The rigid pinning is what guarantees the embedded
array stays inline.

A note on the converse direction: if you write resizable array<T> or fluid array<T>, the FFI examples above stop being safe. The morpher is free to reshape the
buffer, and the kernel call could see stale memory or the wrong address. The compiler doesn't reject @-of-resizable-array outright — there are legitimate uses inside
xmc code — but it warns when such a pointer is passed to an extern declaration whose parameter type implies layout assumptions the array no longer satisfies.


10. Vaults

a. Arenas

You can declare an arena, and put things into it, like this:

arena a1(A1_SIZE);      // the size parameter is optional, if omitted, the arena will use a project-configured arena size, or try to guess at what a reasonable size might be.

list myList:a1 = ["here", "is", "a", "bit", "of", "text"];    // initially fixed in length, holding pointers, placed in a1, to zs8's also placed in a1.

An arena must have the same or greater scope/lifetime of what is put in it so that by the time the arena is destroyed, the things in it are no more.
If an object is in the same arena with something that gets assigned to it, it won't morph into a reference counted object.

var v:a1 := mylist[i];   // here, since myList is also in a1, v becomes a raw pointer to the i'th element of mylist, and mylist is unchanged.

If you remove things from mylist, since it is in an arena, the count of items goes down, but the allocated size remains the same. You can't deallocate from an arena.
You can grow things in there, but you can't shrink them.

b. Heaps

You can declare an application managed heap and put things into it, just like you can with arena:

heap h1(H1_SIZE);       // the size parameter is similarly optional, if omitted, the heap will use a project-configured heap size, or try to guess at a reasonable size.

list myList:h1 = ["here", "is", "another", "list"];     // placed in h1

A heap, like an arena, must have the same or greater scope/lifetime of what is put in it so that when it is destroyed the things pointing into it are also gone.
As with an arena, if an object is in the same heap with something that gets assigned to it, it won't automatically become reference counted.

If you destroy things that are in a heap, their memory is released back to the heap for reuse. 

c. Typed Vaults

You can specify a type for exclusive use within a vault:

arena<Captain> CaptainArena(50 * sizeof(Captain));
heap<Ship> ShipPool(1000 * sizeof(Ship));
heap<Shot> ShotPool(10000 * sizeof(Shot));

Attempting to place something of the wrong type into a typed vault results in either a compiler error or a compiler warning and an error variant:

var captain = SpaceShuttle:CaptainArena("507C-S1");     // captain is a TypeError (an error variant), and a compiler warning is produced
SpaceShuttle:ShotPool captain;                          // no place for a TypeError to go - compiler error

If the type has members that require separate allocations, those allocations are placed in the same vault. 

The size parameter is in bytes, and is an initial size. Vaults will automatically add chunks by default. You can prevent this using the "fixed" refinement qualifier:

fixed arena a1(1024 * 1024);

Attempting to allocate beyond the initially reserved size of a fixed vault results in a error variant.

d. Optimizing Vaults

Typed vaults can often be better optimized than untyped vaults.
If the type itself has a fixed size (like a class or a struct or a fixed sized array, as opposed to a resizable list) that might produce certain optimizations. 
If the type consists of simple types with fixed memory shapes, that will definitely produce optimizations.
Using multiple smaller vaults may result in more optimal memory access than using the system heap, particularly for small, fixed, rigid, materialized types.

11. Loops

xmc supports four loop forms: C-style for, for-in iteration, while, and do-while. Each can carry a label for use with break, continue, and reach (§13, §14).

a. C-style for

for (init; cond; update) statement

The init slot may be a declaration or an assignment, both with the usual semantics — the declared name is in scope for the cond and update slots and for the loop 
body, and goes out of scope when the loop exits. The cond is evaluated before each iteration; the update is evaluated after each body execution but before the next 
cond check. Any of the three slots may be empty.

for (var i = 0; i < 10; i++) print(i);          // "var", here, is optional (§5); but if you don't use it, you will modify any existing i, if there is one.

for (;;) {                                      // infinite loop
    if (DoneCondition()) break;
    Work();
}

A statement body that is a single statement does not require braces; multiple statements are placed in a block.

b. for-in iteration

for (x in expr) statement
for (var x in expr) statement
Iterates x over the values produced by expr. The right-hand expression may be:

    A generator-function call (§19a), in which case the merger inlines the generator into the loop body when both ends are visible.
    A Generator<T> value, in which case iteration calls .next() indirectly (§19b).
    A list, dictionary, or other container, in which case iteration follows the container's natural order.
    A range expression (§19c) such as range(0, 10).

The first form (for (x in expr)) uses an existing x that is in scope; if there isn't one, it declares a new var x for the loop body (per §5). 
The second form (for (var x in expr)) creates a new x for the loop body that shadows any existing x (see §15).

    for (var n in MyGen()) print(n);              // generator, inline-merged
    for (var item in shoppingList) print(item);   // container iteration
    for (var i in range(0, 100, 5)) print(i);     // range

The loop exits when the right-hand side signals exhaustion (§19a) or when the container is fully traversed.

c. while

while (cond) statement

The body executes repeatedly as long as cond is truthy. cond is evaluated before each iteration, so a body may execute zero times.

    while (queue.length > 0) Process(queue.shift());

d. while with binding

    while (var x = expr) statement

Declares x local to the loop body. On each iteration:

expr is evaluated.

If expr is a method on a generator handle that returns a control-flow variant (e.g., gen.next(), see §16), the exhaustion signal exits the loop and the body does 
not run for that iteration. Otherwise, expr's result binds to x, and the result's truthiness determines whether the body runs. A falsy result exits the loop; a truthy
result runs the body with x bound to the value.

    var gen = MyGen();
    while (var n = gen.next()) print(n);            // exits on Finished signal

    while (var line = file.readLine()) Parse(line); // exits on empty/null line

    while (var item = queue.pop()) Handle(item);    // exits when pop returns null

This form unifies the "iterate until exhausted" pattern (generators) and the "iterate while truthy" pattern (queue drains, file reads, etc.) under one syntax. 
The compiler picks the right interpretation based on what expr returns.

e. do-while

do statement while (cond);

The body executes once unconditionally, then cond is evaluated; if truthy, the body executes again. The body always runs at least once. Note the trailing semicolon.

    do {
        var input = ReadLine();
        Process(input);
    } while (input != "quit");

f. break and continue

break exits the innermost enclosing loop. continue jumps to the next iteration — for for, that means evaluating the update slot and then cond; for while, that means
evaluating cond; for do-while, that means evaluating cond; for for-in, that means advancing to the next value. break and continue may be qualified with a label (§13) 
to target an outer loop or block:

    for (var i in range(0, n)) {:outer 
        for (var j in range(0, n)) {
            if (m[i][j] == target) break :outer;
            if (Skip(m[i][j])) continue :outer;
        }
    }

g. Loop labels

Any loop may carry a label for use with reach, break, or continue. Labels follow the form established in §13:

    while (true) {:scan
        once hits = 0;                      // static! Counts the total. Only initialized to 0 once.
        var line = file.readLine();
        if (!line) break :scan;
        if (Match(line)) hits++;
    }

    print("found a total of {scan:hits} matches so far");           // reach into the labeled scope

The label appears immediately after the loop body's opening brace. It is in scope for the body and for any enclosing scope that wishes to reach into the loop after 
it exits.


12. Switch and Case

The switch statement matches a value against a set of cases and runs the body of the first matching case. There is no fall-through; each case body runs only if its case
matches, and execution continues after the closing brace.

a. Basic syntax

switch (key) {
    case_label: case_body
    case_label: case_body
    ...
    *: default_body
}

key is the value being matched. Each case_label is either a value list, an expression using it, or the catch-all *. Each case_body is a single statement or a brace 
block.

switch (n) {
    1, 2, 3: DoThis();
    4, 5: { DoThat(); AndTheOther(); }
    *: IfAllElseFails();
}

The default case (*) is optional. If no case matches and there is no *, the switch is a noop and execution continues past it.

b. Case label forms

A case label is one of three forms. The compiler distinguishes them syntactically:

Value-list form — a comma-separated list of values that the key is compared against using ==. Matches if the key equals any value in the list:

switch (n) {
    1, 2, 3: DoLow();
    100: DoHundred();
    *: DoOther();
}

switch (cmd) {
    "quit", "exit", "q": ShutDown();
    "help", "?": ShowHelp();
}

The values in a value list must be expressions whose value is known at the comparison point — typically literals or named constants. The list does not refer to the key; 
the comparison is implicit.

Expression form — a boolean expression that mentions it (or the named binding, if the switch declares one — see §12d). The case matches if the expression evaluates 
truthy:

switch (m[i][j]) {
    it < 0: HandleNegative();           // An "it" switch key binding is provided that can be optionally used; it's useful when the switch key is an expression
    it > threshold: HandleLarge();
    Special(it): HandleSpecial();
    *: HandleNormal();
}

The compiler recognizes the expression form when it (or the named binding) appears in the case label.
Default — * matches any key the previous cases did not match. 
At most one * per switch, conventionally last.

c. Type-based matching

Two patterns work together for type-based matching. For exact class matching, use typeof(key) as the switch and class names as case labels:

switch (typeof(v)) {
    Captain: HandleCaptain(v);
    Ship: HandleShip(v);
    Shot: HandleShot(v);
}

typeof(v) returns a Class value (see §12f). The class names in the cases are also Class values. Matching is == on Class, which means exact match: a value of a class 
derived from Ship will not match the Ship case. For matching that accepts subclasses, switch on the value itself and use is in expression-form cases:

switch (v) {
    v is Captain: HandleCaptain(v);
    v is Ship: HandleShip(v);
    v is Shot: HandleShot(v);
}

Each is test returns true if the value is the named class or any class derived from it (§14a). The two forms compose. 

Combined match-with-fallback:

switch (v) {
    v is RescueShuttle: SpecialRescue(v);
    typeof(v) == Ship: BasicShip(v);          // exact Ship, not derived
    v is Ship: GenericShip(v);                // any other Ship subclass
    *: Unknown();
}

d. Switch with binding

A switch may declare a name in its parens, just as while and for may. The bound name is in scope inside every case body and inside every expression-form case label.

switch (var result = captainList.find(thisCaptain)) {
    result is NotFoundError: UI:ShowMessage(msgBenched);
    typeof(result) == Captain: UI:ShowMessage(GetCaptainMessage(result));
    *: UI:ShowMessage(msgUnknown);
}

The name follows the optional-var rule (§5): switch (result = expr) declares result if it doesn't exist in scope, or modifies an outer result if it does. The var 
keyword may be added for clarity or to force a fresh local:

switch (database.find(id)) {
    it is User: it.login();
    it is NotFound: ReportMissing();
}

As mentioned above, the provided "it" binding can be used to reference the value of the switch key expression. If "it" is not referenced or reached for, it isn't 
produced.

e. break

break in a switch case body exits the switch immediately. Because there is no fall-through, break is rarely necessary — execution leaves the switch at the end of any 
case body anyway — but it is useful for early exit from a multi-statement case:

switch (state) {
    Running: {
        if (ShouldPause()) break;
        Step();
        Update();
    }
    *: ...
}

A bare break inside a switch nested in a loop exits the switch, not the loop. To exit the loop from inside the switch, use a labeled break (§13):

for (var x in list) {:loop
    switch (x) {
        Bail: break :loop;
        *: Process(x);
    }
}

continue is not used inside a switch; it is a loop construct only. A continue inside a switch case applies to an enclosing loop, with the same labeled-break treatment 
if the loop is not the innermost enclosing scope.

f. typeof and the Class type

typeof(x) returns the runtime class of x as a value of the built-in type Class. Class values can be compared with == and != (which test for exact identity, not inheritance),
assigned to variables, passed to functions, and stored in containers.

Class c = typeof(myDog);
if (c == Dog) ...                   // true if myDog is exactly a Dog
if (c == Animal) ...                // false even if myDog is a Dog (a subclass of Animal)
if (myDog is Animal) ...            // true: is checks inheritance, == does notA bare class name evaluates to a Class value too:Class c = Dog;

if (typeof(myDog) == c) ...         // same as comparing against Dog

The combination of == (exact identity) and is (inheritance-aware) gives both behaviors at the source level: write the one you want, and the compiler does what it 
says. 

Comparing two class names directly is well-formed but constant-folded: Dog == Cat is statically false, and any branch it guards is eliminated as dead code. This case 
is degenerate the useful cases are comparisons involving typeof() calls or Class-typed variables whose values are not known at compile time.


13. Nested Scope Labels and Break;

A scope label is the first token after an opening brace. It exists only on braced bodies. As mentioned in 11g, any loop body can be braced — so any loop can be labeled, 
but only if you use braces. The break verb can be targetted at a label, in which case the break breaks out of the labeled scope.

{:top
    {:nested
        if (someCondition) break :top;         // breaks out of :top and skips DoStuff and DoOtherStuff
        if (otherCondition) break :nested;      // breaks out of :nested and skips DoStuff but DoOtherStuff will be executed
        DoStuff();
    }
    DoOtherStuff();
}


14. Reaching

You can "reach" into a nested scope after the scope to pull out values. A name declared in a nested scope is accessible by ordinary name resolution in the enclosing scope 
after the nested scope exits:

myProc(){
    while (true) {
        var n = start;
        if (f(n)) n++;
        if (k(n)) return;
        if (x(n)) break;
    }
    if (q(n)) HandleQCase(n);
}

A label-qualified name reaches into a labeled block specifically (§15d):

    myProc() {
        while (true) {
            var n = start;
            for (j = 1; j < n; j += 3) {:jscan
                var bar = xNine(j + n);
                if (SpecialCase(j + n)) bar = bar % 15;
            }
            if (n >= maxn) break;
            n += nextN(n);
        }
        if (MeetsQuals(jscan:bar)) signalMeetsQuals(bar);
    }

Reaching uses ordinary name resolution from the reaching context. A bare name resolves to the variable declared in the nearest enclosing scope; a label-qualified 
label:name resolves the identifier within the labeled scope's bindings. In either case, a reach addresses the named scope itself, not any nested scope inside it. 
Reach does not cross function boundaries; you cannot reach into a nested function.

If the path that would have written the variable was skipped at runtime, the read returns null. The compiler arranges this by zeroing the variable's stack slot at 
function entry, but only when both conditions hold:

    - the variable is reached from an enclosing scope, and
    - the compiler cannot prove the variable is written on every path that exits the scope.

Variables that are not reached for, or that the Morpher proves are definitely written before the scope exits, are not zeroed and may contain whatever was previously 
on the stack until the program writes them. This means functions that do not use reach pay no initialization cost; the cost of the "null on skipped path" guarantee 
is paid only by the specific variables that need it.

For variables that should be initialized regardless of whether they are reached for, the programmer initializes explicitly:

    var x = 0;
    i32 a[5] = { 0 };
    zs8 name = "";
    13. Shadowing

A variable is shadowed when an inner scope declares a new variable with the same name. Inside that inner scope, the new declaration takes over: every reference to the 
name resolves to the inner variable. Outside the inner scope — including before the shadow declaration in the same scope, and after the inner scope exits — the 
outer variable is back.

    var u = database.find(id);          // u is variant<User, NotFound, Timeout>
    if (u is User) {
        User u = u;                     // shadow: inside this if, u is strictly a User
        u.login();
    }
    // out here, u is the variant again, unaffected by anything inside the if

The shadow declaration may have a different type than the variable it shadows; that is the most common reason to shadow. The right-hand side u in User u = u; 
refers to the outer variable (the binding visible at the point of the declaration, which is the moment the right-hand side is evaluated); the left-hand side u 
introduces the new binding in the enclosing scope. From the next statement onward, u is the new binding.


15. Shadowing

A shadow has its own storage, separate from the variable it shadows. Assignments to a shadow do not affect the variable it shadows.

a. Shadows from "is" narrowing

Writing Type x = x; after an "is" check is the explicit form of variant narrowing. The shadow has its own storage and is paid for as a regular variable.
Inside the body of if (x is Type), the compiler knows the variant has narrowed to Type for the duration of the body, and uses that knowledge to validate calls and 
dispatch members without re-checking the tag. This is flow-sensitive knowledge about the existing variable, not a new variable — no shadow is created, no copy is 
made, and x inside the body is still the same variable as x outside.

Use the explicit shadow form when you need a separate binding — for example, to pass the narrowed value to a function whose parameter type is strictly Type, or to capture
the narrowed value into a name that will outlive the if-body via labeled-block reach (§15d). For the common case of "I want to call methods on the narrowed type within the 
body," no shadow is needed:

if (u is User) u.login();           // no shadow; u is still the variant outside

if (u is User) {:authed
    User u = u;                     // explicit shadow; reachable as authed:u outside
    u.login();
    RegisterSession(u);             // u passed as User
}

b. Shadows in pipe handlers

The pipe operator's it binding (§16d) is a shadow that exists only inside the pipe handler block. The handler's it shadows nothing in the enclosing scope unless the
enclosing scope happens to have a variable named it — which hopefully it usually doesn't.

c. Shadowing rule in general

Whenever a variable is declared in a scope, that declaration shadows any visible variable of the same name from an enclosing scope, for the remainder of the inner scope.
This is uniform across function bodies, nested blocks, loop bodies, if-then bodies, error blocks, pipe handlers, and labeled blocks. A name in any inner scope always
resolves to the innermost binding visible at the point of reference.

d. Reaching past a shadow

A shadow conceptually creates a nested scope around the shadow declaration: the shadow lives inside this implicit inner scope, and the outer binding is unaffected. 
Reaching follows the §14 rule — it addresses the named scope, not the scopes inside it. So a reach into the scope that contains the shadow declaration reads the outer
binding (the shadow lived in an inner scope, not visible to this reach):

var u = database.find(id);

if (u is User) {
    User u = u;                 // shadow lives in the if's implicit inner scope
    u.login();
}

u                               // outer variant — the shadow's scope is not addressed here

To reach the shadow, label the scope that contains the shadow declaration. The label-qualified reach addresses the labeled scope, and the shadow is one of its bindings:

var u = database.find(id);
if (u is User) {:narrowed
    User u = u;
    u.login();
}

u                               // outer variant
narrowed:u                      // shadow's final value, or null if the if did not fire

The label-qualified name resolves the identifier within the labeled scope's bindings at the syntactic end of the block. Shadows declared in nested-but-unlabeled inner 
scopes within the labeled scope are out of scope at the end of the label and not addressable through it; label the inner scope as well if you need to address it.


16. Control-Flow Variants

Some return values aren't "values" in the ordinary sense — they're signals that the call's normal control flow didn't happen. xmc has two such signals built into the
language: errors, produced by failed operations, and exhaustion, produced by generators that have nothing more to yield (§19). The language treats both with the same 
machinery: they are control-flow variants — variant tags the compiler recognizes specially and routes to designated handling sites instead of forcing the programmer to
inspect them inline. 

The handling sites are the error block ({? ?}), the pipe operator (|), and certain loop-binding forms (§19a). Any control-flow variant returned from an expression in 
one of these sites is intercepted and routed; outside these sites, control-flow variants behave like ordinary variants and the noop-contagion rules of §27 apply.

a. The error Class Factory

There is a special built-in error construct that facilitates this kind of syntax: 

error FileOpenFailed(zs8 message);    // class with an inferred constructor, member, and functor that inherits from XmcError; the functor returns a cloned instance

This declares an error class with a constructor that takes a single parameter, automatically forwarded to a member of the same name and type. 
To instantiate the error:

var result = FileOpenFailed(someErrorStr);    // result.message holds a copy of someErrorStr

error err := FileOpenFailed();                // shared pointer to a heap allocation;
                                              // omitted parameters become null; err.message is ""

b. Error Blocks

An error block is a brace pair in which any expression returning an error variant immediately exits the block:

{? 
    var file = open("data.txt");    // if open returns an error variant, skip to ?}
    var head = file.read(10);       // never runs if open failed
    print(head);                    // never runs either
?}

An error block also exits when a null-returning call sets __missing_attribute — see §16e and §27d.

c. OnError Clauses

An OnError clause is a statement that declares what to do when a preceding error block 
exits via failure of a specific error kind:

{?
    var file = open("data.txt");
    var head = file.read(10);
    print(head);
?}
OnError(FileOpenFailed) { errout("could not open: {it.message}"); return; }
OnError(IoError)        { LogIoError(it); }
OnError                 { errout("unhandled: {it}"); return; }    // catch-all

Each clause names the error kind it catches in parentheses, or none for a catch-all. The clauses are tested in source order against the error kind that caused 
the block to exit; the first clause whose kind matches runs and its body executes. Within the body, the binding it refers to the error object that caused the exit.

A catch-all clause binds it as XmcError, the language's base error type; the body can narrow further via is-checks against specific kinds (§12g).

Multiple OnError clauses may follow a single error block. They are tested in the order written, so kind-specific clauses should appear before any catch-all. An 
OnError clause not preceded by an error block is a syntax error.

If the error block exits via failure but no clause's kind matches, control passes through to whatever follows the clauses — the unmatched error is dropped. Pair an 
explicit error block with a catch-all when that is not what you want.

Each clause's body is a normal statement block. It can return from the enclosing function, break or continue an enclosing loop, or complete and fall through to 
the code after the clauses. Clauses do not fall through into each other — completion of one clause's body leaves the OnError chain entirely.

A clause naming an error kind that none of the block's fallible calls can produce is a compile-time warning (dead clause).

d. The Pipe Operator

The pipe operator handles a control-flow variant inline at the call site, binding the variant to the name it for inspection:

var u = database.find(id) | { errout("find failed: {it.message}"); return; }
u.login();

If database.find returns an error variant, the right-hand block runs; otherwise, the left-hand value binds to u normally. The right-hand block can produce a 
substitute value, return from the enclosing function, or break out of an enclosing loop — whatever fits. If it produces a value, that value binds to the left-hand 
target. The pipe handles any control-flow variant, not just errors:

var n = gen.next() | { -1 }                   // exhaustion: substitute sentinel
var n = gen.next() | { return; }              // exhaustion: bail from the caller
var n = gen.next() | {
    if (it is Finished) break;                // exhaustion: break the enclosing loop
    if (it is ParseError) return;             // error: return from the caller
    0                                         // any other variant tag: use 0
}

When the right-hand block contains only is-discriminated cases for some variant tags but not others, unhandled tags fall through and the pipe re-raises them — 
they continue to be control-flow variants for whatever encloses this pipe site.

This is example shows this re-raising in action:

i32 SumPositives(ref gen) {
    var sum = 0;
    while (var n = gen.next() | { if (it is ParseError) return -1; }) {
        if (n > 0) sum += n;
    }
    return sum;
}

In this example, errors other than PermissionDenied are raised all the way out of the function:

ProcessAllFiles(ref files) {
    for (var f in files) {
        for (var line in f.lines() | { if (it is PermissionDenied) continue; }) {
            // ParseError or any other error here re-raises out of the pipe,
            // out of the inner for, out of the outer for, and out into
            // ProcessAllFiles's caller (or its enclosing error block, if any).
            print(line);
        }
    }
}


e. Thread-Local Error Flags

Several thread-local flags are set by language operations and can be checked by the programmer when needed. The compiler does not generate code to check them unless 
they are used in the source.

__carry and __overflow      — set by numeric operations that produce carry or overflow, as described in §5d.
__missing_attribute         — set when a runtime attribute lookup fails to find the named attribute on a fluid type, cleared when the lookup succeeds. 
                              Reflects the result of the most recent attribute lookup only. Used by error blocks to detect failed attribute lookups on null calls 
                              where no error variant is produced. See §18g.

17. Functions

a. Named Functions

The general function syntax borrows from C and C++.

type SomeFunction(arguments...) {       // arguments separated by commas
    statements                          // statements separated by semicolons
    return somevalue;
}

There's no "void" type, for a void function you just skip the type.

SomeProcedure(arg1, arg2){
    DoSomeThings();
    ..
}

You don't have to return, if you don't, null is returned. You can return without specifying a value, doing that returns null.

b. Anonymous Functions

There is no capture block specified for anonymous functions.

(arguments){statements}

The capture block is inferred based on what is used. Whether a capture is by value or reference depends on its refined type taking into consideration how its used.

c. Variadic Functions

print(arg1, arg2, args...){         
    // arg1 and arg2 are named within the function
    // argc has the anonymous argument count
    // argv has the anonymous arguments, it is used syntactically like an array of variants whose type list depends on the types of arguments that were passed
    // (exactly how argv is implemented depends on the context, hopefully we can come up with some optimizations)
}

d. Parameter Passing Syntax

Parameters can be ommitted from right to left. They can omitted from left to right by naming a parameter.

ThisProc(zs8 name, zs8 handle, u8 age){..}

ThisProc("SuzyQ");

ThisProc(handle="QueenB", 27);

e. Function overloading

If a function has a different return value or different parameters it is recognized as a different function.

zs8 TransformString(arglist);                                           //
s8 TransformString(arglist) { return TransformString(arglist); }        // different function, same arglist, converts a zs8 to an s8 and returns it.
var s = TransformString(arglist);                                       // ambiguous call error.
s8 s = TransformString(arglist);                                        // maybe not terribly efficient, but it works fine.

f. Parameter Type Inference

A parameter declared without a type specifier is implicitly var:

Apply(fn, thing) { fn(thing); }     // fn and thing are var

When a var parameter's type cannot be resolved from the function body alone, the function becomes an implicit generic — the body is cloned and specialized 
per distinct argument type at each call site. See §27a.

A parameter declared with ref is similarly inferred per call site, and the body is similarly cloned per distinct inferred type. The difference is that within each
specialization the type and rmask are pinned at whatever the call-site argument implies, where var leaves further morphing open within the specialization. Use ref 
when you want per-site specialization without subsequent morphing — function-reference parameters are the typical case (§18h).

ApplyToList(list items, ref fn) {
    for (var item in items)
        fn(item);                   // fn's signature is frozen at whatever was passed
}

ApplyToList(myList, myInstance.Process(zs8));    // specialized for this signature


18. Function Pointers and First-Class Functions

Functions in xmc are first-class values. They can be stored in variables, passed as parameters, added to containers, and assigned as runtime 
attributes to fluid types. There are two forms of function variable: inferred (ref) and explicitly typed (function(...)).

a. Inferred Function References

A ref variable infers its function signature from the first assignment and then holds that signature fixed. Subsequent assignments must match 
the inferred signature or a compile-time error is emitted:

ref fn1 = myInstance.AddToList(zs8);   // inferred: bound, signature (zs8) -> null
ref fn2 = MyClass:Swap(i32, i32);      // inferred: unbound, signature (i32, i32) -> null

fn1 = otherInstance.AddToList(zs8);    // ok — same signature, different receiver
fn1 = myInstance.AddToList(u32);       // error — signature mismatch

b. Explicitly Typed Function References

An explicit function(...) type declaration states the signature upfront. This is used when you want to declare a function variable before 
assigning it, or when you want the type to be self-documenting:

function(zs8) -> b fn3;               // declared, nothing bound yet
fn3 = myInstance.AddToList(zs8);       // ok — signatures match
fn3 = MyClass:AddToList(zs8);          // ok — same signature, different target
fn3 = myInstance.AddToList(u32);       // error — signature mismatch

When an explicit type is provided, ref is not used — the function(...) type replaces it entirely.

c. Bound vs Unbound References

A function reference taken from a class name is unbound — the receiver must be supplied at call time as the first argument:

ref fn4 = MyClass:AddToList(zs8);
fn4(myInstance, "hello");              ; receiver passed explicitly

A function reference taken from an instance is bound — the receiver is captured at storage time and the call requires no receiver argument:

ref fn5 = myInstance.AddToList(zs8);
fn5("hello");                          ; receiver already bound

d. Specializing Generic Functions

A generic function — one with unresolved var parameters — cannot be assigned directly to a ref or function(...) variable, because there is 
no concrete calling convention or argument layout to point at until the types are resolved. Instead, a specific specialization must be provided 
at storage time using a type list in parentheses:

ref fn6 = MyClass:Swap(i32, i32);      ; ok — specialization provided
ref fn7 = MyClass:Swap(f32, f32);      ; ok — different specialization
ref fn8 = MyClass:Swap;                ; error — unspecialized generic

The type list in parentheses is not a function call — it is a specialization directive that tells the compiler which concrete version of the generic 
function to produce a pointer to.

e. Pointer-to-Function Types

Function references can be raw or arc pointers using the same * and & syntax as any other pointer type:

*function(zs8) -> b  fn9;             ; raw pointer to a function
&function(zs8) -> b  fn10;            ; arc pointer to a function

This allows function pointers to be stored in arenas, passed through explicit pointer manipulation code, and managed with the same ownership semantics 
as any other pointer.

f. Function References in Containers

Function references can be stored in containers. If references of different signatures are added, the container's elements become heterogenous variants, 
as with any other mixed-type container:

list handlers;
handlers.add(MyClass:Swap(i32, i32));       ; ok
handlers.add(MyClass:Swap(f32, f32));       ; ok — handlers becomes heterogenous
handlers.add(MyClass:Swap);                 ; error — unspecialized generic

Calling a function reference retrieved from a heterogenous container requires either a type check with is, or an error block to handle the case where 
the signature doesn't match the call:

for (ref fn in handlers) {
    if (fn is function(i32, i32))
        fn(a, b);
}

The is operator works with function signatures as well as class and struct types. A signature matches if the parameter types match exactly and the return 
type matches, or if no return type is specified in the is expression, only the parameter types are checked:

if (fn is function(i32, i32))       ; matches any function taking two i32s
    fn(a, b);                       ; regardless of return type

if (fn is function(i32, i32) -> b)  ; matches only if return type is also b
    fn(a, b);

g. Runtime-Added Function Attributes

Function references can be assigned as runtime-added attributes to any fluid type, enabling duck-typed method dispatch:

var handler;
handler.onClick  = Button:HandleClick(i32, i32);    ; ok — specialized
handler.onResize = Window:HandleResize(i32, i32);   ; ok — specialized
handler.onClose  = Window:HandleClose;              ; error — unspecialized generic
handler.onClick(x, y);                              ; runtime attribute lookup, then call

When a runtime attribute lookup is performed, the thread-local flag __missing_attribute is set if the attribute does not exist, 
and cleared if it does. This happens for every attribute lookup, regardless of whether the programmer checks the flag. The flag 
reflects the result of the most recent attribute lookup only.

If the attribute doesn't exist, the call is a noop. The programmer can detect this in several ways:

; option 1 — check existence before calling

; a function reference in a boolean context is true if non-null
if (handler.onClick)
    handler.onClick(x, y);

; option 2 — error block exits if __missing_attribute is set after a function call

{?
    handler.onClick(x, y);
    DoSomethingThatOnlyMakesSenseIfOnClickWasCalled();
?}

; option 3 — check the flag manually afterward

handler.onClick(x, y);
if (__missing_attribute) {
    ; onClick wasn't there
}

For compound expressions involving multiple attribute lookups, evaluation order determines which lookup __missing_attribute reflects at any given point.
Arguments are evaluated before the function lookup, so after a call expression, __missing_attribute reflects the function attribute lookup specifically:

handler.onClick(handler.getX(), handler.getY());    ; after this line, __missing_attribute reflects the onClick lookup,
                                                    ; not getX or getY

h. Function References as Parameters

A function can accept a function reference as a parameter using either ref for an inferred signature or an explicit function(...) type:

ApplyToList(list items, ref fn) {
    for (var item in items)
        fn(item);
}

ApplyToList(myList, myInstance.Process(zs8));       ; specialized at call site

When ref is used as a parameter type, the compiler infers the signature from what is passed at each call site and produces a specialized version of the 
enclosing function for each distinct signature, following the same implicit generics rules described in section 27.

i. Type Testing Function References

The is operator works with function signatures as well as class and struct types. In a function pointer declaration, the full signature including return type 
is always required. In an is check, the return type is optional because the call site is right there and the compiler can see how the return value is used — 
but omitting it carries a risk if the return value is used:

; function pointer declaration — return type required:
function(i32, i32) -> User  fn;

; is check — return type optional but recommended if return value is used:

if (fn is function(i32, i32))
    fn(a, b);                           ; safe — return value unused

if (fn is function(i32, i32) -> User)
    User result = fn(a, b);             ; safe — return type verified by is check

if (fn is function(i32, i32))
    User result = fn(a, b);             ; compiles, but if fn actually returns
                                        ; something other than User, result is null

Parameter types must match exactly. The return type, if specified, must also match exactly. If the return type is omitted from the is check, the function is 
called and the return value is assigned to whatever is on the left hand side — if the types are compatible, the value is received normally; if not, the left 
hand side receives null.

If the matched function is generic and its return type was not fully resolved at specialization time, the return value is an unresolved var. A type anchor 
on the left hand side will resolve it there; if no type anchor is available the result stays unresolved and operations on it follow the noop-contagion rules 
described in section 27d.


19. Generators, ranges and slices

a. Generators

A function containing the yield keyword is a generator. Each yield x; produces a value of the function's declared return type.
Running off the end of the body, or executing a bare return signals exhaustion.

i8 MyGen() {
    for (var i = 1; i < 10; i++)
        yield i;                        // produces an i8 to the consumer falling off the end signals exhaustion
}

A generator-function call returns a handle to an unmerged generator. The handle is a value: it can be stored in a variable, passed to functions, 
returned from functions, and stored in containers. The compiler merges generator code with consumer code wherever it can see both ends.
Where it cannot, the handle materializes as a runtime value of type Generator<T> and iteration proceeds via indirect dispatch on the handle.

The simplest consumption form is for (x in expr):

for (var n in MyGen()) print(n);        // merged: generator inlined into loop body

Here the consumer and generator are visible to the compiler in the same scope, so merging happens at compile time: each yield becomes "assign to n, 
run the loop body, continue"; the implicit end becomes "exit the loop." No handle exists at runtime. The cost is identical to a hand-written loop.

The manual consumption form uses .next() on the handle:

var gen = MyGen();
var sum = 0;
while (var n = gen.next())              // exhaustion signal exits the loop
    sum += n;

In a while (var x = expr) loop, expr may be either a generator-handle method (in which case the exhaustion signal becomes the loop-exit edge) or any other 
expression (in which case a falsy result produces an exhaustion signal). This unifies the truthy-value pattern while (var line = file.readLine()) and the generator 
pattern under one form.

gen.next() outside a loop binding context returns variant<T, Finished>. As with error variants, the programmer can handle the variant explicitly with the pipe 
operator, or ignore it and let noop contagion handle the exhausted case:

var n = gen.next();                     // discards exhaustion: n is null T if exhausted
var m = gen.next() | { return; };       // pipe handles exhaustion: bail from the caller
var k = gen.next() | { -1 };            // pipe handles exhaustion: use sentinel value

The pipe handler binds the variant to it for inspection:

var n = gen.next() | {
    if (it is Finished) break;
    if (it is ParseError) return;
    0                                   // any other variant tag: use 0
}

Calling .next() on an already-exhausted handle continues to return the exhaustion signal — there is no separate "after exhaustion" state to worry about.
List comprehensions are themselves generators, so a comprehension over a generator is a generator-within-a-generator:

var l = [n * 2 for n in MyGen()];       // l is a list; both generators inline-merged

Generators are single-use. A handle, once exhausted, stays exhausted. To restart, call the generator function again — for (n in MyGen()) ... written a second time
 is a fresh iteration. Re-instantiation is free in the merged path.

b. Generator handles across function boundaries
Within a single function, the merger can see both the generator and its consumer. When a handle escapes the function that created it — returned from a factory, stored in a container, passed to a function with a var parameter — the merger can no longer inline both ends, and the handle materializes as Generator<T>:
Generator<list> MakePager(list lines, u32 pageSize) {
    return PageGen(lines, pageSize);    // handle returned across function boundary
}

PageGen(list lines, u32 pageSize) {
    var i = 0;
    while (i < lines.length) {
        yield lines[i:i+pageSize];
        i += pageSize;
    }
}

// caller side

var pager = MakePager(myLines, 50);
for (var page in pager) print(page);    // iteration via indirect .next() dispatch

Generator<T> is the polymorphic handle type. It carries a function pointer to the generator's boxed .next() and an arc reference to the captured state 
(arguments and any locals that need to survive across yields). The cost is one indirect call per iteration — comparable to virtual dispatch in any other language. 
Containers of Generator<T> are homogeneous in shape regardless of which specific generators they hold:

list<Generator<i8>> gens;
gens.add(MyGen());
gens.add(OtherI8Gen());
for (var g in gens)
    for (var n in g) print(n);          // outer: list iteration; inner: virtual .next()

The specific generator type produced by each generator function is opaque and cannot be named in source. Use ref to capture the specific type within a scope 
(preserving merge-ability), var or Generator<T> to accept the polymorphic form (forcing materialization).

ref g1 = MyGen();                       // specific type; merger sees both ends
var g2 = MyGen();                       // var → materializes to Generator<i8>
Generator<i8> g3 = MyGen();             // explicit polymorphic form

Because materialization carries a real cost, prefer ref and same-scope consumption in performance-critical loops; reach for Generator<T> when the polymorphic 
flexibility is what you want.

c. Ranges

range() is a standard-library generator returning successive integers:
for (var i in range(0, 10)) print(i);            // 0 through 9
for (var i in range(0, 100, 5)) print(i);        // 0, 5, 10, ... 95
The handle returned by range() is just a generator handle like any other — merged when consumed in the same scope, materialized to Generator<T> if it crosses a 
function boundary.

d. Slices

Slices are a separate mechanism: they produce a new list or string from a subscript range, not a lazy iteration handle. Syntax follows Python:

var a = [1, 2, 3, 4][1:];               // [2, 3, 4] — second element to end
var b = "hello"[1:4];                   // "ell"
var c = "hello world!"[6:11];           // "world"
var d = "hello"[::-1];                  // "olleh" — reversed

A slice produces a new value of the same kind as the source (list slice → list, string slice → string). Unlike generators, slices materialize their result eagerly — 
there is no handle, no merging, no exhaustion signal. For lazy iteration over a subset, wrap the source in a generator instead:

i8 EveryOther(list xs) {
    for (var i = 0; i < xs.length; i += 2)
        yield xs[i];
}

for (var x in EveryOther(myList)) print(x);


20. Comprehensions

a. list comprehensions

List and dictionary comprehension syntax and native json syntax is borrowed from Python, but the implementations morph from ~ifmrhrss to whatever ends 
up being required.

var l = [upper(s) for s in name];   // list comprehension. Generates a list of whatever type of elements are in name. Initially ~irmrhrss.

Let's say name is a zs8 and its elements are c8's.

l = [n*n for n in range(10)];       // now the elements of l are variants. Either c8's or integers. Not exactly ideal, but possible.

l[6] = "woohoo!"                    // now the list element variant types are u8, c8 or zs8.

var v = l[6]                        // the variable v shadows whatever v was before. Now it's a variant, for the rest of the scope.

if (v == 6*6) { print("maybe, but no"); }
else if (v == "woohoo!") { print("there you go"); }
else if (v == someObjectInstance) { print("nope"); }     // comparison's don't morph things, they just return false if they don't make sense.

b. map comprehensions

keys = ['a', 'b', 'c']
values = [1, 2, 3]
combined = {k: v for k, v in zip(keys, values)}


21. Threading

- spawn blocks

var myPromise = spawn DoStuff(arg1, arg2);  
var result = myPromise.Wait();

Here, myList is being accessed from a spawn block, which may trigger refinement morphing:

var myPromise = spawn {
    DoStuff(myList);        
}


Here, myList may not need to morph:

var myPromise = spawn {
    list myList;
    DoStuff(myList);        // might produce morphing or might not, depends on parameter rmask
}

- parameter refinements

Parameters in function signatures have minimum and maximum refinement masks which represent the nimimum refinements required and maximum refinements 
allowed for that function. By default a parameter is marked as serial, in the function, it is reference assigned into any non-local, or passed to another 
function in a parameter with a concurrent requirement, then the parameter is marked as concurrent. 

Literals are refinement-limited to being materialized (non-virtual, that is, non-reference assignable). 
If a literal is passed to a function that reference assigns that parameter, an error is thrown: "Error: [funcname]-[argname] must be reference-assignable."
The error should show the source code line (with filename and line number) that creates the rule and the one that violates it.

22. Enums

An enum is a named scope containing a set of named integer values. It exists to give meaningful names to a small fixed set of states — directions, error codes, log
levels, parser states — without paying for a struct's storage overhead and without polluting the enclosing scope with a flock of bare constants.

Per the dot-vs-colon rule (Key Decision #11), the enum name is a scope and its values are reached through the colon:

enum Direction { North, South, East, West };

Direction d = Direction:North;
print(d == Direction:South);            // false

The values are members of Direction's scope, not the enclosing scope; bare North does not resolve to anything. This matches the way ClassName:staticMember works for
classes — the scope name on the left of the colon, the member name on the right.

a. Declaration

The basic form lists value names in braces:

enum Direction { North, South, East, West };

enum Severity {
    Debug,
    Info,
    Warning,
    Error,
    Fatal
};

Each name becomes a value in the enum's scope: Direction:North, Direction:South, Severity:Debug, and so on. The names follow the same identifier rules as variable
names; they cannot collide with each other within the enum, but two different enums may use the same value name without conflict (Severity:Error and Action:Error are
distinct).

b. Underlying Integer Type

By default an enum's underlying type is the smallest unsigned integer that fits all of its values. A four-value enum like Direction picks u8; an enum with values up to
300 picks u16; and so on. The morpher chooses based on the value range after all values are computed.

To pin the underlying type explicitly, use the angle-bracket form:

enum<i32> ErrCode { OK, NotFound, Denied, Internal };
enum<u32> Magic { Header = 0xDEADBEEF, Footer = 0xCAFEBABE };

The explicit form is the right choice when the enum will cross an FFI boundary, when its values need a specific bit-width, or when it embeds in a struct whose layout
must be stable. The bracketed type may be any of the integer numeric types: i8, i16, i32, i64, u8, u16, u32, u64.

c. Explicit Values and Auto-Increment

Values may be specified explicitly with =, and any unspecified value takes the previous value plus one. The first value, if unspecified, is zero:

enum Status {
    OK,                                 // 0
    Warning,                            // 1
    Error                               // 2
};

enum HttpCode {
    Continue        = 100,              // 100
    Switching,                          // 101
    OK              = 200,              // 200
    Created,                            // 201
    BadRequest      = 400,
    NotFound        = 404,
    ServerError     = 500
};

Two values within the same enum may share an integer value — useful for aliases like Severity:Critical = Severity:Fatal, or for any reason you happen to want.
The compiler does not warn either way; if you wrote it, it's what you wrote.

Specified values must fit the underlying type. enum<u8> with a value of 300 is a compile-time error.

d. Use and Comparison

An enum value is, at runtime, exactly its underlying integer. Direction:East is a u8 with value 2; nothing more. The enum type is a compile-time hint and a named scope
for the values — useful for intellisense, useful for documenting intent — but it carries no runtime tag and imposes no extra storage cost.

This means enum values participate in arithmetic and comparison the way any integer of their underlying type does:

Direction d = Direction:East;
u8 prev = d;
if (d == Direction:North) ...;
if (d == 0) ...;                        // equivalent: Direction:North is 0
if (d < Direction:South) ...;           // relational comparisons work too
if (d == Severity:Info) ...;            // cross-enum comparison: just an integer compare

Cross-enum comparison is allowed and means exactly what it says: compare the integer values. The compiler does not warn — if you wrote that comparison, you presumably
meant it. The same goes for comparing an enum value to a bare integer or to an arithmetic expression.

e. Storage in Variants and Structs

In structs, enum members occupy exactly the underlying type's bytes — no tag, no padding beyond what the underlying type would need on its own:

struct Player {
    zs8 name;
    Direction facing;                   // one byte
    Severity logLevel;                  // one byte
    i32 score;
};

In a var, an enum-typed assignment behaves like assigning the underlying integer plus a compile-time hint that this slot is "an enum of type T":

var u = SomeFunction();
u = Direction:North;                    // u is now u8 with the Direction hint

The morpher can use the hint to keep intellisense working at later use sites, but the runtime value is just the integer.

f. Conversion to and from Numbers

Enum values and integers convert in both directions without ceremony:

Direction d = Direction:East;
u8 raw = d;                             // raw is 2 (East's auto-incremented value)
i32 wider = d;                          // widens to i32 normally
Direction back = raw;                   // just an assignment; raw was already a u8

There is no runtime range check, no casting requirement, no compiler complaint when an out-of-range integer ends up assigned to an enum-typed variable. The variable
holds whatever bit pattern was assigned. Subsequent comparisons against named values simply tell the truth: an out-of-range value is equal to none of them.

If you genuinely need defensive conversion — checking that a wire-format byte is one of the legal values before treating it as the enum — write it explicitly:

u8 n = ReadByte();
if (n <= Direction:West) {              // values are 0..3
    Direction d = n;
    ...
}

This is not idiomatic enum code; it is defensive integer handling that happens to land in an enum-typed variable. Most enum use never needs it.

g. Bitwise Operations and Hex Bitmaps

Because enum values are integers, the bitwise operators (|, &, ^, ~, <<, >>) work between them with no extra machinery. Hex literals as enum values give you a bitmap:

enum<u32> Permissions {
    Read    = 0x01,
    Write   = 0x02,
    Execute = 0x04,
    Delete  = 0x08,
    All     = Read | Write | Execute | Delete
};

Permissions p = Permissions:Read | Permissions:Write;

if ((p & Permissions:Read) != 0) ...;
if ((p & Permissions:All) == Permissions:All) ...;

The All value is a perfectly fine enum value — the named-values list is exhaustive only if you choose to keep it that way; combining named values with | to form composites
is normal use.

h. The use Directive

Within any scope, the use directive brings an enum's values into the local lookup chain so they can be referenced without the enum-name qualifier:

use Permissions;
i32 mask = Read | Write | Execute;
if ((mask & Delete) == 0) ...;

The qualifier-less names are looked up in the order: local declarations, then names brought in by use, then enclosing scopes. A local variable named Read shadows
Permissions:Read; an outer-scope Read is shadowed by the use.

Multiple use directives in the same scope are fine, even if their enums share value names. The compiler raises no error or warning at the use directives themselves.
The check happens at each unqualified reference: if a name resolves to more than one candidate brought in by use, the compiler reports an ambiguity at that reference
and asks for the qualifier. References to non-colliding names from either enum continue to work without qualification.

use Permissions;                // has Read, Write, Execute, Delete
use FileMode;                   // has Read, Write, Append

i32 mask = Permissions:Read;    // fine — qualified
i32 m    = Append;              // fine — only FileMode has Append
i32 r    = Read;                // error: Read is ambiguous (Permissions:Read or FileMode:Read?)

This makes use a tool the programmer can apply liberally without worrying that bringing in a second enum will retroactively break code that references the first.
Only the names that actually collide need disambiguation, and only at the points where they are used.

use is scoped: it applies from the directive to the end of the enclosing scope. It is allowed at namespace scope, function scope, or any nested block. It is a
declaration, not an executable statement, and may appear anywhere a variable declaration can.

i. Switch Integration

The switch statement (§12) accepts enum values directly. Value-list cases use the colon-scoped form:

switch (d) {
    Direction:North, Direction:South:   doVertical();
    Direction:East:                     doEast();
    Direction:West:                     doWest();
}

The case-label terminator colon and the enum-scope colon are syntactically unambiguous: the enum-scope colon is followed by an identifier, the case-label colon by an
expression or block. With a use Direction; directive in scope, the same switch reads:

use Direction;
switch (d) {
    North, South:   doVertical();
    East:           doEast();
    West:           doWest();
}

There is no exhaustiveness check. A switch that doesn't cover every named value is fine; if no case matches, execution continues past the switch (the standard noop
behavior from §12).

j. is and typeof

is and typeof on a plain enum-typed variable are settled at compile time, because there is no runtime type tag to consult:

Direction d = ReadDirection();
if (d is Direction) ...;                // always true; the compiler folds it
if (typeof(d) == Direction) ...;        // same

These forms are generally not useful for plain enum variables — the compiler knows the answer — but they're not errors either. Where they do useful work is when the
enum lives inside a var:

var u = SomeFunction();
u = Direction:North;                    // u's variant set now includes Direction
if (u is Direction) DoDirectionThing(); // runtime check via the variant's tag

In a variant, the enum participates in the standard tag-based dispatch. Without the variant, the enum is just an integer with a name.


23. Structs

a. Struct Declaration

Struct syntax is borrowed from C/C++, for the most part.

struct TypeName         // TypeName is skippable, in which case we have an anonymous ustruct type
{
    typename variablename = defaultvalue;
    ..
} instanceName;         // instanceName is skippable, in which case nothing is instantiated.

Structs don't have function members, per se, but they can have named pointers to functions.

b. Instantiation and Initialization

A struct always has a default constructor that takes arguments matching its member list.

var s = SomeStruct {membervalue1, ...};         // initialization block parameters are assigned to struct members in the order they appear in the struct declaration
                                                // they can be omitted right to left, ommitted values become null

SomeStruct s = {membervalue1, membername3 = membervalue3, membervalue4, ..};       // they can be named, they can be skipped left to right by naming them

s = {arglist..};                                // they can be reassigned later using brace syntax.

s = [{arglist..}, {arglist..}, ..][selector];                                       // list comprehension used to select an assignment specifier
s = {"this": {arglist..}, "that": {arglist..}, "other": {arglist..}}[selector];     // dictionary comprehension used to select an assignment specifier

c. implicit construction

You can pass construction parameters to functions that take a struct just like you can in C++.

d. Struct Inheritance

A struct can inherit from multiple other structs. (See section 25 on inheritance.)

e. Anonymous Structs

An anonymous struct is a struct type the programmer declares by initializer rather than by name. The compiler synthesizes a fresh internal type tag, the struct's members
are exactly the fields named in the initializer (in the order they appear), and member types are inferred from the initial values:

var p = { .x = 1, .y = 2 };       // anonymous struct: { i32 x, i32 y }
var u = { .name = "Alice", .age = 30 };

print(p.x);                       // member access works as for any struct
u.age++;                          // member assignment works as for any struct

Anonymous structs are useful when a small, single-use composite value needs no name of its own — return aggregates from a function, group a few related values to pass as 
a single argument, build up a data record without declaring a struct first.

Type identity is structural. An anonymous struct's synthesized type tag is composed deterministically from its field names and field types — independent of where it was
written. Two anonymous-struct literals with the same fields and field types produce the same type, and values of these literals can be assigned and compared as one would 
expect:

var p1 = { .x = 1, .y = 2 };
var p2 = { .x = 3, .y = 4 };      // same anonymous type as p1
p1 = p2;                          // ordinary same-type assignment

LookUp(i32 id) {
    var entry = table.find(id);
    return { .found = entry != null, .name = entry ? entry.name : "" };
}

Process(let r) { print(r.found); }

var r1 = LookUp(42);
var r2 = LookUp(43);
Process(r1);
Process(r2);                      // both calls work; r1 and r2 share a type

Field order in the source does not affect type identity (the compiler sorts fields by name when composing the type tag), but field names and types do. A field renamed 
or with a morphed type produces a different anonymous type. Field morphing follows the usual rules: the morpher tracks each field's current type, and two anonymous-struct
values are the same type only if every field has morphed to the same base type at both sites.

The internal layout of the struct (field order in memory, padding, alignment) is determined by the compiler for performance and is independent of source order or the
synthesized type name. Promoting an anonymous struct to a named type. Once you find yourself writing the same field set in multiple places, declaring a named struct lets 
all of them share a single type:

struct Point { i32 x; i32 y; }

Point p1 = { .x = 1, .y = 2 };    // designated initializer for the named struct
Point p2 = { .x = 3, .y = 4 };    // same type as p1

The same { .field = value } syntax that produces an anonymous struct in inferred-type contexts produces a named-struct initializer when the type is declared. The literal
form is shared; the type identity is determined by where the value is used.

Functions returning anonymous structs. A function can return an anonymous struct by inferring its return type:

LookUp(i32 id) {
    var entry = table.find(id);
    return { .found = entry != null, .name = entry ? entry.name : "" };
}

var r = LookUp(42);
if (r.found) print(r.name);

The function's inferred return type is the anonymous struct synthesized at the return site. Callers receive a value of that type and can access its fields. The synthesized
type is shared across all calls to the same function — every caller of LookUp gets the same anonymous-struct type — so passing the result around behaves as it would for any
named type.

Anonymous struct in a generic context - When a generic function returns an anonymous struct whose field types depend on the generic's type parameters, the synthesized type
is parameterized along with the function: each specialization of the generic gets a corresponding specialized anonymous struct. The morpher tracks these alongside its other 
generic-specialization machinery (§27).


24. Classes

Classes are the same as structs except they have function members, and you can supply coded constructors and destructors.

class MyClass {
    MyClass() {};
    ~MyClass {};
}


25. Inheritance

A class may inherit from one or more other classes by listing them after a colon in the declaration:

class Animal { ... }
class Talker { ... }
class Dog : Animal, Talker { ... }

A struct may similarly inherit from other structs (section 23). The rules in this section apply equally to class and struct inheritance, with the
single difference that structs cannot declare function members directly and so the inherited-function-call discussion below applies to classes
only.

Inheritance in xmc composes member sets. A derived class has access to all members declared by itself and by every class in its base chain, and
the member-lookup rules below define which member a given access resolves to when more than one candidate exists.

There is no virtual dispatch, all member calls resolve statically based on the static type at the call site. Polymorphic-style behavior is expressed 
through var parameters and the implicit-generics mechanism of section 27, not through runtime type tags or vtables.

a. Classes as Scopes

A class is a scope, the same kind of thing a file, a namespace, or a nested block is. The colon qualifier reaches into a scope by name to
resolve the next identifier:

filename:freeFunction         ; into a file scope
mymodule:helper               ; into a namespace scope (section 3)
MyClass:GetCount              ; into a class scope (static member, section 26)
Talker:Speak                  ; into a base class scope (inherited member)

These are the same operator. What differs across the four lines is what kind of thing the identifiers name.

b. Member Lookup

When the compiler resolves a member access — myDog.Speak(), or just Speak() inside a Dog member function where the receiver is implicit —
it uses a breadth-first search over the class and its bases.

The algorithm:

    Start with the receiver's class as the current frontier.
    Repeat:
        Examine every class in the current frontier for a directly-declared member matching the name and signature. If one or
            more matches are found, the leftmost match (by base-list order at this depth) wins. Stop.
        Otherwise, expand the frontier to include the next level: every class in the current frontier contributes its own bases, in
            left-to-right order, to a new frontier. Classes already visited during this lookup are not re-added.
        If the new frontier is empty, the lookup has failed.

A failed lookup is not a compile error. It produces a warning at the call site and the access becomes a noop, consistent with the noop-
contagion rules of section 27. A read of a missing member yields a typed null in context; a call to a missing function performs no
operation; an assignment to a missing data member is discarded. This keeps inheritance lookups uniform with fluid-type lookups, var-type
lookups, and other places in xmc where missing members are recoverable rather than fatal.

The visited-set rule handles diamond inheritance correctly. In:

class X { ... }
class A : X { ... }
class B : X { ... }
class C : A, B { ... }

a lookup starting at C visits C at depth 0, then A and B at depth 1, then X at depth 2 — exactly once, reached through A by the leftmost
rule. There is no double-traversal of X and no double-counting of its members.

c. Why Breadth First Search is used

BFS is the rule that matches what most programmers expect when multiple inheritance is used to compose capabilities rather than to
build deep hierarchies.

class Animal { Speak() { print("generic animal noise"); } }
class Loud : Animal { }
class Talker { Speak() { print("hello"); } }

class Dog : Loud, Talker { }

myDog.Speak();                ; resolves to Talker:Speak

The DFS reading would walk Loud's chain to Animal first, find Animal:Speak there, and use it — silently ignoring the directly-
declared Talker:Speak that's right in Dog's face. BFS finds Loud and Talker at depth 1, no match in Loud, match in Talker, done. The
member that's "closer" in the inheritance graph wins, and depth beats left-position.

When two classes at the same depth both match, the leftmost wins. This is the only place the base-list order matters for lookup.

d. Disambiguation by Override and Delegate

When a programmer wants a different resolution than BFS produces, the mechanism is to declare an override on the derived class that explicitly
delegates to the desired base:

class Dog : Animal, Talker {
    Speak() { Talker:Speak(); }       ; explicit override + delegate
}

The override declares Speak directly on Dog, which BFS finds at depth 0 before reaching either base. The body uses the colon qualifier to call
Talker:Speak explicitly. The implicit self of the enclosing Dog:Speak body is forwarded as Talker:Speak's receiver — the call from inside a
member function does not require an explicit receiver, because self is already in scope.

This pattern is the universal mechanism for any disambiguation question in xmc:

  - Want the leftmost-but-deeper inheritance member instead of the rightmost-but-shallower one? Declare an override on the derived
    class that delegates to the leftmost.

  - Want to call a base's version from inside an override that does its own work first? Declare the override and call Base:Member() at the
    appropriate point in the body.

  - Want to expose a base member that BFS would shadow? Declare a forwarding wrapper on the derived class.

The pattern is verbose at the declaration site, but the verbosity is the point — disambiguation is visible in source. A future reader sees Dog's
Speak override and can immediately tell which parent it routes to, instead of having to remember the BFS rule and trace the inheritance
graph mentally.

e. Calling Inherited Members from Inside a Member Function

Inside a member function of class C, the qualified form Base:Member() calls Base's version of Member with C's self forwarded as the receiver,
provided Base is one of C's bases (directly or transitively). The qualifier expresses two things together: which scope to look in
(Base's), and that the call is bound to the enclosing self.

class Dog : Animal {
    Speak() {
        Animal:Speak();          ; calls Animal's Speak with self = this Dog
        print(" — and woof");
    }
}

The same form used outside a member function, or with a class that is not a base of the enclosing class's static type, does not get implicit
self forwarding — it's an unbound reference per section 18c, and either needs an explicit receiver argument or is being used as a function
reference rather than a call. The Morpher distinguishes the two cases by the syntactic context (call form vs. reference form) and by whether
the qualified class is a base of self's type.

f. Receiving a Derived Instance Through a Base Parameter

A function declared to take a Base parameter can be called with a Derived argument:

class Animal { Speak() { print("generic"); } }
class Dog : Animal { Speak() { print("woof"); } }

MakeNoise(Animal a) { a.Speak(); }

MakeNoise(Dog());                ; legal — Dog is-an Animal

Inside MakeNoise, a has static type Animal. The call a.Speak() resolves through BFS starting at Animal — it cannot reach Dog's Speak. The body
sees only the Animal portion of the passed instance.

This is the static-dispatch promise: what the function does is determined at compile time by the parameter's declared type. The Dog
that was passed in does not change MakeNoise's behavior. Programmers who want behavior to depend on the actual passed type use a var
parameter:

MakeNoise(var a) { a.Speak(); }

MakeNoise(Dog());                ; cloned and morphed for Dog → calls Dog:Speak
MakeNoise(Animal());             ; cloned and morphed for Animal → calls Animal:Speak

The var parameter route invokes section 27's implicit generics: the body is cloned at each call site and the clone is morphed against the actual
argument type. Each call site gets its own specialization, and member accesses inside the clone resolve against the actual type rather than a
declared base. This is xmc's polymorphism — opt-in, expressed in the parameter type, no runtime dispatch overhead.

g. Generic Members Inherited Through Bases

A function member declared with var parameters in a base class can be inherited by a derived class. Calls to it from a derived instance follow
the implicit-generics rule of section 27: the body is cloned at the call site and morphed against the actual argument types, with self
substituted for the derived type rather than the base type.

class Container { Add(var item) { items.append(item); } }
class Bag : Container { }

ref b = Bag();
b.Add(42);                       ; clone of Container:Add, self = Bag, item = i32
b.Add("hello");                  ; another clone, self = Bag, item = zs8

Inside each clone, member accesses through self resolve against Bag's class scope using BFS. If Bag declares its own items field that shadows
Container's, the clones see Bag's. If not, BFS finds Container's items at depth 1 and uses that. Either way the resolution is static and
specific to the clone.

h. The is Operator and Inheritance

The is operator (section 18i) returns true when a value's static type is the queried type or is derived from it:

class Animal { ... }
class Dog : Animal { ... }

ref d = Dog();
d is Dog                         ; true
d is Animal                      ; true — Dog inherits from Animal

The check walks the declared inheritance graph at compile time when both sides are known, and emits a constant true or false. When the left side
is a var or a fluid type whose actual class is not statically known, the check becomes a runtime test against the type tag carried by the value.

For function-signature is checks, inheritance does not enter into it — signatures match exactly on parameter and return types per section 18i,
and a function taking Dog is not interchangeable with a function taking Animal even though Dog is-an Animal.

i. Hard Errors vs. Noop Failures

Most inheritance-related lookup failures are noops: a missing member through BFS produces a warning and a runtime null/skip, as described in
subsection b. There is one case that is a hard compile error rather than a noop:

A single class declaring two function members with the same name and the same full signature (parameter list and return type both identical) is a
contradictory declaration. It cannot be resolved by any rule because the class itself has provided two equally valid candidates with no
distinguishing context. The compiler reports this as a fatal error at the second declaration's site.

This is distinct from the "no member found" case (which is a noop) and from the "multiple candidates across bases" case (which BFS resolves
deterministically). It catches a class-author mistake at the source of the contradiction.

j. Class Declaration Completion

A class's member set is fully known only after the closing } of its declaration. During parsing, references to a class's members from other
files or from later in the same file may need to wait for the class to complete before BFS can run.

The compiler tracks per-class declaration completion as a synchronization point. A reference that arrives before the class has finished parsing
records itself as a pending lookup; when the class's declaration completes, all pending lookups against it re-run.

This is an implementation concern that programmers do not need to think about, but it shapes the order in which diagnostics may appear: a
warning about a missing member is emitted only after the receiver's class has fully declared, even if the call site was parsed earlier.

k. Restrictions

A class may not directly or indirectly inherit from itself. The inheritance graph must be a Directed Acyclic Graph. Cycles are detected during class-
declaration processing and reported as fatal errors at the declaration that closes the cycle.

A class may not inherit from a struct, and a struct may not inherit from a class. The two are kept separate because their semantics around
function members and constructors differ (section 24) and mixing the two would produce unclear member-set semantics. A class that wants
struct-like data layout with no functions can simply declare none.

There is no concept of private, protected, or public inheritance. All inherited members are accessible through the derived class. Access
modifiers in general are not part of xmc's design — visibility is managed through file and namespace scoping rather than through per-
member access flags.


26. Static Function Members and Once Data Members

xmc provides two modifiers for class-scoped members that depart from ordinary instance semantics:

  static  -- on function members. The function belongs to the class scope, takes no implicit receiver, and is called with the
             colon qualifier. Conceptually similar to "static methods" in C++/C#/Java.

  once    -- on data members and on local variables in any function body. The storage initializes lazily on first access and
             persists for program lifetime. Initialization is thread-safe and runs at most once.

The two modifiers are independent. A class may have static members, once members, both, or neither. A non-class context (a free function
body) may have once locals but cannot have static or once *members* in the class-scoped sense; the static/once-member discussion below
specifically covers class-level declarations.

Inheritance interactions for static and once members are covered in section 25 (Inheritance). This section focuses on the modifiers
themselves.

a. Once Locals in Function Bodies

The once modifier is permitted on local variable declarations inside any function body, including free functions, member functions, static
functions, and nested functions. A once local has a single storage location associated with that source-position declaration, shared
across every invocation of the enclosing function. The initializer runs the first time control flow reaches the declaration; subsequent
reaches skip the initializer and use the existing value.

The classic singleton pattern falls out directly:

*Service GetInstance() {
    once *Service instance = new Service();
    return instance;
}

The first call to GetInstance allocates a Service, stores the pointer in instance, and returns it. Every subsequent call returns the same
pointer with no allocation, no synchronization beyond the load of the once flag, and no per-call cost beyond what an ordinary function call
already incurs.

The * (raw pointer) form is appropriate here precisely because once storage outlives every caller. There is no aliasing question to
manage, no reference count to maintain, and no possibility of the pointee's storage going away while a caller holds the pointer. raw
pointers to once-allocated storage are the safe, cheap form for this pattern. The arc form (&Service) would also work and might be 
appropriate if the pointee's lifetime were less than program-lifetime, but for once-managed allocations it adds overhead without buying 
anything.

b. Initializer expressions and capture

A once local's initializer can reference anything in scope at the declaration point: free functions, class-level data, file-scope
values, function parameters, and other locals declared earlier in the function body. The initializer runs once, on the call that first
reaches the declaration, with whatever values those references hold at that moment. Subsequent calls do not re-run the initializer; they
use the value captured by the first call.

This is sometimes the desired behavior:

Logger GetLogger(zs8 path) {
    once Logger logger = Logger(path);
    return logger;
}

The first caller's path is captured. Subsequent calls receive the same Logger regardless of what path they pass. This is a deliberate
language feature for first-call-wins caching.

When the desired behavior is per-key caching rather than first-call-wins, a map keyed on the parameter is the appropriate structure, not
a once local. The two patterns express different intents.


c. Once locals in nested scopes

A once declaration inside a conditional or nested block initializes the first time control flow reaches it, which may be never. A function
that declares ten once locals across ten branches pays initialization cost only for the branches the program actually executes:

zs8 LookupOrFail(i32 key) {
    if (key < 0) {
        once zs8 fallback = ComputeFallback();
        return fallback;
    }
    return Table[key];
}

If LookupOrFail is never called with a negative key, ComputeFallback never runs and fallback is never allocated. Each once declaration
incurs storage and initialization cost only on demand.


d. Once locals in member functions

A once local in a member function is per-function, not per-instance.
Two instances of a class calling the same member function share the
same once local, because there is one declaration site in the source
and once binds to the declaration site, not to the instance:

class ServiceLocator {
    *Service GetService() {
        once *Service instance = new Service();
        return instance;
    }
}

ref sl1 = ServiceLocator();
ref sl2 = ServiceLocator();
sl1.GetService();        // constructs Service, returns pointer
sl2.GetService();        // returns same pointer; instance already initialized

Programmers who want per-instance lazy initialization should use an
instance data member, not a once local. The two patterns are distinct
and the difference is observable.


e. Once locals in generic functions

A function with var parameters is cloned per call site under section 27's implicit-generics rule. Each clone is a separate function for
the purpose of once-local storage: the clone for MakeNoise(Dog) and the clone for MakeNoise(Cat) have independent once locals that
initialize independently and do not share state.

This is consistent with the implicit-generics model -- clones are separate functions sharing source -- and is usually the desired
behavior for caching keyed implicitly on type. Programmers should be aware that a once cache inside a generic function is per-
specialization, and a "this should be one cache for all callers" assumption breaks when the function becomes generic.


f. Once locals and immutable

A once local can be combined with the immutable qualifier (introduced in earlier sections) to express "compute once on first access, then
never change":

immutable once Config c = ParseConfigFile(path);

The first call computes c. Subsequent calls cannot reassign c, even though the storage is mutable in principle. The combination is a
runtime-computed-then-frozen value, distinct from compile-time constants but equally fixed once initialized.

Assigning or otherwise altering the value ot an immutable results in a compile time error.

g. Once Data Members on Classes

A class can declare once data members alongside its instance data members:

class Counter {
    once i32 instances  = 0;
    once i32 totalCalls = 0;

    i32 myId;

    Counter() {
        myId = instances;
        instances += 1;
    }

    static i32 GetCount() { return instances; }
}

Once data members occupy storage shared across all instances of the class. There is exactly one Counter:instances regardless of how many
Counter objects exist. The storage location is associated with the class, not with any particular instance.

Access uses the colon qualifier on the class name, or the dot form on an instance which the compiler rewrites to the colon form:

Counter:instances              // direct access through the class scope
ref c = Counter();
c.instances                    // rewritten to Counter:instances at compile time

The dot-form rewrite means the natural-feeling instance.member access works for once members the same as it does for instance members.
Adding or removing the once modifier on a member does not break call sites that use the dot form. The instance expression in the dot form
is evaluated for any side effects, then discarded; the access reads or writes the class-level storage.

Once data members are reassignable. The once modifier governs initialization, not mutability. To express "initialize once, never
change," combine once with immutable:

class Config {
    once immutable zs8 buildId = ComputeBuildId();
}

h. Static Function Members on Classes

A class can declare static function members that belong to the class cope rather than to instances:

class Counter {
    once i32 instances = 0;

    static i32 GetCount() {
        return instances;
    }

    static Counter Create() {
        return Counter();
    }
}

A static function takes no implicit receiver. Inside its body, there is no self. It can read and write once data members of its own class
and call other static members; it cannot reference instance data members or call instance functions, because there is no instance to
reference.

Static functions are called with the colon qualifier:

i32 n = Counter:GetCount();
ref c = Counter:Create();

A static call written through an instance is rewritten by the compiler to the class-qualified form, parallel to the rewrite for once data:

ref c = Counter();
i32 n = c.GetCount();          // rewritten to Counter:GetCount()

The instance c is evaluated for side effects but not passed to the function. As with once data, this rewrite means refactoring a function
between static and instance does not break call sites that use the dot form, only sites that depend on receiver-binding semantics.

i. Static and Once Together

A class often has static functions that operate on once data:

class Registry {
    once map<zs8, *Service> entries = map<zs8, *Service>();

    static Register(zs8 name, *Service svc) {
        entries[name] = svc;
    }

    static *Service Lookup(zs8 name) {
        return entries[name];
    }
}

Registry:Register("auth", new AuthService());
ref svc = Registry:Lookup("auth");

The combination is the natural xmc spelling of what other languages call "module-level state with module-level functions." A class with
only static and once members fills that role. xmc has no separate namespace or module construct beyond file scope; a class with only
once and static members is the finer-grained alternative when the grouping deserves an explicit name distinct from the source filename.

j. Function References to Static Members

A function reference taken from a static member needs no receiver, because there is no receiver to capture. Static function references
are taken with the colon qualifier:

ref fn = Counter:GetCount;     // reference to a static function
i32 n = fn();                  // called with no receiver

This parallels the dot form for instance methods (section 18c), which yields an unbound reference requiring an explicit receiver argument.
The colon form yields a callable directly; the dot form yields a callable that takes the receiver as its first argument.

For a generic static function, the specialization syntax from section 18d applies:

class Math {
    static var Max(var a, var b) { return a > b ? a : b; }
}

ref fn = Math:Max(i32, i32);   // specialization
i32 m = fn(3, 7);

Static function references can be stored in containers, assigned as
runtime attributes on fluid types, and managed through pointer types
exactly as in sections 18e through 18g.

k. Static Functions vs. Free Functions

A static function on a class with only once and static members is
functionally identical to a free function organized under that class
name. The choice between

class StringUtils {
    static zs8 Trim(zs8 s) { ... }
}

StringUtils:Trim(" hi ");

and

zs8 Trim(zs8 s) { ... }                  // file-scope free function

strings:Trim(" hi ");                    // qualified by file
Trim(" hi ");                            // unqualified inside the same file

is one of organization rather than capability. A class with only static members groups names under an explicit qualifier that does not
depend on the source filename and that can be moved across files without breaking call sites. A free function inherits its qualifier
from the file. Use whichever expresses the intended grouping more clearly. For names that conceptually "belong with" a particular type,
making them static members of the class is usually the clearer choice; for names that don't have a strong type association, file
scope is simpler.

l. Initialization

Once data members and once locals share one initialization rule: storage is zero-filled at program start and the initializer runs the
first time the program reads or writes the storage. 

[Zero-filled at program start? This needs review. I don't think that should be the case. Once locals are our only way of achieving local 
statics. If they want them to be zero-filled, let's give them a way to express that with code. We want things to be "fast by default", 
and then slow down, only when and to the extent necessary, as the programmer calls for higher level functionality.]

Compile-time constant initializers

When an initializer expression is fully resolvable at compile time -- arithmetic on literals, references to other compile-time-constant
values, and similar side-effect-free computations -- the compiler evaluates it during compilation and emits the result directly into
the data section. No initialization code runs at all for such members or locals; their first access is an ordinary load.

This handles the common case of once integer constants, lookup tables, and small constant structures with zero runtime overhead.


m. Runtime initializers

When an initializer cannot be evaluated at compile time, the compiler emits an initialization function and arranges for first access to
trigger it. The trigger uses an in-line flag stored alongside the storage when alignment permits, so an access loads both the flag and
the value in a single memory operation. The fast-path access pattern is:

    load flag-and-value
    if flag indicates done: use value
    else: call initializer, then use value

The initializer runs at most once. It uses an atomic compare-and-swap on the flag to claim the right to run; concurrent threads that lose
the race  wait for the winner to publish the done state  and then proceed with the initialized value. Once initialization completes,
all subsequent accesses take the fast path with no synchronization beyond the load itself. (The implementation is designed to take 
advantage of CPU branch prediction.)

n. Self-referential initializers

An initializer that reads its own storage -- directly or transitively through calls into other code that eventually accesses the same
storage -- observes a zero-initialized state. [Are these cases readily detectable? Maybe we zero fill, as part of the run-time-triggered 
initialization process, when we detect a self referencing initialization.] The flag at that point is in the "running" state, not "done"; 
rather than deadlock or recurse infinitely, the read returns the underlying zero bytes interpreted as the storage's type.

Under xmc's null-and-noop semantics (section 27), this typically yields a typed null in the reading expression's context, and noop
contagion handles the rest gracefully. The compiler emits a warning when it can statically prove a self-reference. The runtime behavior
is well-defined and recoverable rather than undefined or fatal. 

o. Order of initialization

Because initialization is triggered by first access, the order in which a program's once values initialize is determined by which one
the program touches first, then which ones that initializer touches, and so on transitively. There is no implicit alphabetical, declaration-
order, or file-order rule. 

This rule replaces the static-initialization-order problem familiar from C-family languages. There is no concept of "before main" 
initialization in xmc, and no need to reason about translation-unit ordering. Initializers run when the value is first needed, and the
order is always derivable from program execution rather than from build-system layout.

A consequence: if two once values each have side effects in their initializers (printing, registering with a global, opening a resource), 
the relative order of those side effects depends on access pattern. Programmers who rely on a specific order should make it explicit by 
reading the values in the desired order at a known point, typically near the top of main. This is documented as a discouraged
pattern; initializers should be side-effect-free or idempotent in practice.

p. Pointer and arc-pointer once values

A once value of pointer or arc-pointer type whose initializer yields null is indistinguishable from an uninitialized one: both states are
zero. The compiler recognizes this and elides the flag for such values when their initializer is the literal null or absent. First
access loads the pointer; the load itself is the test. This restores the C-level cost model for the common case of nullable pointer
fields.

q. Destruction

Once values of class types invoke their destructors at program exit in the reverse order of initialization. Values whose flag is still in
its initial state (never initialized) have no destructor call; their storage remains zero throughout the program lifetime and there is
nothing to destroy. This avoids the C++ pattern of every static destructor running at exit regardless of whether the corresponding
constructor ran in any meaningful sense.

For once locals specifically, the "reverse order of initialization" ordering applies across all once locals and once data members
program-wide -- there is one global initialization order determined by access sequence, and one reverse-of-that destruction order at
exit. [This means there is code emitted into the executable which creates some kind of concurrent stack that destruction code 
pointers are pushed onto at run time and popped off of on exit. I suppose that's acceptable, but we have to remember to implement it.]

r. Restrictions and Edge Cases

Constructors and destructors cannot be declared static or once. They are inherently per-instance operations.

A static function cannot reference instance data or call instance functions. The compiler reports such references as errors. A static
function can read and write once data and call other static functions freely.

Once data members cannot be declared with parameters or with anything that varies per-instance, because there is no per-instance context at
the once initializer's evaluation point. (Once locals inside a member function can reference instance state through self -- they execute in
the context of a particular call -- but their initialization is still shared across all calls.)

A class with only once and static members has no instance state and cannot be instantiated. Attempting to construct an instance of such a
class produces a warning and yields a value that is functionally a stateless handle to the class scope. This is rarely useful; the
recommended pattern is to call the class's static members directly through the colon qualifier rather than constructing an instance.

The static keyword on function members is unrelated to the static keyword in C/C++ that denotes file-local linkage on free functions.
xmc has no equivalent of C-static linkage; visibility is managed by file and namespace scoping (section 3), not by per-symbol linkage
flags.

27. Generic Code, Member Resolution, and Null Propagation

Functions may specify types they require, or they may take var types:

class User {
    
    zs8 name;
    list inventory;

    SetName(zs8 name)           - if a variant is passed that has an zs8, the zs8 component (which may be null) is passed.
    {
        me.name = name;                 - name gets a copy of the reference-passed name
        LastAssignedName := name;       - the (declared elsewhere) global, LastAssignedName, becomes a shared pointer, if it wasn't already. Lots of cascading compiler work.
    }
   
    AddInventoryItem(var thing)
    {
        inventory.add(thing)    - inventory's items will become variants (tagged unions) of each type of thing that is passed in here.
    }
}

..

arc zs8 LastAssignedName;       - a shared pointer to a zero-terminated string. 
arc var LastAssignedItem;       - a shared pointer to an inferred type.

a. Implicit Generics via Unresolved Parameters

The xmc language does not have explicit templates, however similar generic behavior can be achieved using var parameters. A function with 
a var parameter whose type is not resolved inside the function is inlined at each call site and the types are resolved from the context there: 

RemoveNode(**var pp, *var target) {
    while ($pp != target)
        $pp = $pp->next;
    $pp = target->next;
}

A separate version of this function is produced at each call site where the types differ. 

b. Member Resolution

An instance of a type in xmc can have members in three ways:

    - Declared members of a class or struct
    - Runtime-added attributes (present when the type has morphed to fluid)
    - Neither

A member access is never a compile-time error due to the member not existing. Instead the compiler resolves it as early as it can: If the 
type is rigid and the member is a declared class or struct member, the compiler resolves it statically and emits a direct field 
access — optimal code, no overhead. If the type is fluid, the compiler emits a runtime attribute lookup that checks whether the instance 
currently has that attribute and returns its value, or an error variant if it does not. If the type is an unresolved var, the compiler defers 
resolution to the call site. If the call site provides a rigid type with the member declared, static access is emitted. If the call site 
provides a fluid type, runtime lookup is emitted.

c. Missing Members and Error Variants

When a member lookup fails at runtime — whether because the type is rigid and never had that member, or because the type is fluid and the 
attribute hasn't been added yet — the result is an error variant rather than a crash or undefined behavior. This error variant:

    - Resolves to null if not explicitly extracted, with the null type determined by context
    - Propagates through the error block mechanism {? ?} in the normal way
    - Can be caught with the error operator |

The type of the resulting null is determined by the nearest available type anchor — the declared type of the left hand side, the type of 
another operand in the same expression, or the type expected by the context. If no type anchor exists anywhere in the context, the null 
remains untyped.

d. Noop Contagion

An untyped null in a typed context resolves cleanly to a typed null of that type. But an untyped null in a fully untyped context — where no 
type information is available at any level — becomes a noop, and noop is contagious: any operation that depends on a noop result is itself 
a noop. This is analogous to NaN propagation in floating point, but more general.

In practice noop contagion is contained, because most expressions have at least one type anchor nearby. It only spreads in deeply generic 
var-everywhere code where no type information is available at any level. Specifically:

    - Untyped null in a boolean context → false
    - Untyped null in a typed variable context → null of that declared type
    - Untyped null as one operand in an expression with a typed other operand → null of that other operand's type
    - Untyped null in a fully untyped context → noop, spreads to containing expression

A noop in a boolean context — such as a while loop condition — evaluates to false, causing the loop or branch to exit or not execute. This 
means generic code that operates on a type lacking an expected member degrades gracefully: the loop exits, the branch is skipped, and no 
crash or undefined behavior occurs. The behavior may not be what the programmer intended, but it is safe and predictable.

e. Generic Code Example With Graceful Degradation

PrintName(var thing) {
    print(thing.name);     ; static access if thing has a declared name member,
}                          ; runtime lookup if thing is fluid,
                           ; noop (print receives null) if neither

struct Dog  { zs8 name; }
class  Cat  { zs8 name; }

PrintName(Dog { "Rex" });  ; static — Dog:name is a declared struct member
PrintName(Cat { "Miso" }); ; static — Cat:name is a declared class member

var mystery;
mystery.name = "Ghost";    ; mystery is fluid, name is a runtime attribute
PrintName(mystery);        ; runtime lookup — finds "Ghost"

var nameless;
PrintName(nameless);       ; nameless has no name member or attribute
                           ; thing.name produces an error variant
                           ; resolves to null zs8 (print's parameter type provides the anchor)
                           ; print receives a null zs8 and prints nothing

For the RemoveNode example, if a type without a next field is passed in, $pp->next produces an error variant, the assignment to $pp is a noop, 
$pp is unchanged, $pp != target remains true on the next iteration — which is an infinite loop. This is the one case where graceful degradation 
is not sufficient and an explicit guard is warranted:

RemoveNode(**var pp, *var target) {
    while ($pp != target) {?
        $pp = $pp->next | { return; }    ; exit if next is null or missing
    ?}
    $pp = target->next;
}

A. Implementation Notes

- compiler input and output

When the compiler is invoked, CommandLineReader parses argc/argv into a CommandLineReader::Args struct (ProjectName, ConfigName, FileFilter, 
TestFilter, Full, Test, Suite). ProjectFileReader then reads <project>.xmc, merges the base section with the active config, and exposes typed 
accessors for all settings (IntPath, OutPath, Sources, LibPaths, logging flags, etc.).

Builder::Build derives one or more CompileJob values from Args and the project. Normally one CompileJob covers all source files. When a 
file_filter is active, or when --test or --suite is used, one CompileJob is produced per matching source file. 
Each CompileJob carries: SourceFiles, IntDir, ObjPath, ExePath, Full, InjectTestMain, TestFilter, and per-component log file paths.

For each CompileJob, Builder calls Compiler::Compile (which drives the pipeline described below and produces the .obj) and then, if that succeeds, 
Linker::Link (which invokes the platform linker to produce the .exe).

- incremental build

Here's the incremental compilation process:

The source folders are walked and filenames and modtimes are loaded.
Files whose .xm's are newer than their .xmo's (or whose .xmo's aren't present) and considered modified.
The symbols, their types, their min and max refinements, the indexed machine code blocks and their lists of external dependants are loaded from all the unmodified .xmo's
The memory buffer for an .xmo is sized so that the parse tree can be loaded if needed.
The source code is loaded from the modified xm's.
The parse trees are regenerated for the modified xm's and added to the .xmo's in memory.
    - symbols are interned, their base types are inferred, and their refinements are morphed upward from defaults as requirements are encountered.
    - this doesn't happen by forward scanning, it happens by putting information into dependency graphs and back filling.
    - if morphing of refinements spreads to unmodified xmo's, this triggers a job shape determination and a parse tree scan to redetermine all base types and refinements. (job shape: we know which xmo's depend on the 
    - if base types (or variant type lists) change in unmodified xmo's, this triggers a job shape determination and a reparse to redetermine all 
      symbol types
Once symbol base types and refinements are determined, a second scan of the loaded xmo parse trees is performed to generate vectors of code block id's
for each parse tree. (see "code block generation" below)
The xmo's that are marked as dirty are saved at the end of the compilation process. 
Once xmo's are finalized and saved, the coff file is written out and the linker is invoked.

Full parse tree scan:

If a full xmo scan is triggered, the parse trees for all the unmodified .xmo's are loaded, and a single scan of all the xmo parse trees
is performed to remorph all the symbol refinements. A hashset is cleared and then filled during this, to determine which symbols have not yet been
visited and therefore remain at their default levels. Symbol refinements are copied from the global symbol table to the Xmo's that export or import them.
Xmo's whose symbol refinements are changed by this are marked as dirty.

- full build

Here is the full build compilation process -

The command line is parsed, the project file is parsed.
The intermediate directory is cleared, the .obj file and the .exe are deleted.
The source folders are walked and the .xm files are loaded.
An .xmo file is created in memory for each .xm file.
Parse trees are generated for each xmo.
    - Symbols are interned, their base types are determined, and their refinements are determend (morphed upward from defaults).
The parse trees are rescaned and code block id vectors are put into the parse trees.
The code block buffers of the xmo files are populated.
The xmo's are saved.
The coff file is written.

Symbol dependency vectors:

The symbol table contains vectors of symbols whose refinenments depend on that symbol's refinement. This information is saved in the xmo.
A symbol has a scope id vector that has the file id and the scope ids for the namespace scopes in the symbol's namespace path.

- testing

Three testing modes are supported, all driven through the normal build path:

Builder produces per-file CompileJobs for --test and --suite in the same way it
does for a file_filter build. The only differences are:

    --test   InjectTestMain is set on each CompileJob. The compiler fabricates a
             main() that enumerates test blocks by name, filters them against
             TestFilter (a wildcard; namespace paths use a single colon separator),
             and runs the matching ones. After Builder::Build completes,
             Tester::RunTests invokes each linked binary, captures its debug and
             error stream output, and reports pass/fail per test block.

    --suite  InjectTestMain is not set; each source file's own entry point (as
             declared in the project file's EntryPoint key) is used. After
             Builder::Build completes, Tester::RunSuite invokes each binary and
             compares its stdout, stderr, and per-component log files against
             golden files found alongside the source file:
                 <sourcefile>.lexer  <sourcefile>.parser  <sourcefile>.morpher
                 <sourcefile>.stdout  <sourcefile>.stderr
             A pass/fail report is produced for each file and each golden file
             that exists.

- Morpher vs Typer/Refiner

Typing and refinement are performed by a single Morpher stage, as noted above. They were originally separate (Typer feeding Refiner via a work queue) but were 
merged because refinement changes can induce type changes — e.g., a singular -> variant morph adds a type to the symbol's variant list — so the two phases cannot 
be cleanly separated into stages with a one-way dependency.

Gosh, there are a lot of little semantics in this language. I wonder how difficult it will be to actually implement them and get them all right.

The basic motivation for the language is to be able to write your game engine in the same language that you write your game in, and have 
a decent chance of releasing your game while you're young. With this language, your game code can basically add actually-fast features to the engine. 

In other words, it tries to give you a way to do very rapid prototyping, and then come back around and tighten up the things that turn out to need tightening,
hopefully without tremendously too much effort.

But I don't know how well I've done at designing such a language. With all these detailed semantics - you still have a lot to manage and think about.
