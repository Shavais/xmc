
Compiler overview - 

Here's a tentative plan for the parsing and emitting phases of the compiler
There is a (threadpool thread) task per source file, and within each source file thread, 
each of these are a separate spawned tasks which run concurrently alongside each other:

Lexor: 
	- table driven state machine that posts tokens into a concurrent queue for the parser 
	- emits syntax errors like "syntax error - file:line - expected one of [_,_,..] got [_]"
	- logs file:line:token entries into a token log file for a given project when lexor logging is requested
		(in the project file for the active config).
	- cleans up and halts if a "full scan required" event is raised

Parser (thread per file):
	- consumes tokens from the token queue
	- adds symbol and action parse tree nodes to a parse tree
	- posts newly created parse tree node references into the morpher queue
	- logs lines about created parse tree nodes into a parser log when parser logging is requested
	- cleans up and halts if a "full scan required" event is raised
		
Morpher:
	- consumes parse node refences from the morpher queue 
	- builds a base type dependency graph (symbol A's base type has dependents b & c, etc.) 
       with references to parse tree nodes and symbols
	- builds a refinement dependency graph (symbol A's refinements have dependants b & c..)
    - maintains a umap<symbol_key, 
	- discovers base type resolutions and updates the base types in the symbol table (which are referenced by the parse tree nodes)
    - discovers refinement morphs and increases the minrmask and decreases the maxrmask of symbols 
	- streams type resolution and refinement morphing errors to an oserror stream when error conditions arise
	- logs lines about discovered type inferences and refinement mask changes into a morpher log when morpher logging is requested
	- sets a job ReparseRequired flag and stops if a base type (or variant type list) needs to be changed in an unmodified source file whose 
        source has not been loaded
	- sets a job ParseTreeScanRequired flag and stops if a refinement needs to be changed in an unmodified source file whose parse tree has
        not been loaded (in its xmo)

Reviewer:
	- starts for a given source file when the lexor, parser and morpher have finished if no reload/restart sequence is required and no errors occurred
	- scans the depencency graphs for unresolved types 
    - scans 
	- streams unresolved type and refinement errors to oserror
	- checks for and performs some parse node level optimizations, when possible and convenient
	- notifies the coder if reviewing completes without errors
	
Coder: 
	- starts for a given xmo (source object file being constructed) when the reviewer notifies it of successfull completion
	- walks the parse tree
	- fills parse node code block id vectors:
		- looks up a vector of base type block id vectors for the node type, symbol basetype and action 
		- looks up a vector of refinement block id vectors for the node type, symbol refinement and action
		- stitches together block id vectors, tracking register usage
		- chooses alternate block id vectors when the defaults require registers that are in use
		- performs some block id vector level optimizations, when possible and convenient
	- notifies the emmitter upon successful completion


Emitter: (already implemented and tested.. contains no intelligence) 
    - awakens when the coder notifies it of successfull completion
	- walks the parse tree, consumes parse node block id vectors and produces machine code in xmo code buffers 
    - saves loaded xmo files that end up with a dirty state
	- consumes xmo code buffers and produces the single COFF output for the project
	  (Section 20 of "The xmc programming language.txt" file has notes about xmo files and incremental and full builds.)

I see three distinct types of testing that we need to facilitate.

One type consists of individual, small tests that we run as we're building out the compiler. 
We'll write a test and the implement the parts of the compiler that that test tests and then we'll compile and run the test.
It would be nice to have some way of building and running those test programs without having to copy and edit a project file for each one.

Another type is a set of pass/fail regression tests that we add to with each new thing we get working so we can convince ourselves
that recently added functionality hasn't broken anything. Maybe those are all in one UnitTest project or something.

A third type of testing that it would be nice to facilitate are application tests that a programmer puts into their program that they
can call out individually or as a whole.

5/9/2026

- New Plan

I want to avoid recursive descent style code. Having a function per production is too much code and too much thrashing of the stack.

Also, I'm suspicious that the threading model is a little bit over-threaded. A little bit too much sleeping on blocking queues.

Also, we're having a lot of trouble with the DFA -> LALR notion, I think probably because the node tree shape we seem to be intending to produce doesn't match 
the node tree shape naturally present in the BNF closely enough.

So, I have a slightly different idea for how to approach the compiler. It's evolutionary, not revolutionary, but I think it is a rewrite, or at least 
a substantial refactor. It might be essentially the same, but in my mind it is a bit different.

- The BNF shape is the parse tree shape

The non-terminal nodes in the BNF can be parse nodes in our parse tree, except that collections of tokens (such as numeric values) are collapsed into noun values.

We'll drive the compiler from the lexer. It'll have a big StringView (a struct with a "start" index and an "end" index that we locate in a CompileJob arena) that 
covers an entire source file (whose contents we have in a byte buffer called source). In the course of what it does, it'll create a StringView per non-terminal node. 
We'll have a current StringView that we're working with. 

We'll have a NextState table that looks like this:

[state: {nextchars: nextChars, nextStates: {Char: {nextstate: {action, nodekind, basetype, usertype, minrmask, maxrmask, LCR, consume}, ..}, ..}, ..}, ..]

where..

const char* nextchars;
uint16_t state;
char Char;
uint16_t nextstate;
uint16_t action;
uint16_t nodekind;
uint8_t basetype;
uint32_t usertype;
uint8_t minrmask;
uint8_t maxrmask;
int8_t LCR;
int8_t consume;

The Lexer starts (in a thread pool task for the file being compiled) with currentState = ST_START. 
It has a currentStringView which, after initialization, covers the entire source buffer: start = 0, end = length.
It begins by calling a Parser::CreateNode function to create node of NodeKind NK_FILE and sets TopNode to that node and currentNode to that node.
Then it enters the Driver Loop for the whole compiler.

Driver Loop:

Set target to currentStringView.start.
Look up the NextState entry for currentState, call it nextStateEntry
Move target forward until source[target] is one of nextchars.
(let currentChar = source[target])
Look up currentChar in nextStates to get nextState
If nextState nodekind is not NK_NONE:
    call Parser::CreateNode, passing a copy of currentStringView, nodekind, basetype, usertype, minrmask and maxrmask.
        Parser::CreateNode reports the node creation if ParserLog == 1
    call Parser::InsertNode, passing a current node reference and a new node reference and LCR. 
        Parser::InsertNode inserts the node into the parse tree either as a left sibling, a child, or a right sibling.
        Parser::InsertNode calls Morpher::ProcessNoun if the nodekind is one of the noun nodekinds.
            Morpher::ProcessNoun spawns a Morpher thread pool task to process the noun as far as it can so far.
                Morpher::ProcessNoun reports the noune if MorpherLog == 1
Move the target forward by consume (which might be negative).
Set currentStringView.start = target
Set currentState = nextState
If LexerLog == 1, report the transition

I'm not completely sure, but I think we can construct the Morpher such that by the time the driver loop reaches the end of the file,
all the right types and masks are propagated to their correct locations in the parse tree.

I'm not sure if we can write a python script to generate the NextState table. Maybe. We might have to hand tune a lot of the values,
or we might have to alter our BNF so that it conveys enough information to the python script.

We may also need to alter the BNF so that the node tree shape that falls out of it is the node tree shape we want, with operators being parents of LHS and RHS expressions.

Let's chat about this before we do anything. What do you think of this plan? Can it work? Do you think its less efficient? Do you think it can help resolve our DFA -> LALR difficulties?
