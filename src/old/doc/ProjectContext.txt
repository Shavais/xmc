The Lexer — Current State
The lexer has been fully refactored. The delivered files are Lexer.h and Lexer.cpp in namespace xmc. 

Key design:

Lexer::TokenType — enum class with all token types (literals, keywords, operators, punctuation, trivia, diagnostics)

Lexer::Token — struct with type, line, col, srcStart, srcLen, flags. Has FLAG_PRECEDED_BY_SPACE used by the parser to enforce no-whitespace rules on @ and $ operators.

Lexer::Result — contains std::vector<Token> tokens (all tokens including trivia) and bool hadErrors

Lexer::Lex(filename, source, parserQueue, logStream) — const member method. Takes a moodycamel::ConcurrentQueue<Token>& reference; parser-visible tokens (no trivia, no errors) are enqueued into it as they are produced. Returns a Result.

Lexer::InitTables() — public static, initialises DFA tables once (thread-safe). Safe to call at startup for pre-warming.

Lexer::TokenTypeName() — public static, returns human-readable name for logging.

IsParserToken() — private static.

Implementation internals (LexState, CharClass, LexAction, Trans, static tables s_charClass, s_trans, s_keywords) all live as file-scope statics inside Lexer.cpp only — not exposed in the header.

The lexer is a table-driven DFA. The tables are built once and shared across all Lexer instances.

Dependencies in Use

pch.h — precompiled header, included first in .cpp
Previously used third-party libs include parallel_hashmap (phmap) and BS_thread_pool

Compiler Pipeline — Design Intent

Threading model: One threadpool thread per source file. Within that, each compiler stage (Lexer, Parser, Typer, etc.) runs as its own threadpool task. 
One instance of each compiler class per source file. The DFA tables in Lexer are static so all instances share one copy.

Arena tiers:

Global SymbolArena — owned by Compiler, lives for the full compilation run. Holds interned strings and the global symbol table. 
Already sharded (SymbolTable[256]) with per-shard shared_mutex for concurrent access.

Per-file Arena — lives inside each Xmo, scoped to that file's compilation. Parse tree nodes, patch symbols, dependency graph nodes etc. 
are allocated here.

Lexer ? Parser handoff: Via moodycamel::ConcurrentQueue<Lexer::Token> — these two stages can genuinely overlap on the same file.

Typer — design intent:

* Receives parse nodes as they arrive from the Parser
* Builds a dependency graph of base type relationships
* Resolves types as anchoring information arrives (concrete types, literals, explicit annotations), propagating resolutions through the graph
* Defers unresolved nodes rather than blocking
* Cross-file base type dependencies are directed (caller infers from return type, or callee infers parameter type from call site) — never mutual, 
so never inherently cyclic. True cycles are user errors, not compiler limitations.
* A short second pass after all per-file Typers finish closes any chains that depended on cross-file symbols [I don't think this should be needed; if it is, I think that's an error condition, that should be caught by the Reviewer.]

Refiner — design intent:

* Same graph-build-then-propagate pattern, but for refinement morphing
* Refinement bits only ever get set, never cleared — atomic OR on Symbol::minrmask/maxrmask handles concurrent contributions from multiple files naturally
* A usage in any file can widen the refinement of a symbol defined in another file (e.g. B adding elements to a list passed in from A makes that list resizable)

[The Typer and Refiner have been merged into "Morpher"]

Reviewer — design intent:

* Not an inference stage — a validation and error reporting stage
* Traverses the Typer and Refiner dependency graphs after all nodes have been processed
* Any node still unresolved at that point represents a programmer error
* The graphs should carry enough context (where the dependency was introduced, what was expected, what was found) to allow the Reviewer to emit helpful, specific error messages
* Also validates that inferred refinement state is consistent with any explicit programmer declarations

    Empty | HeadersOnly | Full | Modified | Regenerating | LoadFailed

State on CompileJob (atomic):
    ErrorOccurred, ReparseRequired, ParseTreeScanRequired
    (Stages check these at loop heads with relaxed loads and unwind

=== SymbolTable ===

Per-CompileJob, not global. Constructed by Compiler with a
maxNamespaceDepth value sourced from ProjectFileReader's
"maxnamespacedepth" key (default 5, range 1-32).

Sharded 256 ways on the low byte of FNV-1a hash of the name.
Each shard owns its own Arena (name bytes and Symbol objects
cluster by hash for cache locality) and a phmap::flat_hash_map
keyed by InternedString.

Symbol is exactly 64 bytes and cache-line aligned:
    name (InternedString, 24)
    xmoIdx (uint32, 4)         -- index into Compiler::xmos
    baseType (atomic<u32>, 4)
    offset (uint32, 4)
    minrmask (atomic<u8>, 1)
    maxrmask (atomic<u8>, 1)
    allocType (1), pathLen (1)
    union { inline_path[5]; overflow_path; }   (20)

For pathLen <= 5 the path is inline (zero indirection). For depths
6-32, overflow_path points at arena memory. Symbols are placement-
new'd into the arena and never destructed -- their atomic and
trivial-only field set permits this.

InternSymbol must be idempotent on the regeneration path:
calling it with an (name, path, xmoIdx) that already exists must
return the existing Symbol pointer, not create a duplicate. This
is a TODO; current implementation always creates.

=== Xmo class ===

One Xmo per translation unit. Compiler owns
vector<unique_ptr<Xmo>>; xmoIdx into that vector is stable for
the entire job (slots are reserved before async load completes,
and regeneration mutates contents in place rather than replacing
the slot).

Xmo holds: identity (name, internedName, index), state
(atomic<XmoState>), content (Arena, codeBuffer, exports, relocs,
scopeTree, parseTree pointer, FileMapping), ownedSymbols vector
(non-owning back-references for fast reset and export emission),
and the dependency graph (observedBy: SymbolKey -> vector<XmoKey>;
importsOf: XmoKey -> unordered_set<SymbolKey>).

Concurrency:
    state is atomic.
    Dependency-graph mutation (RecordImport, RecordObserver,
    ResetForRegeneration, ScrubDependencyEdges) takes a per-Xmo
    shared_mutex.
    Other content fields are owned by one pipeline stage at a time;
    the barrier between stages is the synchronization point.

=== Incremental build flow ===

Two break-out flavors driven by the Morpher:

    ReparseRequired  - a structural change (base type morph, or a
                       structural refinement bit like Variant, Arc,
                       Concurrent, Fluid being set) on a symbol
                       observed by an unmodified xmo. The observer's
                       parse tree may compile to different node
                       shapes; source must be re-parsed.

    ParseTreeScanRequired - a non-structural refinement narrowing
                       on a symbol observed by an unmodified xmo.
                       Parse-tree shape stays valid; only the Morpher
                       needs to re-run on the loaded parse tree.

Cascade is a DAG walk over per-symbol observer edges. Per-symbol
(not per-xmo) observers because xmo-level granularity would force
regeneration of every observer when only a single symbol changed.

Cycle detection:
    Type-inference cycles are caught by a thread-local
    "currently inferring" set on Symbol* during recursion.
    Self-collision = real cycle (error). Collision with another
    thread's set = ordinary contention, wait via existing locks.
    Cross-xmo cycles aren't separately tracked; small-world DAG
    walks terminate naturally absent real cycles.

Breaking changes are programmer errors:
    A modified file removing or renaming a namespace-scoped symbol
    that unmodified xmos still import is detected by Reviewer
    (importsOf walk against current SymbolTable state) and reported
    as a fatal diagnostic. The compiler does not attempt to
    auto-update the unmodified files.

=== Error handling ===

Two tiers:
    Fatal errors stop the job (cycles, broken imports, corrupt xmo,
    unreadable source).
    Diagnostic errors are collected and reported but participate
    in expression tainting -- a typer that can't determine a type
    assigns Error and propagates Error silently, so a single bad
    expression contaminates downstream uses without producing a
    cascade of secondary diagnostics.

=== File format (planned, not yet implemented) ===

.xmo layout under design:
    [XmoHeader]
    [string pool]              -- all interned names referenced below
    [exports]                  -- per-symbol baseType/refinement/path
    [scopes]                   -- scope-tree records
    [observer edges]           -- (symbol_key, observer_xmo_name)
    [imports edges]            -- (origin_xmo_name, symbol_key)
    [code buffer]
    [relocations]
    [parse tree]               -- optional; only on Full-state saves

Names are stored as offsets into the string pool and re-interned
into the new job's SymbolTable on load. xmoIdx is never serialized
(reassigned at load time per the run's vector ordering).

Lexer tokens are not persisted: lexing is fast enough that caching
tokens isn't worth doubling the on-disk xmo size.

--- Parse tree as dependency graph ---

ParseTreeNode (allocated in the Xmo's arena):
    kind, parent, children
    baseType, minrmask, maxrmask     -- atomic; resolution state
    declaredSymbol                   -- for declaration nodes:
                                        the Symbol declared here
    payload                          -- kind-specific data

The parse tree IS the dependency graph. Resolution flows along
parent/child links: when a node resolves, the Morpher walks its
parent up to check whether the parent can now resolve.

Cross-symbol propagation uses a separate subscriber index because
references to a symbol may originate in many xmos and live in
many parse trees -- the symbol's declaration node is not the
right home for the list. See "Subscribers index" below.

Forward references (calling a function whose definition appears
later) are handled with Stub parse nodes: when Parser hits a
reference to an undeclared name, it interns a placeholder Symbol
(Unresolved type), creates a Stub ParseTreeNode for the future
declaration, and registers the Stub as the symbol's declaration.
Subsequent references subscribe to the Stub the same as to any
declaration. When the real declaration arrives, the Stub is
mutated in place into the real declaration kind, preserving its
subscriber list.


--- Subscribers index (in-memory, per Morpher run) ---

[ This needs to be per compile job. Morpher threads are using this to push morphs through their own parse trees and potentially through each other's. ]

    phmap::flat_hash_map<SymbolKey, SubscriberList*, SymbolKeyHash>
        subscribers[256];
    std::shared_mutex subscriberShardMtx[256];

SubscriberList is a lock-free append-only Treiber stack of
ParseTreeNode*. SubscriberLists are allocated from shard-local
arenas; addresses are stable for the run.

Operations:
    Append (Parser, on cross-reference resolution):
        shared lock on shard for map lookup, then CAS-push
        onto the list.
    Iterate (Morpher, on symbol morph):
        shared lock on shard for map lookup, snapshot head,
        release lock, walk list with no further locking.
    Insert new entry (Parser, on first reference to a symbol):
        unique lock on shard.

Subscribers are never removed. Inferences and refinement morphs
are cumulative -- a symbol may morph more than once at different
sites, and dependents must re-run each time. Stale subscriber
notifications are harmless: a re-fired node that's already at
its current resolved state simply no-ops.


--- Concurrent morph propagation ---

ParseTreeNode::baseType / minrmask / maxrmask are atomic.
Morphs are monotonic over the type lattice (base types only
become more specific; refinement bits only narrow toward
minrmask), so:

  - Concurrent updates to the same parse node serialize through
    the atomic store. Last write wins; with monotonic morphs,
    every order produces the same final state.
  - Re-fired notifications are idempotent. A re-enqueued node
    that's already at its current state is a no-op.
  - Walking parent pointers up the tree after a node resolves
    requires no locks. Each parent visit does an atomic store
    of its own resolved state if applicable.

Parse-tree mutation that grows children-vectors (e.g. a Call
node receiving its child argument-list when a forward-referenced
callee resolves) needs synchronization at the affected node only.
A targeted per-node-kind mutex (or a sharded lock pool indexed
by node address) handles this without burdening the common case.
Specific design TBD; deferred until profiling demands it.


--- Break-out recovery and frontier discovery ---

A break-out is signaled when the Morpher narrows a Symbol whose
xmo has observers in HeadersOnly state. Structural narrowing
sets ReparseRequired; non-structural sets ParseTreeScanRequired.

When the dispatcher detects either flag at the barrier, it
computes the affected closure:

    seed = { xmos that originated the morphs }
    expand seed by transitive closure over importsFromXmos
    until no new xmos are added.

Closure expansion uses file-on-file edges only. Per-symbol
granularity cannot be used for frontier discovery because we
don't yet know which exports of an unmodified xmo will morph
until we re-run the Morpher on it -- which requires its
dependencies to already be loaded.

ReparseRequired action: for each xmo in the closure, load source
if not loaded, ResetSymbolsForRegeneration, ResetForRegeneration,
restart Lexer -> Parser -> Morpher.

ParseTreeScanRequired action: for each xmo in the closure, load
parse tree if HeadersOnly, restart Morpher only.

Per-symbol subscriber index is rebuilt naturally as Parser/Morpher
process the closure -- no persistence required.


--- Function classification ---

The Morpher classifies each function based on whether all of
its var/let parameters resolve internally:

  - Fully resolved: function compiles to a single body. Coder
    may inline small instances at call sites for efficiency
    but is not required to.

  - Partially resolved (one or more var/let parameters with
    types that depend on call-site arguments): function is
    treated as a template. Coder inlines the body per call
    site against the actual argument types. Different call
    sites may produce different code.

A var parameter whose only internal use is in s8 contexts has
its base type resolved to s8 internally -- it does not require
template treatment.

minrmask on a parameter acts as a requirement on call-site
arguments: an argument that cannot morph to satisfy a
parameter's minrmask is rejected.

Function pointers carry a signature type (callable with these
argument types, returning this type). Calls through the pointer
resolve from the signature, not from any specific body. Function
pointers are not eligible for template-style inlining.


--- Reviewer ---

Per-file parallel pass after the pipeline barrier. For each
Symbol in each Xmo's ownedSymbols:

  - If baseType == Unresolved: emit warning ("inference did not
    reach symbol X"). Coder will treat assignments to this
    symbol and expressions involving it as no-ops.
  - If baseType == Error: ignore (already reported).
  - Otherwise: no action.

Reviewer does not set ErrorOccurred. Coder runs regardless.


--- Error and warning model ---

Two streams:

  Synchronous diagnostics from Parser/Morpher. Emitted at the
  failure site. The offending symbol or expression node is
  marked BaseType::Error. Error propagates silently through
  dependent operations (no cascade).

  Reviewer warnings for genuinely unresolved symbols (never
  visited by the Morpher because their pending dependency never
  fired). Distinct from Error -- these become runtime no-ops
  rather than compile errors.

Errors set ErrorOccurred and skip Coder/Emitter. Warnings do not.
